0一分钟全景：优化推理，就是拧这 5 个旋钮

别被一堆名词吓到。在 AMD 显卡上让大模型「推理」（也就是对外提供问答/生成服务）跑得又快又省，说到底就是拧下面这 5 个旋钮。这篇文章就是逐个把它们讲清楚。

最省事
📦
① 让模型变小
量化：用更少的位数存模型，体积小一半到四分之一，还更快。
⚡
② 换更快的零件
加速库：同样的计算，换个更聪明的实现（Flash Attention 等）。
最重要
🛠️
③ 调好 vLLM 引擎
一个开关 + 几个参数，收益最大。大多数人主要在这一层花时间。
🔧
④ 亲手打磨 kernel
进阶：自己写/调底层计算代码。大多数人用不到，了解即可。
🌡️
⑤ 先测量再动手
用工具量出「到底哪里慢」，别靠猜。这是贯穿始终的习惯。

一句话记住

拧旋钮的顺序建议是：先量化（省事收益大）→ 再调 vLLM（收益最大）→ 顺手开加速库；只有当你「测量」发现某个具体环节卡住、而且现成手段救不了时，才需要去「亲手打磨 kernel」。

几个名字先混个脸熟（后面都会展开）

ROCm＝AMD 显卡的软件平台（相当于 NVIDIA 那边的 CUDA）。Instinct MI300X / MI350X＝AMD 面向数据中心的高端计算卡。vLLM＝目前最主流的大模型推理引擎之一，AMD 卡上也能跑。AITER＝AMD 专门为自家卡写的一套加速代码包（本文杠杆③的主角）。

1先弄懂：推理为什么会慢？ 入门必读

不搞懂这一点，调优就是瞎拧。好消息是：原理特别简单，一个比喻就能讲明白。

大模型生成回答分成两步，它们「累」的地方完全不同：

📖 第一步：读题（Prefill）
  • 把你输入的一整段话「一口气」读进去、理解。
  • 这一步计算量大，显卡的算力被吃得满满的。
  • 好比：一次性读完整道题目。
✍️ 第二步：逐字作答（Decode）
  • 一个字一个字往外蹦，每蹦一个字都要把整个模型在显存里过一遍。
  • 这一步算力大量闲置，卡在「搬数据」上。
  • 好比：写一个字，就把整本教材翻一遍。

🍳 关键比喻：厨师很快，但食材来回搬得慢

「逐字作答」这一步的瓶颈，不是显卡算得慢，而是数据从显存搬到计算单元太慢——就像一个手速飞快的厨师，大部分时间却在等人从仓库把食材搬过来。这叫显存带宽受限（memory-bandwidth-bound）。

那怎么办？让厨师一次多做几份菜。食材搬一趟，同时服务好几桌客人，搬运成本就被摊薄了。这就是「批处理（batching）」：同时处理更多请求，显卡才不闲着。

🍳「厨师快、食材搬得慢」——逐字作答就卡在这根细管子上。
把 batch 做大 = 搬一趟食材同时炒好几桌，把搬运成本摊薄。

由此推出整条优化逻辑（全文的主线）

逐字作答是「搬数据」受限 → 想快就得一次伺候更多请求（大 batch） → 能同时伺候多少，取决于显存里放得下多少「KV cache」（模型记住的上下文，越聊越占地方）→ 所以优化的核心动作就两类：①把东西变小（省显存）②把该省的开销省掉、该并行的并行起来。后面 5 个杠杆全都在服务这条主线。

想更深入理解 prefill / decode / KV cache / 批处理的机制，可以看本站的姊妹篇 vLLM 全面教程，那篇讲得更细。本篇聚焦「在 AMD 卡上怎么调」。

2杠杆① 让模型变小：量化 最省事

量化是投入产出比最高的第一步，很多时候改一两个参数就能省一半显存。

🖼️ 什么是量化？

模型里全是数字。原本每个数字用 16 个二进制位（bit）存，量化就是改用 8 位甚至 4 位来存——好比把一张高清照片存成压缩版：体积小了、加载快了，画质（精度）掉一点点，但肉眼几乎看不出。省下的显存可以用来同时伺候更多请求，于是更快。

两件不同的事：压「模型权重」 vs 压「KV cache」

初学最容易混。它们都能省显存，但压的是不同东西，可以同时做：

  • 压权重：把模型本身变小（FP8、INT4 等）。模型加载更小、算得更快。
  • 压 KV cache：把「对话上下文」的缓存变小（--kv-cache-dtype fp8）。这是最容易吃到的红利——一行参数省一半上下文显存，几乎无损，能聊更长/接更多人。

在 AMD 卡上常用的几种量化方式

方式压到几位省多少显存什么时候用它
FP8（8 位浮点）8 bit约省一半生产首选，速度/精度最平衡。MI300 及以上原生支持。
FP8 KV cache8 bit上下文省一半几乎所有场景都值得开，一行参数的事。
AWQ / GPTQ（4 位整数）4 bit约省四分之三显存特别紧、要塞下大模型时。精度损失略大些。
MXFP4（4 位）4 bit约省四分之三只有较新的 MI350 / MI355 才支持。
Quark（AMD 官方工具）FP8 / INT42×～4×想用 AMD 官方调好的「预量化模型」，拿来即用最省心。

💡 最容易上手的两招

1. 开 FP8 KV cache（几乎无脑收益）：
2. 直接用 AMD 调好的预量化模型（省去自己量化的功夫）：

# 招式1：给任意模型开 FP8 KV cache，上下文显存立省一半
vllm serve meta-llama/Llama-3.1-8B-Instruct --kv-cache-dtype fp8 --calculate-kv-scales

# 招式2：直接拉 AMD 已经量化好的模型（名字里带 FP8-KV）
vllm serve amd/Llama-3.1-8B-Instruct-FP8-KV --dtype auto

⚠️ 一个必须记住的注意点

量化不是「免费的午餐」——它可能轻微影响回答质量。省得越狠（比如 4 位）风险越大。上线前一定要用你自己的任务测一测质量，别只看显存省了多少就发布。

3杠杆② 换更快的零件：加速库 开箱即用

同一个计算，用「更聪明的实现」就能更快。这些库大多是「装上/打开开关」就生效，不用改模型。

加速库用大白话说，它干嘛怎么用
Flash Attention让「注意力」计算少搬很多数据，尤其让第一个字更快出来。在 vLLM 里通常自动启用（见下一节的后端选择）。
TunableOp启动时自动试几种矩阵乘法实现，挑最快的记下来，之后一直用它。PYTORCH_TUNABLEOP_ENABLED=1
torch.compile把模型「编译」一遍，把零碎操作合并、提速。PyTorch 里 torch.compile(mode="max-autotune")
hipBLASLtAMD 优化过的矩阵乘法库（很多加速的底座）。TORCH_BLAS_PREFER_HIPBLASLT=1

AMD 特色：Flash Attention 有两个「引擎」

在 AMD 卡上，Flash Attention 有两套实现：CK（默认，性能强） 和 Triton（更灵活）。用一个环境变量切换：FLASH_ATTENTION_TRITON_AMD_ENABLE="TRUE" 用 Triton，"FALSE" 用 CK。多数情况用默认即可——真正的后端选择，vLLM 会在下一节帮你搞定。

💡 好消息

如果你用 vLLM（下一节），上面这些大部分已经自动帮你用上了。这一层你通常不用手动折腾，了解它们「是什么」即可。

4杠杆③ 调好 vLLM 引擎 收益最大 · 重点

这是全文最重要的一节。绝大多数人在 AMD 卡上部署，主要就是在这里花时间。好在关键动作不多。

第一件事，也是最重要的一件事：打开 AITER

🏎️ AITER 是什么？

AITER（AI Tensor Engine for ROCm）是 AMD 专门为自家显卡手写的一整套加速代码包——好比给车装了「原厂涡轮增压」。它覆盖了推理里最耗时的那些计算（矩阵乘法、注意力、MoE 等）。在 AMD 卡上，这是收益最大的单个开关。

# 打开 AITER，然后正常启动服务
export VLLM_ROCM_USE_AITER=1
vllm serve meta-llama/Llama-3.3-70B-Instruct

⚠️ 新手最常踩的坑

就算你在下面「手动指定了注意力后端」，也必须同时设 VLLM_ROCM_USE_AITER=1。因为这个总开关还负责矩阵乘法、归一化、MoE 等一大堆加速——不开，这些就走回慢速老路，你会白白损失性能。记住：AITER 总开关任何时候都要开。

第二件事：按你的模型，选对「注意力后端」

「注意力」是模型里最核心也最耗时的计算。AMD 上有好几种实现（后端），选对了能快好几倍。照下面这张表对号入座即可：

🦙 Llama / Qwen / Mistral 等普通模型→用 ROCM_AITER_FA（比老实现快 2.7～4.4 倍）
🐋 DeepSeek-V3 / R1（这类叫 MLA 模型）→用 ROCM_AITER_MLA（比旧实现快 1.2～1.5 倍）
🤖 gpt-oss 系列→用 ROCM_AITER_UNIFIED_ATTN
🎮 消费级 Radeon 卡 / 上面都不行→用 ROCM_ATTN（兜底，最稳）

💡 其实更省心的做法

你只要开了 VLLM_ROCM_USE_AITER=1，vLLM 通常会自动帮你选好合适的后端——普通模型自动用 FA、DeepSeek 自动用 MLA。上面这张表是当你想手动确认或强制指定时用的。
额外提速：普通模型 + 高并发时，再加 VLLM_ROCM_SHUFFLE_KV_CACHE_LAYOUT=1 还能让「逐字作答」再快 15～20%。

第三件事：多张卡，怎么分工？

一张卡放不下、或想更快时，就要用多张卡。有 4 种「分工方式」，用大白话理解：

分工方式大白话比喻什么时候用
TP 张量并行一道菜，4 个厨师一起切一起炒模型太大单卡放不下。首选，但要卡间高速互联。
PP 流水线并行流水线：这几张卡做前半段，那几张做后半段卡特别多、或要跨多台机器时。
DP 数据并行开 4 个一模一样的窗口，各自接客模型放得下单卡，纯粹想提高总吞吐。
EP 专家并行把不同「专家」分给不同卡（仅 MoE 模型）DeepSeek、Mixtral 这类「专家混合」大模型。

怎么选？两条黄金法则

1. 能不用多卡就别用——每多一种并行都有「沟通成本」。
2. 一台机器内（≤8 卡）优先用 TP；超过 8 卡或跨机器才上 PP。
特例：DeepSeek 这类 MLA 模型在高并发（同时几百个请求）下，用 DP + EP 组合比纯 TP 快 16～47%，因为纯 TP 会在每张卡上重复存一份上下文、浪费显存。

第四件事：三个一定要会调的旋钮

参数管什么怎么调
--gpu-memory-utilization让 vLLM 用多少比例的显存（默认 0.9）想更快调到 0.95；一张卡跑多个模型时调低（如 0.45）。
--max-num-seqs最多同时伺候几个请求（默认 1024）显存够就调大提吞吐；显存紧就调小。
--max-num-batched-tokens每一步最多处理多少字（在线默认 8192）调大→首字更快；调小→逐字更稳。

一个简单的心算公式

要占多少上下文显存 ≈ 同时伺候的请求数 (max-num-seqs) × 每个请求最长上下文 (max-model-len)。如果启动报「显存不够/KV cache 放不下」，就把这两个数里的某一个调小。

💡 一个反直觉但很实用的技巧

假设你有 8 张卡、模型单卡放得下：与其把 8 张卡合起来跑「一个」服务，不如拆成 8 个各跑一个服务（每张卡一个），再用负载均衡分流。总吞吐通常更高——因为省掉了卡之间的沟通开销。

把这一节串起来：一条能直接用的启动命令

# 普通模型 + AITER + FP8 KV cache 的典型起步姿势
export VLLM_ROCM_USE_AITER=1
vllm serve meta-llama/Llama-3.3-70B-Instruct \
  --tensor-parallel-size 8 \        # 8 卡一起分工（TP）
  --kv-cache-dtype fp8 \            # 上下文省一半显存
  --gpu-memory-utilization 0.95 \  # 尽量多用显存
  --max-model-len 8192

5杠杆④ 亲手打磨 kernel 进阶 · 了解即可

前面都是「用现成的」。这一层是「自己造/改零件」。90% 的人用不到——但知道有这一层、以及各工具的定位，很有帮助。

🔧 什么是 kernel？

kernel（核函数）就是「显卡上实际跑的那一小段计算代码」，比如「做一次矩阵乘法」。前面的加速库本质上就是别人写好的高质量 kernel。这一层说的是：当现成 kernel 都不够快时，自己写一个专门的。

三种自己写 kernel 的工具，难度递增

工具定位难度
Triton用类似 Python 的方式写显卡 kernel，最容易上手。中
Composable Kernel (CK)用 C++ 模板拼出高性能 kernel，能把多个操作「融合」成一个。高
Gluon最底层，手动控制到寄存器/缓存级别，专门榨干「热点」kernel。很高

什么时候才需要下沉到这一层？

只有当你：①用了前面所有现成手段 → ②用「测量」（下一节）发现某一个具体 kernel 是瓶颈 → ③而且现成的实现确实救不了它。满足这三条，才值得自己写。否则性价比很低。

如果你真要调 Triton，几个最常用的旋钮先混脸熟：num_stages（流水线级数：单个矩阵乘设 2、注意力设 1）、BLOCK_M/N/K（分块大小）、matrix_instr_nonkdim（在 MI300X 上 16 通常比 32 快）。诊断时用两个环境变量导出中间代码看瓶颈：MLIR_ENABLE_DUMP=1 和 AMDGCN_ENABLE_DUMP=1。

6杠杆⑤ 先测量，再动手 贯穿始终

最重要的习惯：别猜瓶颈在哪，测出来。 AMD 提供了一整套工具，按「你想解决什么问题」对号入座就行。

想快速看：到底是 CPU 慢还是 GPU 慢？→PyTorch Profiler（最简单，出一张时间线图，用 Perfetto 网站打开看）
确认 GPU 慢，想看每个 kernel 的细节→rocprofv3 命令行 / ROCm Compute Profiler（带图表、Roofline 分析）
想看整个系统全景（CPU+GPU+多卡）→ROCm Systems Profiler（全景追踪，探索式找瓶颈）
程序崩了 / 报显存访问错误→ROCr Debug Agent（抓出是哪个 kernel、哪块显存出的错）

工具速查表

工具层次一句话
PyTorch Profiler高层新手首选，几行代码，看 CPU/GPU 时间线。
rocprofv3底层命令行，不改源码就能抓各种 trace 和硬件计数器。
ROCm Compute Profiler
（旧名 Omniperf）		kernel 级自动采集 + 图形界面，帮你看懂瓶颈，省去手动解析。
ROCm Systems Profiler
（旧名 Omnitrace）		系统级CPU+GPU 全景追踪，不预设瓶颈、探索式排查。
ROCr Debug Agent调试专抓显存访问崩溃，定位到具体 kernel。
 
# 最简单的一招：用 PyTorch Profiler 抓一段，然后拖到 https://ui.perfetto.dev 看
from torch.profiler import profile, ProfilerActivity
with profile(activities=[ProfilerActivity.CPU, ProfilerActivity.CUDA]) as prof:
    model(inputs)
prof.export_chrome_trace("trace.json")

7MI300 vs MI350 速查 选卡/调参前先看

AMD 数据中心卡目前主要是两代。它们的差别会直接影响你能用哪些优化，所以值得先看一眼。

看点MI300 系列
(MI300X / MI325X)		MI350 系列
(MI350X / MI355X)		对你的影响
显存192 / 256 GB288 GB越大 → 能塞更大模型、更长上下文。
片上缓存 (LDS)64 KB / 核160 KB / 核MI350 能用更大的「分块」，kernel 有更多提速空间。
4 位超低精度 (MXFP4)✗ 不支持✓ 支持想用 MXFP4 极致省显存，得用 MI350。
FP8 格式FNUZ 变体OCP 变体两代格式不同！量化模型要匹配对应格式，别搞混。

一句话

如果你用 MI350 / MI355，额外解锁了 MXFP4 这类更省的 4 位量化，而且底层 kernel 因为缓存更大还能再调优一档。跨代迁移时，务必重新测一遍性能、并确认 FP8 格式匹配，别把 MI300 的最优参数直接照搬。

8拿来就用：优化行动清单 按性价比排序

如果你只想要一份「照着做」的清单，就是下面这个顺序——从最省事、收益最稳的开始。

用官方 Docker 镜像起步
AMD 提供了预装好 ROCm + PyTorch + vLLM 的镜像，很多加速参数已默认开好，别从零手搭环境。
打开 AITER 总开关
export VLLM_ROCM_USE_AITER=1——AMD 卡上收益最大的单个动作，永远要开。
开 FP8 KV cache
--kv-cache-dtype fp8——上下文显存立省一半，几乎无损。
确认注意力后端选对了
普通模型 → FA；DeepSeek → MLA。开了 AITER 通常自动选好，跑起来 grep 一下日志确认。
调好三个旋钮 + 选对多卡分工
显存利用率 / 最大并发 / 每步 token；单机 ≤8 卡优先 TP。按你的 SLO（延迟还是吞吐优先）调。
需要更省显存再上权重量化
FP8 起步；显存实在紧再上 AWQ/MXFP4（4 位）。量化后一定测质量。
测量：跑 benchmark + profiler
用 vllm bench 和 profiler 量出真实瓶颈，同时看「首字延迟 / 逐字延迟 / 总吞吐」三个指标。
（仅当必要）下沉到 kernel 层
只有测出某个 kernel 是瓶颈、现成手段救不了，才用 Triton/CK/Gluon 自己写。

三个指标一起看，别只盯一个

首字延迟（TTFT）：多久蹦出第一个字。逐字延迟（TPOT/ITL）：之后每个字的间隔（打字快不快）。吞吐（Throughput）：整个系统每秒能产出多少字。调参时它们经常此消彼长，一定要三个一起报，按你的业务需求取舍。
