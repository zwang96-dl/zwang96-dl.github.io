# 第 02 章 · 什么决定 Kernel 的速度

> 原文:[What Makes a Kernel Fast](https://mlc.ai/modern-gpu-programming-for-mlsys/chapter_roofline/index.html)

> **本章要点(TL;DR)**
>
> - **屋顶线模型(roofline model)** 给一个核函数(kernel)设定了**性能天花板**,这个天花板要么由**内存带宽(memory bandwidth)** 决定,要么由**算力吞吐(compute throughput)** 决定——取两者中更慢的那条路。
> - 决定"撞上哪个天花板"的,是 **算术强度(arithmetic intensity, AI)**:每搬运一字节数据,能做多少有用的算术运算(FLOP/byte)。
> - **算术强度低 → 内存受限(memory-bound)**。出路是:少搬字节、多复用数据、做算子融合(fusion)、用更小的 dtype——本质都是"把 kernel 往屋顶线右边推"。
> - **算术强度高 → 可以做到算力受限(compute-bound)**。此时核心任务变成"**别让张量核(Tensor Core,专做矩阵乘加的硬件单元,见第 0 章)闲着**"。
> - 在现代 GPU kernel 里,最大的杠杆是 **重叠(overlap)**:只要依赖图允许,就让 TMA(硬件托管的批量数据搬运引擎)、张量核、收尾(epilogue,把结果转格式并写回的尾部阶段)、写回这几件事**同时在跑**。
> - 一个性能数字脱离了天花板就没有意义。**先估算术强度、定位天花板、判断是内存受限还是算力受限,再去优化真正卡脖子的那个资源**——这是贯穿全书的优化工作流。

> **前置知识**:读这一章前,最好先懂**内存层级(寄存器 / 共享内存 / 全局内存)、Tensor Core(张量核)、重叠(overlap)**,以及第 01 章讲的**执行层级**。没把握的话,先翻一下 [第 0 章 · 极简入门](./ch00_gpu_ml_primer.md),以及第 01 章。本章会默认你已经认识这些词。

---

第 01 章带你认了认硬件:有哪几层执行单元、有哪些内存、有哪些专门算数和搬数据的引擎,等于先画了张地图。这一章我们聊个更实在的问题——**一个 kernel 到底快不快,是什么说了算?**

先说结论:不是什么玄学技巧。它是一套能让你"拿数字算账"的框架,叫屋顶线模型。把它学会,你还没动手写代码,就能先判断出劲该往哪儿使。

> **关键**:一个性能数字,光盯着它本身看,啥也说明不了。你得**拿它跟天花板比**,它才有意义。
>
> 举个例子你就懂了。330 TFLOP/s,听着是不是挺唬人?可假设这颗 GPU 能稳稳跑到 2 PFLOP/s 左右(稠密 fp16/bf16 张量核),那这数字**连六分之一的算力都没摸到**,大半个芯片都在那儿干瞪眼呢。说白了,没有天花板当参照,你压根分不清:这 kernel 是已经榨到硬件极限了,还是还有一大把性能躺着没人用。

那这天花板是从哪儿冒出来的?这正是屋顶线模型的活儿。它的思路特简单:一个 kernel 干的事,无非两类——**搬字节**和**做算术**。然后看是谁在拖后腿。要是数据喂不上来,那带宽就是你的上限;要是数据反复用、算术活儿也够重,这才轮到算力吞吐来当上限。

> **注意(本章数字约定)**:跟第 01 章一样,我们还是拿 **NVIDIA B200** 当例子,而且天花板都取个整方便推算:稠密 fp16/bf16 张量核吞吐算 **2 PFLOP/s** 上下,HBM3e 带宽算 **8 TB/s** 上下。这些数会随着型号、时钟、功耗墙、怎么测来回变,所以你就把它们当成**数量级**(order-of-magnitude)的参考,别当成数据手册上一分不差的常数。

## 一、屋顶线模型:两条路取更慢的那条

前面说了,每个 kernel 都干两件事:搬数据、做算术。屋顶线模型的精髓就一句话——**这两条路谁慢,谁就是你的上限**。

- **算力天花板 / compute ceiling**:就是硬件算术能力的顶。比如 B200 上跑张量核 GEMM(通用矩阵乘,即稠密大矩阵相乘),这个顶就是张量核的吞吐;而要是个逐元素(elementwise)的小 kernel,这个顶可能是 CUDA 核(通用算术核,逐个线程做标量运算)的吞吐,也可能是别的某个功能单元。
- **内存天花板 / memory ceiling**:它等于**带宽 × 算术强度**。每搬一字节只算那么一丁点,内存带宽就成了瓶颈;反过来,一字节能撑起一大堆运算,那内存基本就不卡你了。

把这两条凑一块儿,就是屋顶线那个基本不等式:

```
可达 FLOP/s  ≤  min( 峰值 FLOP/s ,  内存带宽 × 算术强度 )
```

其中算术强度定义为:

```
算术强度 = 有用的 FLOP 数 / 搬运的字节数        (单位:FLOP/byte)
```

> **注意**:一提算术强度,你**先得说清楚是按哪一级内存算的**。按 HBM 画屋顶线,字节就指 HBM 字节;按 L2 算,就是 L2 字节;按 SMEM 算,就是共享内存字节。这一章如果不特别说明,默认都在聊 **HBM 屋顶线**。

### 1.1 读懂这张图:斜屋顶、平屋顶与脊点

先教你怎么看这张图。**横轴是算术强度**(FLOP/byte),**纵轴是你能跑到的性能**。图上有两条"屋顶":

```
内存屋顶(斜线):   性能 = 带宽 × 算术强度
算力屋顶(平线):   性能 = 峰值 FLOP/s
```

这俩线会撞到一起,撞上的那个点叫**脊点 / ridge point**:

```
脊点 = 峰值 FLOP/s / 带宽
```

脊点就是那个临界值——算术强度到了这儿,你正好从内存受限切到算力受限。拿 B200 取整后的数字代进去算算:

```
脊点 ≈ 2000 TFLOP/s / 8 TB/s
     ≈ 250 FLOP/byte
```

把斜屋顶、平屋顶、脊点这三样画一块儿,差不多就是下面这样:

```text
Perf (FLOP/s)
  ^
  |                           compute ceiling (flat) = peak ~ 2 PFLOP/s
  |                +--------------------------------------
  |               /
  |             /    <- memory ceiling (slope) = BW x AI
  |           /
  |         /  * ridge ~ 250 FLOP/byte
  |       /
  |     /
  |   /    :                              :
  | /      :        region A             :        region B
  +--------+----------------------------+----------------> AI (FLOP/byte)
```

图注:横轴是**算术强度 AI**(FLOP/byte),纵轴是**可达性能**(FLOP/s)。

| 标记 | 含义 |
| --- | --- |
| memory ceiling(斜线) | 内存屋顶 = 带宽 × 算术强度 |
| compute ceiling(平线) | 算力屋顶 = 峰值 FLOP/s ≈ 2 PFLOP/s |
| ridge(`*`) | 脊点 ≈ 250 FLOP/byte,两条屋顶相交处 |
| region A(脊点左侧) | 内存受限区:被带宽卡住 |
| region B(脊点右侧) | 算力受限区:可被算力卡住 |

> **关键**:屋顶线真正值钱的,**不是这张图长啥样**,而是它一眼就能告诉你——**到底是哪个资源在卡你**。
> - 算术强度**落在脊点左边 → 内存受限**:这会儿你再怎么打磨算术指令也是白搭,因为字节根本喂不上来。
> - 算术强度**落在脊点右边 → 有机会做到算力受限**:这会儿抠那几个零碎字节也没用,你该干的是把算力单元喂饱,把性能顶到平屋顶上去。
>
> 一句话:内存受限的 kernel,不会因为你"数学指令写得漂亮"就变快;算力受限的 kernel,也不会因为你"省了几个不相干的字节"就变快。**所以优化的第一步永远是:先看清楚自己落在脊点哪一边。**

## 二、常见 workload 的算术强度

这里有个特别重要的发现:**算术强度多半是算法自己定的,跟你怎么写实现关系不大**。这有啥好处?好处大了——你连一行 kernel 都还没写,就能先把它的算术强度估个八九不离十,提前知道劲该往哪儿使。下面挑三类最典型的 workload 看看。

| Workload | 算术强度 | 在脊点哪一侧 | 优化目标 |
| --- | --- | --- | --- |
| 逐元素 / 归约(GELU、RMSNorm) | **低**(常数级) | 远在左侧,内存受限 | **逼近内存带宽屋顶** |
| GEMM(矩阵乘) | **高**(随规模增长) | 右侧,算力受限 | **喂饱张量核 + 重叠** |
| 注意力(Attention) | **居中**(取决于实现) | 看情况 | 先抬高 AI,再做调度 |

### 2.1 逐元素与归约:天生内存受限

逐元素 kernel(像 **GELU**)和归约式 kernel(像 **RMSNorm**)有个通病:它们要读写**老大一个张量**,可摊到每个元素上,**就做那么两三次 FLOP**。

搬一大堆、算没几下,算术强度能不低吗?所以它们**远远地趴在脊点左边**。这类 kernel 你别惦记着去够张量核屋顶,踏踏实实去贴**内存带宽屋顶**就对了。换个说法,你要问的全是些很"机械"的访存问题:

```
- 读写是否合并(coalesced)?
- 每个字节是不是只搬了一次?
- 能不能和上游(producer)或下游(consumer)算子融合?
- dtype 能不能再小一点?
- 能不能用 TMA 或向量化访存来帮忙?
```

> **关键**:一个 kernel 要是**既没法复用、又没法融合**,那内存屋顶就是它实打实的头顶,谁也别想往上抬。这时候你的目标不是去"超过"屋顶——超不过去——而是把它"**贴满**"(怎么贴,第四节细说)。

### 2.2 GEMM:算术强度随规模增长

GEMM 是另一头的极端。它的算术强度**问题越大、它越高**。为啥?因为**一块 tile(大矩阵切出来的小方块)搬进来,它能被反复拿去做好多次乘加(MMA,矩阵乘加运算)**——加载一回,用上好多回,FLOP/byte 可不就上去了。

拿个方阵 fp16 矩阵乘来说(`M = N = K`),理想情况下算术强度差不多是:

```
AI ≈ 2N³ / (3 · 2N²)
   = N / 3   FLOP/byte
```

> **注意(这个估算有几个前提)**:它假设 A、B 各只读一遍、C 只写一遍、`beta = 0`、片上复用一点不浪费,而且没有任何多余的元数据、padding 或者冗余流量。实际 kernel 搬的数据**比这个理想值多**,不过拿来"估个量级"还是够使的。
>
> 公式别背,搞懂它咋来的就行。先看分母 `3 · 2N²`:三块矩阵(A、B、C),每块 `N²` 个元素,每个元素 2 字节(fp16)。再看分子 `2N³`:这是整个矩阵乘的浮点运算总数——每个输出元素得做 N 次乘加,一次乘加算 2 个 FLOP。

代进去 `N = 4096` 试试:

```
AI ≈ 4096 / 3 ≈ 1365 FLOP/byte
```

1365 早把 B200 的脊点(约 250 FLOP/byte)甩到**左边**老远了。所以结论很清楚:**大 GEMM 在 HBM 屋顶线底下,是算力受限的**。既然这样,它要优化就**别**去省 HBM 流量了,该干的是:**把张量核使起来、喂饱它,再让搬数据和算数据重叠着跑**——这么干才摸得到算力屋顶。

> **关键**:这正好解释了一个看着挺拧巴的现象——**朴素 GEMM 算术强度明明很高,却可能慢得要命**。咋回事?因为算法**准许**你跑得快,不等于你的实现**真的**跑得快。代码写得糙一点,张量核照样能大段大段地闲着没事干。说到底,算术强度只是给你画了个天花板,够不够得着,那是另一码事(看第三节那个"优化阶梯")。

### 2.3 注意力:介于两者之间的混合问题

注意力(Attention,Transformer 里让每个 token 看其他 token 的核心算子)就卡在前面这俩中间。它的算术强度不是个定数,好几样东西都能让它变:序列长度、头维度(head dimension)、tiling、掩码(masking),还有最要命的一样——你**有没有把中间张量物化(materialize)** 到内存里去。

标准注意力最疼的地方,就在**分数矩阵 / score matrix(query 和 key 两两点积得到的中间大矩阵)**。kernel 要是把分数矩阵先写进 HBM、回头再读回来,那等于让一个巨大的中间张量在内存里白白跑了个来回。**Flash Attention(书里拿 Flash Attention 4 举例)** 的高明就在这儿:它把要用的 tile **摁在片上**,这趟 HBM 往返直接省了,算术强度"噌"一下就**上去了**。

> **关键**:所以优化注意力是个"两步走"的活——**前半段是屋顶线问题,后半段是调度问题**。第一步先改算法,把流向 HBM 的字节砍下来(也就是把 AI 抬上去);第二步再调度 kernel,让剩下的搬运和计算重叠起来。你品一品就会发现,这不就是把 GEMM 那套打法直接搬过来用嘛。

## 三、算术强度低的时候怎么办

一个 kernel 落在脊点**左边**,它就是内存受限的。这会儿张量核、CUDA 核很可能大段大段地干等着,因为卡你脖子的是**字节**,不是算术指令。咋办?两条路。

### 3.1 第一条路:抬高算术强度(高杠杆)

这条路更划算,因为它能把 kernel **往算力受限那边推**。具体有三招。

**第 1 招:算子融合(fusion)——这招最关键。** 算术强度低,最常见的病根就是"一个中间张量刚写进 HBM,下一个算子转头又把它读回来"。怎么治?把上游的 producer 和下游的 consumer **塞进同一个 kernel**。这么一来,中间张量就一直待在寄存器(register,线程私有的最快存储)、SMEM(共享内存,一个 block 内线程共用的片上内存)或者 TMEM(Blackwell 上专给张量核累加器用的片上内存)里,**那趟 HBM 往返直接没了**。几个典型例子:

```
- GEMM + 逐元素 epilogue 融合
- normalization 折叠进相邻算子
- 注意力不物化完整分数矩阵(即 Flash Attention)
```

**第 2 招:为复用而分块(blocking for reuse)。** 道理简单得很:一块 tile 搬进来一次,趁它还没被踢出去,能多用几次就多用几次,这样每个字节就摊上了更多算术活儿。GEMM 算术强度高,靠的就是这个。别的 workload 只要也有"同一块 tile 反复用"的机会,这招照搬就行。

**第 3 招:让每个数变小(换更小的 dtype)。** 从 fp32 往下降到 fp16、fp8 乃至 fp4,一箭双雕:流量小了,FLOP/byte 还顺带抬高了。

> **注意**:实际省下来的,通常**没有**dtype 比例算出来的那么多。因为低精度格式常常得拖着元数据、缩放因子(scale factor),要么就得多走一道转换——**block-scaled fp8/fp4** 就是这样。话虽如此,换更小的 dtype 还是把 kernel "往屋顶线右边推"最干脆的办法之一。

### 3.2 第二条路:认了,就把内存屋顶贴满

不是每个 kernel 都救得回来。有些就是**没东西可融、也没复用可榨**:一次纯拷贝、一个简单的逐元素操作、对着一个大张量扫一遍做归约——它们生下来就是内存受限,你改不了。

> **关键**:碰上这种,别惦记着去"打败"屋顶,想办法把它**贴满(saturate)** 就行。怎么贴?

```
- 每个字节只搬一次
- 避免冗余读取
- 用合并(coalesced)或向量化访存
- 对规则的批量 tile 用 TMA
- 保持足够多的内存请求在途(in flight),把延迟藏住
- 算法允许时,用更小的存储 dtype
```

> **关键**:一个内存受限的 kernel 一旦把内存屋顶贴满了,**你再去抠算力那边的优化就是白费功夫**。这时候想让它更快,只剩一条路:**回头改算法,让它少搬点字节**。

## 四、优化阶梯:从"可能"到"做到"

屋顶线只跟你说**什么有可能做到**,它可不管**够到那个极限有多费劲**。这两码事你得分清楚。

打个比方,一个大 fp16 GEMM 理论上是算力受限的。可这话的意思无非是"HBM 屋顶不是主要拦路虎",**绝不是说你随手写个实现就能摸到张量核屋顶**。从"理论上能到"到"实际真跑到",中间隔着一道坎;想填平它,你得用对指令、摆对布局(layout)、做对暂存(staging),还得把同步和调度都安排明白。

> **关键**:这道坎,本书第三部分在 B200 上拆成了**一级一级的台阶**(就是后面「用 Warp 专精与集群扩展 GEMM」那一章)。这里有个点你留意:每往上迈一级,**底层算法压根没动**,变的只是 tile **怎么算、怎么调度**。

GEMM 优化阶梯:性能一级一级往上走,可**算法自始至终没变**,变的只是 tile 怎么算、怎么调度。

```text
Perf
  ^
  |                                        * S6
  |                                  *
  |                          * S4
  |                    * S3
  |          * S2   <- first big jump: thread-copy -> TMA
  |   *  S1
  +------------------------------------------------> optimization step
```

| 台阶 | 名称 | 说明 |
| --- | --- | --- |
| S1 | 同步 tiled 基线 | thread-copy 分块路径,作为起点 |
| S2 | TMA 路径 | **第一次能测出来的大跳**:thread-copy → TMA |
| S3 | Warp 专精 | 分离 producer / consumer 角色 |
| S4 | CTA 集群 | 更大的协作 tile 与多播 |
| S6 | 多消费者执行 | 顶部:多个消费者同时取用结果 |

- **第一次能测出明显提速的大跳**,是从 **thread-copy 分块路径** 换到 **TMA 路径**。TMA 干的是这事儿:那种"规则的 GMEM → SMEM tile 搬运",本来压在 CTA 线程身上,现在交给硬件托管的批量拷贝去喂张量核。线程解放了,搬运还更快了。
- **再往后的提速,基本全靠重叠和调度**:TMA 提前把后面要用的 tile 预取进共享内存;`tcgen05.mma` 异步地算;epilogue 这边同时排干(drain)上一批结果;最后再用 **软件流水线(software pipelining)** 加 **warp 专精(warp specialization)** 把这几段编排到一块儿,让好几个硬件引擎**一起开工**。

> **注意**:别想当然觉得"每爬一级都得比上一级快"。像 warp 专精这种步子,可能会**先**把资源砸进一个"眼下并不马上提速"的结构里。可只要它能**解锁那些更简单结构压根做不到的重叠**,这步就走对了——它是在给后面铺路呢。

## 五、重叠是最大的杠杆

假设一个 GEMM 已经是算力受限、张量核也用上了,那还差点啥?**多半就差在"空档"上**——总有些引擎在那儿干等,时间白白溜走了。

来看看一个朴素 kernel 是咋跑的:

```
load  tile k
compute tile k
store tile k
load  tile k+1
compute tile k+1
store tile k+1
```

这种一步压一步的排法,会把硬件**晾在那儿大把空转**:加载在跑的时候,张量核干瞪眼;张量核在算的时候,拷贝引擎没事干;轮到写回排干,前面俩可能又都在等。说白了,任何一个时刻,基本就一个引擎在干活。

流水线化的 kernel 就两样了,它让**互不相干的几个阶段同时开跑**:

```
load    tile k+1     ← 预取下一块
compute tile k       ← 算当前块
store   tile k-1     ← 排干上一块
```

**朴素(串行,引擎大量闲置)**——任一时刻**只有一个引擎在干活**:

| 引擎 | t0 | t1 | t2 | t3 | t4 | t5 |
| --- | --- | --- | --- | --- | --- | --- |
| TMA | `[load k]` | | | `[load k+1]` | | |
| Tensor | | `[mma k]` | | | `[mma k+1]` | |
| Store | | | `[st k]` | | | `[st k+1]` |

**流水线(重叠,多引擎齐忙)**——同一时刻**多个引擎都在干活**:

| 引擎 | t0 | t1 | t2 | t3 |
| --- | --- | --- | --- | --- |
| TMA | `[load k+1]` | `[load k+2]` | `[load k+3]` | ... |
| Tensor | | `[mma k]` | `[mma k+1]` | ... |
| Store | | | `[st k-1]` | `[st k]` ... |

这就是本书后面 Blackwell kernel 结构的主心骨:**TMA** 负责异步搬数据,**`tcgen05.mma`** 负责异步算张量核,**epilogue 和写回** 管输出这头,再拿 **`mbarrier`**(异步屏障对象,用来等"某件事干完了"再放行)这么个对象把各个阶段拴到一起,让每个消费者**只在它真要用数据的那一刻,才去等一下**。

> **关键**:这里你得想明白一件事——重点**不是把依赖消掉,而是绕着依赖来排活儿**。依赖是消不掉的:tile `k` 的 MMA 非得等 tile `k` 加载完才能动;tile `k` 的 epilogue 非得等 MMA 算完才能读累加器。可妙就妙在——tile `k+1` 的加载**往往可以**趁着 tile `k` 的 MMA 还没算完就先跑起来;tile `k-1` 的写回**往往可以**同时在那儿排干。正是因为有这些"能并行的空档",后面好几章才专门来讲各种异步机制:

| 机制 | 作用 | 对应后续章节 |
| --- | --- | --- |
| **TMA** | GMEM → SMEM 的搬运 | Async Data Movement: TMA |
| **mbarrier** | 标记加载完成、做资源交接 | Async Coordination: mbarriers |
| **`tcgen05`** | 异步张量核计算 | Special Memory: TMEM |
| **TMEM** | 存放长生命周期的累加器 | Special Memory: TMEM |
| **warp 专精** | 分离 producer / consumer 角色 | 用 Warp 专精与集群扩展 GEMM |
| **集群(cluster)** | 更大的协作 tile 与多播(multicast) | 同上 |

这些机制长得各不一样,可它们都在伺候**同一个调度目标**:让有用的活儿**同时**跑在好几条硬件路径上,而不是挤在一条上。

## 六、占用率与资源压力

> **关键**:藏延迟的法子不止"重叠"这一招。还有个更老、更通用的机制,叫 **占用率(occupancy)**。

先说占用率是个啥:就是**同一时间常驻在一个 SM(流式多处理器,GPU 上独立调度执行的计算核心,见第 0 章)上的活儿有多少**。它咋帮你藏延迟呢?是这样:一个 warp(线程束,32 个线程的小班,见第 0 章)卡住了(stall),调度器立马切到另一个已经准备好的 warp 去跑。只要你手头总攒着一池子"招手就来"的独立 warp,延迟就这么神不知鬼不觉地盖过去了。

那占用率被啥卡着?被**单个 SM 的资源**卡着,主要这四样:**寄存器、共享内存、warp 槽位、CTA 槽位**。一个 kernel 要是每个线程吃一大堆寄存器,或者每个 CTA(协作线程阵列,即一个 block,会被整体调度到同一个 SM 上)啃掉一大块共享内存,那它占用率一般就高不了——因为一个 SM 上根本塞不下几个 CTA、几个 warp。

这里有个反常识的地方:**不少现代张量核 kernel 是成心"大手大脚"用资源、成心把占用率压低的**。多级共享内存流水线把 SMEM 吃掉,大块的寄存器片段(fragment,张量核运算时操作数在各线程寄存器里的分片)把寄存器吃掉,TMEM 分配把张量内存吃掉,warp 专精更狠,**一整个 warp 一整个 warp 地划走**,专门给 producer 或 consumer 角色用。图啥呢?

> **注意(这是成心做的取舍)**:因为这些 kernel 换了套藏延迟的玩法。它们不再靠"一大堆不相干的 warp 常驻"来藏,而是改成**在少数几个常驻 CTA 内部,用明摆着的重叠去藏**。所以你会看到个挺有意思的结果——**一个占用率很低的 kernel,只要它的流水线能让 TMA、张量核、写回都没闲着,照样能跑得飞快**。
>
> 这两套玩法没有谁绝对更高明:有些 kernel 因为访存不规则、或者腾不出地方做显式重叠,就**只能**吃高占用率这碗饭;另一些则**非得**靠深度暂存加专精不可,因为只有这样才喂得饱张量核。所以你真正该问的,从来不是"占用率高不高",而是——**正在干活的那些硬件单元,到底喂饱了没有?**

## 小结

这一章说来说去就一句话:**先看病,再下药。** 后面整本书都会一遍遍绕回这三个诊断问题:

```
- 这个 kernel 在哪个屋顶之下?
- 是哪个资源在卡脖子?
- 哪个改动能把它推近那个屋顶?
```

情况不一样,答案也就很清楚了:

| Kernel 类型 | 诊断 | 优化方向 |
| --- | --- | --- |
| **内存受限** | 落在脊点左侧,被带宽卡住 | 减少字节 + 提升带宽利用:融合、合并、向量化、TMA、更小 dtype |
| **算力受限 GEMM** | 落在脊点右侧 | **先上张量核,再做重叠**:暂存操作数、发异步 MMA、填满流水线、不卡住计算路径地排干结果 |
| **Flash Attention** | 先抬 AI 再调度 | 先把分数 / 概率 tile 留在片上抬高 AI;之后复用 GEMM 那套重叠工具(tiled 搬运、SMEM 暂存、异步计算、资源交接) |

这么一来,就有了一套能照着做的优化流程:

> **估算术强度 → 定位屋顶 → 判断是内存受限还是算力受限 → 只去优化那个真正设定天花板的资源。**

少了这一步,优化 kernel 就成了拍脑袋瞎猜;有了它,**你每动一处都说得清为什么**——要么是把算术强度抬上去了,要么是把内存路径往带宽峰值推近了,要么是把算力屋顶底下那点空闲时间给砍掉了。
