先看问题：不这么做会付出什么代价

为什么值得花一整课学「一个请求的生命周期」？因为推理引擎最核心的价值， 就藏在「同时来了多个请求时怎么办」里。用本课真实 workload 的两个请求做对比 （req_0 在第 0 迭代到达、要生成 6 个 token；req_1 在第 1 迭代到达、要生成 4 个 token）：

引擎做法req_1 何时拿到首 token两个请求全部完成
朴素排队（FIFO：一次只跑完一个请求）第 7 迭代（干等 req_0 跑完 6 步）第 10 迭代
本课的生命周期 + 连续批处理第 2 迭代第 6 迭代

同样的两个请求、同样的计算量，只是换了一种编排方式：总耗时从 10 步降到 6 步， 而后到的 req_1「等待首字」的时间从 6 步压缩到 1 步。差别不来自更快的硬件，而来自 把请求拆成可调度的「状态 + 阶段」，让它们在同一个 batch 里穿插推进——这正是本课要建立的心智模型， 也是 vLLM 高吞吐的根。所以在学 Attention/KV Cache 之前，先把「一个请求如何流过引擎」看透。

系统问题场景 & 你的预测 Learn

设想推理服务器同时收到两个请求：一个长一点、一个短一点，还错峰到达。引擎必须决定 先处理谁、每一步处理多少、内存放哪里、什么时候释放。

先预测，再看动画
  1. 第一步应该把整段 prompt 一次算完，还是一个 token 一个 token 地算？
  2. 生成阶段，每一步的输入长度是「不断变长」还是「始终为 1」？
  3. 一个请求结束后，它占用的显存（KV block）会怎样？

一句话解释
推理引擎把每个请求当作一个状态机：等待（WAITING）→ 运行（RUNNING，先 prefill 一次吃完整段 prompt， 再逐 token decode）→ 结束（FINISHED）并归还它占用的 KV 内存。调度器则决定「每一步让哪些请求、各推进多少 token」。

直觉类比：批改流水线上的多份试卷。每份试卷（请求）先被「通读题干」一次（prefill 一次吃掉整段 prompt）， 之后进入「一行一行写答案」的节奏（decode 每步只写 1 个 token）。监考老师（调度器）不必等一份卷子写完才发下一份—— 他可以在同一轮里，让 A 卷写下一行的同时把刚到的 B 卷先通读一遍。这就是 continuous batching 的直觉。

类比的边界（别被带偏）
写字的人凭记忆，而模型每写一个 token 都要重新「回看」前面所有内容；如果每步都把历史从头重算， 开销会随长度爆炸——这正是 KV Cache（Lesson 7）存在的理由：把每个历史 token 的中间结果缓存下来，decode 时直接取用。 另外，「通读题干」并非只发生一次就永远够用：每生成一个新 token，它也会变成后续步骤要「回看」的历史之一。

精确工程定义：请求状态机 Learn

「状态机」不是比喻，而是代码里一个真实的枚举 RequestState（见 text_pipeline.py）。每个请求任一时刻恰好处于一个状态， 状态之间的跳转由明确的条件触发——把这些条件说清楚，你就读懂了 LifecycleSimulator.run()。

状态含义进入条件（对应代码）
WAITING已到达，尚未被调度请求刚创建（SimRequest 默认 state=WAITING）
RUNNING正在被处理，含子阶段 phase = "prefill" | "decode"被准入时置位：arrival < iteration 且 len(running) < max_num_seqs
FINISHED正常结束，KV block 已全部归还满足 len(generated) >= max_new_tokens（即 req.done）
ABORTED被中止（本课不触发，占位）超时 / 客户端取消等（Lesson 后续展开）

关键在于：RUNNING 内部还有一个子状态 phase。请求被准入时先进入 prefill；第一次 prefill 产出首 token 后，立刻翻转成 decode 并一直停在那里，直到写满而结束。 整个生命周期只有这几条合法边（其余跳转都不会发生）：

prefill → decode 的翻转只发生一次（首 token 之后）；decode 会自我循环 N-1 次。 ABORTED 是一条本课不触发的旁路，故未画出。

这套状态机在 Lesson 10（Continuous Batching）会正式展开；这里先建立直觉： 请求可以随时加入或离开同一个正在运行的 batch——动画里 req_1 就是在 req_0 已经开始 decode 之后才被准入的。

代码映射
枚举定义在 class RequestState(enum.Enum)；准入逻辑在 LifecycleSimulator.run() 的 while pending and len(running) < self.max_num_seqs and pending[0].arrival < iteration； prefill → decode 的翻转是 prefill 分支末尾的 req.phase = "decode"； 结束判定用 SimRequest.done 属性（len(generated) >= max_new_tokens），随后 pool.release() 归还 block。

小规模手算：跟着真实 trace 走完 6 步 Learn

光看状态图还不够，我们用本课真实 workload 亲手推一遍整轮调度，最后和 python3 course.py run 0 --trace 的输出逐行对上。配置来自 configs/lesson_00_quick.json：

  • req_0：prompt = "Hello, mini-vLLM!"，字节级 tokenize + BOS → prompt_len = 18（17 字节 + 1 个 BOS），要生成 6 个 token，第 0 迭代到达。
  • req_1：prompt = "你好，推理引擎"，每个汉字 3 字节、全角逗号 3 字节 → 6+3+12 = 21 字节 + BOS = prompt_len = 22，要生成 4 个 token，第 1 迭代到达。
  • 调度参数：block_size = 8、num_blocks = 16、max_num_seqs = 2、token_budget = 32。

每一步的规则很简单：先准入（把已到达且还等待的请求塞进 running，上限 max_num_seqs）； 再调度（在 token_budget 内排 token，prefill 花 prompt_len 个、decode 只花 1 个）； 最后收尾（写满的请求置 FINISHED 并归还 block）。需要的 block 数 = ⌈total_len / block_size⌉。

手算结果与终端完全吻合：总迭代 6、调度 token 总数 = 18+23+2+2+2+1 = 48、 KV block 峰值 6 / 16、两个请求都走到 FINISHED、结束后占用归零（无泄漏）。 这几个数字就是 run 0 摘要里那几行的来历。

从这张表能读出的三件事
  • prefill 一次、decode 逐步：req_0 只在第 1 步花了 18 个 token（吃整段 prompt），之后每步固定花 1 个。
  • 连续批处理：第 2 步里 req_0 正在 decode，req_1 才刚 prefill——两个不同阶段的请求同处一个 batch。
  • token_budget 是闸门：第 2 步 23 ≤ 32 所以两者同批；若把 token_budget 调到小于 22，req_1 的 prefill 就会被卡住（长 prompt 阻塞，正是 Lesson 15 chunked prefill 要解决的）。
代码映射
这张表就是 LifecycleSimulator.run() 主 while 循环的逐行展开：准入段（步骤 1）→ 调度段的 cost = req.prompt_len（prefill）/ cost = 1（decode）与 budget 检查（步骤 2）→ _ensure_blocks()（用 _blocks_needed = ⌈length/block_size⌉ 调 _BlockPool.allocate()）→ 收尾段对 req.done 的检查与 pool.release()（步骤 3）。

Tensor Shape、数据流与代码调用图 Learn

虽然 Lesson 0 用的是「玩具下一个 token」而非真实模型，但数据流与 shape 是真实的—— --trace 打印的 input_ids_shape / logits_shape 就是真实 vLLM 里同名张量的形状。 每个 logits 张量都是 3 维 [batch, seq, vocab]，逐符号解释：

  • batch：这一次 forward 里塞了几条序列。本课把每个请求单独喂，故恒为 1。
  • seq：这一次喂进去的 token 个数。prefill = 整段 prompt 长度；decode = 1（只喂刚生成的那 1 个新 token）。
  • vocab：词表大小 259（= 256 个字节 + BOS/EOS/PAD 三个特殊 token，Lesson 1 讲来历）。它是「对每个可能的下一 token 打的分」的长度。
阶段输入 input_ids输出 logits代入 req_0 的具体数字
Prefill[1, prompt_len][1, prompt_len, 259][1, 18] → [1, 18, 259]
Decode[1, 1][1, 1, 259][1, 1] → [1, 1, 259]

注意一个反直觉点：prefill 明明喂了 18 个 token、也算出了 [1, 18, 259] 这么大一坨 logits， 但只有最后一个位置（第 18 个）的那 259 个分数被用来选首 token，前 17 个位置的 logits 在推理时并不需要。 decode 则天生只有 1 个位置，取它即可。这解释了为什么 decode 的输出永远是 [1, 1, 259]。

请求到达 → tokenize → [调度器] → [KV block 分配] → [ModelRunner]
   WAITING        encode()     schedule()      allocate()       forward()
       └──────────────────────────► RUNNING(prefill→decode) ──► FINISHED(free)
动画动作 ⟶ 对应代码
  • 「为 req_0 分配 physical block 3」 ⟶ _BlockPool.allocate()（预览 BlockAllocator.allocate()）
  • 「prefill 一次吃掉整段 prompt」 ⟶ LifecycleSimulator.run() 中 phase == "prefill" 分支
  • 「结束并归还 block」 ⟶ pool.release() + 断言 pool.in_use == 0（无泄漏）

常见误解 Learn

这几条是初学者对「请求生命周期」最容易形成的错误心智模型。每条都能用本课的 trace 直接证伪：

误解为什么错
「decode 每步要把整段上下文重新喂进模型。」 trace 里 decode 的 input_ids_shape 恒为 [1, 1]——只喂 1 个新 token。 历史内容靠 KV Cache（Lesson 7）保留，cache_len 那个字段就是「已缓存的历史长度」，它逐步增长（18→19→20…），而输入长度不变。
「prefill 和 decode 是两段不同的模型 / 两份不同的前向代码。」 是同一个 forward，只是喂进去的 seq 长度不同（N vs 1）。所以它们的 logits shape 形状规律一致（[1, seq, 259]），走的是同一条代码路径。
「请求得像排队一样，前一个彻底结束后一个才能开始。」 连续批处理允许请求随时加入正在运行的 batch。trace 第 2 迭代里 req_0 正在 decode、req_1 才刚 prefill，两者同批推进——这正是把总耗时从 10 步压到 6 步的原因。
「KV block 只是逻辑概念，不会真的用完。」 物理 block 数量是有限的（本课 num_blocks = 16）。_BlockPool.allocate() 在 free list 空时会抛 MemoryError（模拟 OOM）——真实系统这时才需要抢占 / 换出。
「block_size = 8 意味着一个请求最多只能有 8 个 token。」 block_size 是每个 block 能装多少 token，不是请求上限。装不下就再要一个 block：trace 第 5 迭代里 req_1 上下文涨到 25，就临时申请了第 4 个 block（⌈25/8⌉=4）。

本节总结 & 唯一推荐的下一步

  • 推理引擎把请求当作状态机：WAITING → RUNNING(prefill→decode) → FINISHED。
  • prefill 一次处理整段 prompt；decode 每步只处理 1 个 token。
  • KV 内存以 block 为单位分配，结束即释放（无泄漏）。
  • 三分工：网页理解、编辑器改码、终端运行——网页不替你运行、不改你的代码。
