为什么需要这个知识 Learn

多模态推理里，「重复」出现在三个完全不同的层级，成本也天差地别：把一张图 resize/normalize（便宜）、把它送进 vision encoder + projector 编码成视觉 embedding（很贵）、 以及文本模型逐层的 K/V（Lesson 7 已缓存）。如果不加区分，要么白白重算昂贵的编码， 要么用错缓存拿到别的图的 embedding。这一课就是把这三层缓存各归各位，并搞清 「缓存的 key 到底要包含什么」——这是多模态里最隐蔽、最容易踩的一类 bug。

先看问题：不分层、或 key 太粗会怎样 Learn

设想两个很常见的场景（本模型里，一张 16×16 图 → grid 2×2 → 4 个 visual token， 每次编码要跑一遍 patch→vision encoder→projector）：

  • 同一张图被多次用到（多轮对话反复贴同一张图、或一个 batch 里两条请求引用同一张图）： 若不缓存编码结果，vision encoder 就每次都白跑一遍——这是最贵的一步。
  • 换了 encoder / projector，key 却没变：比如你把 projector 换成另一套权重， 但缓存 key 只用「图片内容」做索引 → 新请求命中了旧 encoder 算出的 embedding， 于是模型「看」到的是用错模型编码的图 —— 这就是 stale cache，结果悄悄出错还不报错。

结论：缓存能省大钱，但命中条件必须精确——只有「一切影响结果的因素都相同」时才允许命中。

直觉类比 Learn

一句话解释
三层缓存 = 在多模态流水线的三个不同工位，各放一个「这活儿我之前做过就别重做」的备忘本； 每个工位备忘的内容不同，查备忘用的钥匙（key）也不同。

类比：三道工序的车间。①切菜（预处理）② 炒菜（编码，最费火）③ 装盘上桌（文本解码）。 切好的菜、炒好的菜、上桌的菜是三种不同的半成品，各自的「回收利用」规则也不同—— 你不会拿「切好的菜」当「炒好的菜」用。三层缓存正是这个道理。

类比的边界
车间的「同一道菜」靠肉眼认；缓存靠 key 精确匹配。key 少写一个因素（比如「用了哪套炒锅」=encoder 身份）， 就可能把「用 A 锅炒的」当成「用 B 锅炒的」端出去。所以下面 key 的每一项都不能省。

三层缓存精确定义（对照真实代码）Learn

三层各缓存什么、key 是什么、命中省下什么——全部对照 mini_vllm/multimodal/cache.py：

缓存层缓存的内容key 包含命中省下
ProcessorCache预处理后的像素（resize+normalize）、抽帧、grid metadata 媒体内容 hash + processor 配置（image_size / mean / std）重复的预处理（便宜）
EncoderOutputCache视觉编码 + projector 的输出（投影后的视觉 embedding） 上面全部 + encoder 身份 + projector 身份 + dtype + schema 版本整个 vision encoder + projector（最贵）
LLM KV Cache（Lesson 7）文本模型各层的 K/V（多模态 prefill 后的上下文） （每条序列各一份，随 token 增长）解码时重算历史 K/V
EncoderOutputCache 的 key 为什么要多这么多项？（stale cache 的根源）
cache.py 里 encoder key = content_hash(pixels) + processor_cfg + encoder_id + projector_id + dtype + SCHEMA_VERSION。 其中 encoder_id 形如 "tinyvis-s4321-h48-l1"（含随机种子、维度、层数）， projector_id 形如 "proj-s4321-t32"。任何一项变了，key 就变，绝不会误命中旧结果—— 这正是防 stale cache 的关键。ProcessorCache 只到「像素+预处理配置」即可，因为它不涉及编码模型。

小例子：同一张图两次到来，省了多少 Learn

用 run 27 会看到的真实过程（同一张 seed=7 的图）：

时刻EncoderOutputCachevision encoder 是否运行
第 1 次编码这张图miss（key 不在表里）运行 1 次，并把结果 put 进缓存
第 2 次编码同一张图hit（key 命中）跳过，直接取投影后的视觉 embedding
换 encoder 身份后再来miss（encoder_id 变 → key 变）重新运行（不会误命中旧的）

content_hash 是像素 JSON 的 sha256 取前 16 位十六进制——它保证「同图必同 key、异图必异 key」（run 27 与 tests.lesson_27 都验证了这一点）。缓存底层是 LRU（容量满就逐出最久未用），并统计 hits/misses/evictions。

常见误解 Learn

误解为什么错
「三层缓存差不多，用一个就行」它们缓存不同层级的半成品（预处理像素 / 视觉 embedding / 文本 K/V），互不能替代。
「缓存 key 用图片内容就够了」视觉 embedding 还取决于 encoder/projector/dtype/schema；只看内容会命中用错模型的旧结果（stale cache）。
「视觉 embedding 命中了，KV 也能省」每条请求的文本上下文不同，LLM KV Cache 是各序列独立的，不能跨请求顶替。
「命中缓存会让输出变化一点」缓存的是精确的中间量；命中只加速、不改变结果（前提是 key 精确匹配）。

代码映射 Learn

ProcessorCache.key(pixels, cfg)            = "proc|" + content_hash(pixels) + "|" + cfg
EncoderOutputCache.key(pixels, cfg, enc_id, proj_id, dtype)
        = "enc|" + content_hash + "|" + cfg + "|" + enc_id + "|" + proj_id + "|" + dtype + "|" + SCHEMA_VERSION
content_hash(pixels) = sha256(json(pixels))[:16]     # 同图必同、异图必异
# 底层 _LRU：get()/put()，容量满逐出最久未用，统计 hits/misses/evictions
