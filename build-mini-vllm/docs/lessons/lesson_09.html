为什么需要这个知识 Learn

到这里我们已经能单条地把一个请求跑完（Lesson 8 的 prefill/decode）。 但真实服务里会同时来很多请求——要想让 GPU 一次算多条、把算力吃满，就得把它们凑成 一个「批」（batch）。凑批最朴素的办法叫静态批处理（static batching）：把这批请求 一次性打包、一起算、一起等到全部结束。它简单、够用，但会造成大量浪费。

这一课不写新模型代码，而是把静态批处理的浪费量化出来：它到底浪费在哪、 浪费多少、和什么因素成正比。想清楚这一点，你才会真正理解下一课 continuous batching（Lesson 10） 为什么值得——它正是为了消掉这里的浪费而生。所以本课的重点是看懂一个数字：waste。

先看问题：静态批处理到底浪费了什么 Learn

假设同一时刻来了 4 个请求，各自要处理的 token 总数（prompt + 要生成的 output）不一样。 用本课动画里的真实数字：

请求真实长度（prompt+output）什么时候本该结束
r06第 6 步就该完事
r120第 20 步
r24第 4 步（最短）
r324第 24 步（最长）

静态批处理要求「一批一起走」：为了塞进同一个规整的张量，所有请求都被 padding 到批内最长者的长度 24；然后这一批一起跑满 24 步， 直到最慢的 r3 结束才整批收工。

可是 r2 在第 4 步就该停了，之后的 24 − 4 = 20 步全在陪跑——它要么在算无意义的 padding token，要么已经生成完了却还占着 slot 空转。r0 陪跑 24 − 6 = 18 步， r1 陪跑 24 − 20 = 4 步。把四条的陪跑加起来：18 + 4 + 20 + 0 = 42 步的计算 纯属浪费。这就是本课要盯住的东西。

直觉类比 Learn

一句话解释
静态批处理 = 把一批长短不一的请求补齐到最长、绑在一起跑， 等最慢的那个结束整批才放行；短请求多出来的时间全是浪费。

类比：拼车必须等最后一个人下车。四个人拼一辆车，目的地远近不同。 最近的人（r2）本该第 4 站就下，可这辆车得开到最远乘客（r3）的第 24 站才算这趟结束、才能接下一批客。 近处的乘客只能坐在车上干等——车照样在开、油照样在烧，却没把他送到任何新地方。 乘客长短差越大，这种「陪坐空跑」就越多。

类比的边界（别被带偏）
拼车里「等最慢的人」浪费的是时间；这里浪费的是 GPU 的计算： 每一步都要把整批（含已完成 / padding 的 slot）过一遍模型的前向。另外别把它和 Lesson 7 的 KV Cache 混了——KV Cache 省的是单条请求内部的重复计算， 而这里浪费的是请求之间「快的等慢的」。两者是不同维度的浪费。

精确定义：浪费公式，逐符号拆开 Learn

把成本用「token-步」来记（处理一个 token 过一次前向 = 1 个单位）。设批里有 bs 个请求， 第 i 个的真实长度是 Lᵢ，批内最长为 max_len = max(Lᵢ)。三个关键量：

static_cost=bs×max_len # 静态：每个 slot 都被拉到 max_len，跑满 continuous_cost= ∑Li # 理想：只为真实 token 付费 waste=static_cost−continuous_cost waste_pct= wastestatic_cost ×100%
  • bs：批大小（这一批打包了几个请求）。本例 bs = 4。
  • Lᵢ：第 i 个请求的真实长度 = prompt_len + output_len。本例 [6, 20, 4, 24]。
  • max_len：批内最长长度，也就是 padding 的目标。本例 24。
  • static_cost = bs × max_len：把整批想成一个 bs × max_len 的规整方块，每个格子都要算。
  • continuous_cost = Σ Lᵢ：只有真实 token 才是「必要」的计算。
  • waste：方块里没被真实 token 填满的部分——就是 padding / 已完成 slot 的空算。
输入 / 输出 shape（用具体数字）
tokenizer.pad(batch) 把这批补齐后，返回的 input_ids 形状是 (bs, max_len) = (4, 24)，attention_mask 同形 (4, 24)。整批前向就是在这个 4 × 24 = 96 的方块上算，而真实内容只有 Σ Lᵢ = 54 格。

r3 是「拖后腿的最长者」——它一个人把整批的 max_len 撑到 24，别人全被迫陪它跑满。 去掉 r3，max_len 立刻降到 20，浪费大减。这就是「一个长请求毁掉一整批」的直观样子。

attention mask：padding 位置怎么被「屏蔽」 Learn

补齐用的是特殊 token PAD（本课词表里 PAD_ID = 258；词表共 vocab_size = 259 = 256 个字节 + BOS/EOS/PAD）。但 padding 只是「占位凑长度」， 绝不能让模型把它当成真内容去注意——否则会污染真实 token 的输出。attention mask 就是那张「哪些位置是真的、哪些是填充」的标记表：1 = 真实 token，0 = padding， 下游 attention 用它把 0 的位置屏蔽掉（打到 -inf，softmax 后权重为 0）。

用本课 tokenizer 的真实字节来手算一批两条：

  • "hi" → UTF-8 字节 [104, 105]（h=104, i=105），长度 2。
  • "hello!" → [104, 101, 108, 108, 111, 33]，长度 6。

右侧 padding 到 max_len = 6 后，pad() 返回 shape 都为 (2, 6) 的两张表：

位0位1位2位3位4位5
ids · "hi"104105258258258258
mask · "hi"110000
ids · "hello!"10410110810811133
mask · "hello!"111111
别忘了 mask，否则结果是错的
mask 保证「正确性」，但它不能省掉计算：被屏蔽的 padding 位置照样在张量里、照样过一遍前向， 只是结果被丢弃。所以 mask 解决的是「算对」，而本课的 waste 说的是「算了却没用」—— 这两件事要分开理解。中文一个字 = 3 个 UTF-8 字节 = 3 个 token，所以中文请求更容易长短不齐， padding 浪费也更明显。

常见误解 Learn

误解为什么错
「加了 attention mask，就不浪费计算了」mask 只保证 padding 不影响结果；那些位置照样被算，算力照花。mask 治的是「算错」，不是「浪费」。
「短请求早就生成完了，就不占算力了」整批绑在一起跑满 max_len 步。已完成的 slot 只能空转陪跑到最慢者结束，不能中途下车。
「浪费只和批大小有关，批越大越浪费」浪费的根源是长度差异：waste = bs×max_len − ΣLᵢ。若所有请求等长，waste = 0，再大的批也不浪费。
「把一个超长请求和一堆短请求凑一批更划算」恰恰相反：一个长请求把 max_len 撑高，逼所有短请求陪它跑满，浪费飙升（见执行图里的 r3）。
「continuous_cost 是另一种真实的批处理模式」本课的 continuous_cost = ΣLᵢ 只是「理想下限」，用来对比。真正做到它需要 Lesson 10 的 continuous batching。

代码映射 Learn

本课两处真实代码，正好对应上面两条主线：

  • 浪费的量化 → experiments/lesson_09_static_batching.py 的 run_experiment()：lens = [prompt_len+output_len …]、 static_cost = bs × max_len、actual_cost = sum(lens)、 waste = static_cost − actual_cost——和上面手算的公式一字不差。
  • padding 与 mask → mini_vllm/tokenizer.py 的 ByteTokenizer.pad(batch)：把每条右侧补 PAD_ID(258) 到 target = max(len)，同时生成 attention_mask（真实位 1、 padding 位 0），返回 (input_ids, attention_mask)。实验里就是用 tok.pad([encode("hi"), encode("hello!")]) 打印出上表那个 mask 示例的。
