为什么需要这个知识 Learn

到 Lesson 27 为止，多模态数据流已经跑通：图片/视频经 vision encoder 变成视觉 embedding，插进文本序列里一起 prefill。但有一件事一直被忽略——视觉这部分有多贵， 调度器完全没算进去。前面所有的连续批处理（Lesson 20）、 预算准入都只数「文本 token」。多模态一来，这套账本就漏了两大笔开销： visual token 占的序列/KV 位置，和 运行 vision encoder 本身的算力。

这一课要做的，就是把这两笔开销补进调度预算，让引擎在「接纳一个新的多模态请求」之前， 先掂量清楚它到底要吃掉多少视觉资源。核心只有一个数据类 MultiModalBudget 和它的一个方法 fits()——但要真正理解它，得先看看「不这么做会怎样」。

先看问题：只数文本 token 的调度会怎样 Learn

先记住本课两组真实数字。文本侧用 byte-level tokenizer：一个 UTF-8 字节 = 1 个 token， 一个中文字 = 3 字节 = 3 个 token。视觉侧（VisionConfig）： image_size=16、patch_size=8 → grid 2×2 → 每张图 num_patches=4 个 visual token； 视频按帧编码，n 帧 = n×4 个 visual token。

现在设想一个请求："看这段视频"（5 个中文字 = 5×3 = 15 个文本 token）附带一段 8 帧的视频。它的真实 prefill 成本是：

成本项数值只数文本的调度看到的
文本 token1515 ✓（唯一被算到的）
visual token（占序列位置、进 KV）8 帧 × 4 = 320（漏算！）
prefill 一次要处理的序列长度15 + 32 = 4715
vision encoder 运行次数逐帧编码 = 8 次0（漏算！）

模型 max_seq_len=64：这一条请求的视觉部分（32）就吃掉了半条序列。若调度器只看到「15 个文本 token，很便宜」， 一口气在同一次迭代里准入 3 条这样的请求，实际这一步要处理 3×47 = 141 个 token、 同时运行 3×8 = 24 次 vision encoder——延迟尖峰、编码器激活值 OOM，系统被压垮。 问题的根源：账本里没有「视觉」这一栏。

visual token 不是「虚」的——它和文本 token 一样进 KV
沿用 Lesson 7 的内存公式，本模型每个 token 每层的 KV = 2·num_kv_heads·head_dim = 2·2·8 = 32 个数，2 层共 64 个数。一张图 4 个 visual token = 4×64 = 256 个数的 KV，一段 8 帧视频 = 32×64 = 2048 个数。visual token 是实打实的序列位置和显存开销，理应计入预算。

直觉类比 Learn

一句话解释
MultiModalBudget = 给调度器加一份更全的账本：除了文本 token，还单独记 「视觉 token」和「本次迭代已经运行了多少次 encoder」，任何一栏超支就让新请求再等一步。

类比：餐厅的三口锅。后厨同时受三样东西限制：备菜台的面积（视觉 token 预算，摆不下就先不接单）、 烤箱的位置数（encoder 预算，一次只能烤这么多份）、以及服务员能同时盯的桌数（max_num_seqs）。 领班（调度器）收到新客人前，会逐一核对这几样有没有空位；缺任意一样，就请客人在门口稍等一轮， 绝不硬塞。

类比的边界（别被带偏）
「等一轮」只是推迟，不是拒绝，更不改变任何计算结果——预算只决定「何时算」， 不决定「算成什么」。同一批请求，无论预算松紧，最终每个请求生成的 token 逐值一致； 松预算只是让它们更早、更并行地被处理完。此外，账本上的四栏是各自独立的闸门， 不是加权求和成一个总分——任一栏满了就挡住，这点下面精确定义里会看清。

精确定义：四条预算与 fits() Learn

MultiModalBudget 是一个只有四个上限字段的数据类（括号内是本课默认值）：

字段约束什么默认值类比
text_token_budget一次迭代处理的文本 token 上限64—
visual_token_budget一次迭代处理的 visual token 上限64备菜台面积
encoder_budget一次迭代运行 vision encoder 的媒体数上限4烤箱位置数
max_num_seqs并发 running 请求数上限4能盯的桌数

准入判断由 fits() 完成。它接收「这个请求要新增多少」和「本轮已经用掉多少」，逐项相加比上限， 全部通过才返回 True：

fits(text_tokens, visual_tokens, num_media_to_encode,
     used_text,   used_visual,   used_encoder)  →  bool

  return (used_text    + text_tokens          <= text_token_budget
      and used_visual  + visual_tokens        <= visual_token_budget
      and used_encoder + num_media_to_encode  <= encoder_budget)

逐符号看清楚：

  • text_tokens / visual_tokens：这个候选请求要新增的文本 / 视觉 token 数。视觉数由 runner.estimate() 在不做真正编码的前提下算出——图 = 4，视频 = 帧数×4。
  • num_media_to_encode：这个请求含几个媒体段（一张图算 1，一段视频也算 1）。
  • used_*：本轮迭代到目前为止，前面已被准入 / 已 decode 的请求累计用掉的量。
shape / 数值：一张图的候选参数
请求 "描述这张图"（5 字 = 15 文本 token）+ 1 张图： estimate() 返回 {visual_tokens: 4, num_media: 1}。 引擎在准入时调用的是 fits(0, 4, 1, used_text, used_visual, used_encoder)—— 注意第一个参数（prefill 的文本 token）传的是 0：本课的简化是文本预算只在 decode 阶段计数 （每个 running 请求每步 +1），prefill 的文本 token 不占 text_token_budget； 真正 gate 多模态准入的是 visual_token_budget 与 encoder_budget。
fits() 不管 max_num_seqs
第四条上限 max_num_seqs 不在 fits() 里，而在引擎准入循环开头单独判断： if len(self.running) >= self.budget.max_num_seqs: break。所以「四条闸门」里三条在 fits()、一条在 run()——四条都是独立的与（AND），任一挡住就不准入。

小规模手算：三条请求在松/紧预算下的准入 Learn

设第 1 次迭代开头有三条等待请求 R1、R2、R3，每条都含 1 张图，即 estimate = {visual_tokens: 4, num_media: 1}。此刻 running 为空， used_visual = used_encoder = 0。引擎按到达顺序逐个尝试准入，每准入一个就累加已用量。 下表对比四种预算设置下，这一轮各自能接纳谁：

预算设置R1R2R3结果
默认（visual 64 / encoder 4 / seqs 4） visual 0+4=4✓ enc 0+1=1✓4+4=8✓ 1+1=2✓8+4=12✓ 2+1=3✓三条全准入
紧 encoder = 2 enc 0+1=1✓1+1=2✓2+1=3>2 ✗R1,R2 入；R3 等下一轮
紧 visual = 8 visual 0+4=4✓4+4=8✓8+4=12>8 ✗R1,R2 入；R3 等下一轮
紧 max_num_seqs = 2 running 0<2✓running 1<2✓running 2不<2 → breakR1,R2 入；R3 等下一轮

关键观察：同样三条请求，只因预算不同，一轮里被接纳的数量就不同。被挡下的 R3 不是被拒绝—— 下一次迭代 used_* 归零、running 也可能腾出位置，它就会被准入。这正是本课要建立的直觉： 预算决定「谁先、谁后」，不决定「结果」。

常见误解 Learn

误解为什么错
「visual token 是编码出来的中间结果，不占序列，不用进预算」 visual token 插进文本序列后和文本 token 完全平权：占位置、进 KV、每层要存 2·2·8=32 个数。不计入预算，长视频（32 个 visual token）会悄悄撑爆序列。
「有了 visual token 预算，就不需要单独的 encoder 预算了」 两者约束的是两种不同资源：visual token 预算管「序列/KV 位置」，encoder 预算管「运行 vision encoder 的算力」。一张缓存命中的图 visual token 照占，但可能省掉编码；它们必须分开记账。
「预算调紧会改变模型输出 / 让某些请求被丢弃」 预算只影响准入时机：被挡下的请求下一轮重试，最终照常完成，输出逐值一致。紧预算只是更慢/更少并行，不丢请求、不改结果。
「四条上限会加权合成一个总分来判断」 没有加权、没有总分。fits() 是三条独立的 AND（再加 run() 里的 max_num_seqs），任意一条超限就整体不通过。
「encoder 预算按帧数算，8 帧视频占 8 份 encoder 额度」 本课 estimate() 按媒体段计数：一段视频 num_media=1（尽管它内部逐帧编码）。帧数的膨胀由 visual_token_budget 捕捉（8 帧 = 32 个 visual token），不是 encoder 预算。
