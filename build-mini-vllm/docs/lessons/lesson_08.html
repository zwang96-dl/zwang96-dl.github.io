为什么需要这个知识 Learn

上一课（Lesson 7）我们让 decode「只喂 1 个 token」。可这一步之前， 模型必须先把整段 prompt 读进去——这就是 prefill。一次生成因此天然分成两段： 先一口气吃完 prompt（prefill，产出第一个 token）， 再一个一个往外吐（decode）。这两段的「性格」完全不同，用一个「平均延迟」去描述它， 就像用「平均体温」描述一个上午发烧、下午退烧的人——数字对了，判断全错。

跑 run 8，会看到类似下表的输出（毫秒是墙钟时间，每次运行、每台机器都不同， 下面是某次的示例数量级，你的机器上会有出入）：

场景prompt_lenTTFT（首 token，示例）TPOT（每 decode token，示例）
prompt-heavy（长 prompt）56~40 ms~2.4 ms
decode-heavy（短 prompt）3~3 ms~1.9 ms

盯住两列：prompt 从 3 涨到 56（约 19 倍），TTFT 跟着大约从 3 ms 涨到 40 ms（同数量级的十几倍）； 而 TPOT 几乎没动。这说明——prompt 的长短几乎只影响首 token，之后逐字吐的速度基本恒定。 如果你不把这两段拆开看，就永远解释不了「为什么同样是慢，有的请求是开头卡、有的是全程慢」， 更谈不上后面的批处理、调度、chunked prefill——它们本质都是在 TTFT 与吞吐之间做权衡。 所以 TTFT / TPOT / ITL 是整个推理服务的通用语言，这一课就是把这门语言学扎实。

先看问题：一次生成的「工作量」长什么样 Learn

本课 tokenizer 是 byte-level：每个 UTF-8 字节 = 1 个 token id（0..255）， 所以中文一个字 = 3 字节 = 3 个 token。取 prompt "你好"： 2 字 × 3 字节 = 6 个 token，再加句首 BOS 共 7 个 token。假设要生成 4 个新 token。 用 KV Cache 路径，这次生成被拆成 5 次 forward 调用：

阶段forward 次数每次喂进的 token 数 input_len处理的 token 数产出
prefill1 次7（整段 prompt）7第 1 个 token
decode3 次1 / 次1 / 次第 2、3、4 个 token

关键观察：第一次 forward 就处理了 7 个 token，占全部处理量（7+3=10）的 70%，而且它全部发生在用户看到任何输出之前。 这块「又宽又重、只跑一次」的工作，就是 TTFT 的来源；后面「又窄又轻、跑很多次」的工作，决定逐字吐字的手感（TPOT/ITL）。 一次生成的工作量画像，天生就是「1 个胖块 + N 个瘦块」。

直觉类比 Learn

一句话解释
prefill = 把整段 prompt 一次读进模型、算好所有历史 K/V 并产出第 1 个 token； decode = 之后每次只算 1 个新 token。TTFT 量的是 prefill 有多快， TPOT / ITL 量的是 decode 逐字吐得有多稳、多快。

类比：餐厅上菜。 你落座点了一大桌菜——厨房要先备料、开火（prefill）， 这段时间你干等着，直到第一道菜端上桌（TTFT）。之后菜就一道一道地上 （decode，每道之间的间隔就是 ITL），上菜的平均节奏就是 TPOT。点得越多（prompt 越长）， 备料越久 → 第一道菜来得越慢；但一旦开始上菜，出菜节奏跟你点了多少关系不大。

类比的边界（别被带偏）
备料只做一次，之后端菜是「取现成的」——这点很像。但别忘了：decode 每上一道菜， 都要回头翻一遍前面所有菜的记录（读整块 KV 缓存），序列越长这一步读得越多。 所以「上菜节奏恒定」只在上下文不太长时成立；真正长文本里，后期 decode 会因 KV 越读越多而变慢—— 这正是 decode 是 memory-bound 的根源（见下文「两种瓶颈」）。

四个指标：精确定义 Learn

把一次生成的时间线记成一串时刻。设请求在 t₀ 到达，第 k 个 token 在时刻 t_k 产出（k = 1,2,…,N，本例 N = prompt 后新生成的 token 总数）。四个指标逐符号如下：

指标公式（逐符号）符号含义单位 / shape
TTFT
Time To First Token		t₁ − t₀t₁=第 1 个 token 产出时刻，t₀=请求到达时刻。≈ prefill 一次 forward 的耗时。1 个标量（ms）
ITL
Inter-Token Latency		ITL_k = t_k − t_{k−1}
（k ≥ 2）		相邻两个输出 token 的间隔，即每个 decode 步的耗时。长度 N−1 的数组
TPOT
Time Per Output Token		(1/(N−1)) · Σ ITL_k
= (t_N − t₁)/(N−1)		所有 decode 步耗时的平均（ITL 数组的均值）。1 个标量（ms）
ThroughputN / (t_N − t₀)单位时间产出的 token 数（多请求时看总 token / 总时间）。token/s

三者的关系一步就能推出——端到端延迟 = TTFT + 所有 ITL 之和 = TTFT + (N−1)·TPOT。 所以「首 token 快不快」看 TTFT，「后面吐字顺不顺」看 TPOT，两者共同决定总时长。

TTFT vs TPOT vs ITL：一句话分清
ITL 是逐步的原始序列（数组），TPOT 是它的平均（标量）， TTFT 单指第一个（prefill 那一下）。三者量的是同一条时间线上的不同段落。

小规模手算：从时间线算出四个指标 Learn

沿用上面的例子：prompt "你好"（7 个 token）、生成 4 个新 token。假设我们用 time.perf_counter() 量到每一步的耗时 dt 如下（prefill 是 step 0）：

stepphaseinput_lendt（本步耗时）这是什么指标
0prefill76.0 ms→ TTFT = 6.0
1decode12.1 msITL₂ = 2.1
2decode11.9 msITL₃ = 1.9
3decode12.0 msITL₄ = 2.0

一步步套公式：

  • TTFT = step 0 的 dt = 6.0 ms。（就是 prefill 那一下）
  • ITL 数组 = decode 各步 dt = [2.1, 1.9, 2.0]，长度 N−1 = 3。
  • TPOT = (2.1 + 1.9 + 2.0) / 3 = 2.0 ms。
  • 端到端延迟 = 6.0 + 2.1 + 1.9 + 2.0 = 12.0 ms，也等于 TTFT + (N−1)·TPOT = 6.0 + 3×2.0。
  • Throughput = 4 token / 12.0 ms ≈ 333 token/s。

两种瓶颈：compute-bound 与 memory-bound Learn

为什么 prefill 和 decode 快慢的「卡点」不一样？关键看每一步里 「搬进来多少数据」对「做了多少次乘加」 的比值（业界叫算术强度，arithmetic intensity）。 模型权重（比如本课 FFN 是 SwiGLU，含 gate/up/down 三个 32×64 = 2048 参数矩阵）无论处理几个 token，都得完整读一遍。 区别在于：读进来之后，这些权重被用来算了几个 token。

prefill（7 个 token）decode（1 个 token）
一次 forward 处理的 token71
FFN 权重（2048 参数/矩阵）读入次数1 次，被 7 个 token 复用1 次，只服务 1 个 token
每读一遍权重摊到的算力高（÷7 更划算）低（只算 1 个，权重白读大半）
还要额外读的东西—整块 KV 缓存（每层每 token 32 个数，随上下文线性增长）
典型卡点算力（compute-bound）内存带宽（memory-bound）
一句话记住
prefill 把一堆 token 塞进一次矩阵乘，芯片的乘加单元被喂得很饱 → 卡在算力（compute-bound）； decode 每步只算 1 个 token，却要把整块 KV 缓存和全部权重从显存搬进来， 乘加单元大部分时间在等数据 → 卡在内存带宽（memory-bound）。这就是为什么给 decode 堆更多算力几乎没用， 而把多条 decode 请求拼成一个 batch（让一次搬进来的权重同时服务多个 token）才是提升吞吐的正道——正是后续批处理课的动机。
重要：本课的毫秒数是「玩具计时」，不代表 GPU 行为
compute-bound / memory-bound 是真实世界的正确框架；但本课跑在纯 Python + CPU 上, run 8 打印的那几毫秒只是 Python 循环的开销——它能证明的只是「prefill 处理的 token 比 decode 多得多,所以 TTFT 明显大于 TPOT」。 它无法体现真实 GPU 上「decode 受内存带宽约束」这件事(纯 Python 没有显存搬运、没有 kernel、没有带宽瓶颈)。 所以请把这些数字当作「token 数量差异的直观展示」,而不是「decode 是 memory-bound 的实测证据」——后者需要在真实 GPU 上用 profiler 才能看到。

常见误解 Learn

误解为什么错
「TTFT 和 TPOT 都会随 prompt 变长」只有 TTFT 随 prompt 变长（prefill 要处理更多 token）；TPOT 基本不变——decode 每步永远只处理 1 个 token。看 run 8：prompt 从 3→56，TTFT 3→40 ms，TPOT 却几乎不动。
「decode 慢是因为算力不够，加卡/加算力就好」decode 是 memory-bound：乘加单元在等 KV 缓存和权重搬进来，算力大多闲着。堆算力无用，攒 batch 提高数据复用才有效。
「TPOT 就是 ITL」ITL 是逐步的数组（每个 decode 步一个值），TPOT 是这个数组的平均标量。长上下文里 ITL 会随 KV 增大而缓慢上升，TPOT 只是把它们抹平成一个数。
「prompt 长只是自己多等会儿，不影响别人」一个超长 prompt 的 prefill 会长时间占住计算，把排在后面的短请求的首 token 也拖慢（TTFT 被别人的 prefill 卡住）。这正是 chunked prefill（把长 prefill 切块、和 decode 交错）要解决的问题。
「吞吐高就等于延迟低」两者常常相互拉扯：把更多请求塞进一个 batch 能拉高 throughput，但每条请求的 TTFT/TPOT 可能变差。服务端要在两者间按 SLA 取舍。

代码映射 Learn

本课的指标全部落在 mini_vllm/engine/generate.py 里。每步都记一条 StepRecord（含 phase 和耗时 dt），指标只是对这些记录做聚合：

# generate_cached()：prefill 记为 step 0（phase="prefill"），耗时进 steps[0].dt
t0 = time.perf_counter()
logits = model.forward(list(prompt_ids), list(range(len(prompt_ids))), cache, tracer)
...
res.steps.append(StepRecord(0, "prefill", len(prompt_ids), ...))   # ← TTFT 的来源

# 之后每个 decode 步只喂 1 个 token，各自计时
for step in range(1, max_new_tokens):
    t0 = time.perf_counter()
    logits = model.forward([tok], [cur], cache)   # input_len = 1
    ...
    res.steps.append(StepRecord(step, "decode", 1, ...))          # ← 每条的 dt 就是一个 ITL
  • GenResult.ttft → 直接返回 steps[0].dt，即 prefill 那一步的耗时。
  • GenResult.decode_times → 挑出所有 phase == "decode" 的 dt，就是 ITL 数组。
  • GenResult.tpot → sum(decode_times)/len(decode_times)，即 ITL 数组的均值。
  • generate_cached() → 把「1 次 prefill + N−1 次 decode」的结构真正跑出来： prefill 处理 prompt_len 个 token，decode 每步 input_len=1。这正是前面手算表格的代码版。

本节总结 & 下一步

  • prefill 一次处理整段 prompt（→ TTFT，compute-bound）；decode 每步 1 token（→ TPOT/ITL，memory-bound）。
  • prompt 越长 TTFT 越大；decode 速度相对稳定。
  • 这些指标是后续批处理与调度的优化目标。
