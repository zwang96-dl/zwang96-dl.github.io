为什么需要这个知识 Learn

前面 9 课（Lesson 21–29）我们把多模态引擎一块砖一块砖地砌起来：chat template、 image processor、vision encoder、embedding 合并、占位对齐、视频抽帧、三层缓存、调度预算…… 每一课都单独验证过「这块砖是好的」。但生产系统的真正考验是—— 把所有砖同时受力：多个请求并发、有图有视频、共享媒体、预算有限， 此时六条正确性必须同时成立，缺一条就是线上事故。

不这么综合自检会怎样：一个真实的连环事故
假设两个请求几乎同时到达：请求 A 带图 cat.png，请求 B 带图 dog.png。 如果 encoder 缓存的 key 只用了「图像内容」而漏了 encoder 身份， 某次热更新换了 encoder 权重后，B 会误命中 A 遗留的旧视觉 embedding（stale cache）； 与此同时，如果两个请求共用了同一份 visual_embeds_list，B 的图还会串进 A 的序列； 再叠加占位符没对齐（图要 4 个 visual token，却只插了 3 个占位）， 合并 embedding 时下标全部错位。单独测每块砖时这些都不会暴露—— 只有把它们塞进同一个并发场景，才逼得出来。这就是 Lesson 30 存在的意义。

六条不变量：一句话 + 直觉 Learn

本课的「知识」其实是一张验收清单：一个正确的多模态引擎，在任意 workload 下都必须 同时守住下面六条不变量（invariant，即「无论怎么跑都不能被破坏的性质」）。

#不变量一句话被破坏时的症状
1对齐每段媒体的占位 token 数 = 它的 visual token 数，顺序/区间严格对应视觉 embedding 插错位置，答非所问
2无串用每个请求各持一份 KV Cache 与视觉 embedding，互不共享可变状态A 的图影响 B 的输出
3无 stale cache缓存 key 覆盖所有影响结果的因素（内容+身份+版本+dtype）换了 encoder 却命中旧结果
4timestamp 保留视频抽帧的时间线 metadata 一路透传到请求上，不丢模型说不出「第几秒发生了什么」
5visual 预算visual token 与 encoder 工作量都计入调度预算一张大图撑爆一次迭代，别的请求饿死
6encoder 不重复vision encoder 只在 prefill 跑一次，decode 阶段绝不再跑每个 decode step 重编码，TTFT/吞吐崩塌
直觉类比：机场的一次值机
把一个请求想成一位带行李的旅客。对齐=每件行李都贴对了行李条（占位）； 无串用=你的行李不会被推到别人的推车上； 无 stale cache=安检机认「这件行李+这台机器的当前程序」，换了程序就重扫，不复用旧扫描图； timestamp 保留=行李条上的时间戳一路跟到目的地； 预算=每条安检线每分钟能过的件数有限； encoder 不重复=安检只在入口扫一次，登机口不再重扫。
类比的边界（别被带偏）
行李是「物理唯一」的，天然不会串；而视觉 embedding 只是内存里的一份 list， 只要代码不小心共享了同一个引用，就会真的串——所以「无串用」不是自动成立的， 是靠「每个请求 build_inputs 各自新建 list、各自 KVCache(cfg)」这些具体写法换来的。 另外，安检「扫一次」是硬件决定的，而「encoder 不重复」在我们这里是用一句 assert 主动守护的 （见下文代码映射），不是白来的。

接着算各项资源账：

  • 视觉预算：estimate() 不做编码就报 visual_tokens = 2×4 = 8、num_media = 2。 默认预算 visual_token_budget=64、encoder_budget=4， fits 判断 8≤64 且 2≤4 → 准入通过。
  • encoder 运行次数：冷缓存时 2 张图各编码一次 → encoder_runs += 2； decode 阶段 encoder 一次都不能再跑（下节 assert 守护）。
  • KV Cache 增长：每 token 每层存 K+V = 2·kv_heads·head_dim = 2·2·8 = 32 个数， 2 层共 64 个数/token。prefill 一次性写入 19×64 = 1216 个数；之后每 decode 一步 +64。
  • merge 维度校验：视觉 embedding 是 (4, 32)，文本 hidden 也是 32——一致才允许逐位替换； 若 projector 忘了投到 32，这里会直接抛错（stale/错配的第一道防线）。

逐条拆解：每条不变量守在代码哪里 Learn

本课不写新代码，重在读懂已有引擎如何守住六条不变量。下表把每条不变量精确定位到 mm_engine.py 及其协作模块的具体位置：

不变量守在哪里（类/函数）怎么守的
1 对齐placeholders.validate_placeholders()（被 merge_multimodal_embeddings 调用）校验 len(ranges)==媒体数、每段 length==visual token 数、media_index==k、不越界、不重叠；任一不满足抛带定位信息的错。
2 无串用MultiModalEngine._prefill()（r.kv = KVCache(cfg)）+ runner.build_inputs()每个请求在 prefill 时新建自己的 KV Cache；visual_embeds_list 每次调用都是新 list，不存在跨请求共享的可变引用。
3 无 stale cachecache.EncoderOutputCache.key()key = 内容哈希 + processor_cfg + encoder_id + projector_id + dtype + SCHEMA_VERSION；换任一因素都不会误命中。
4 timestamp 保留runner._encode_video()→build_inputs()→_prefill()（r.media_meta）抽帧时 VideoFrameSampler.timeline() 产出 {frame_index, timestamp}，塞进 media_meta[k]["timeline"]，再随 prefill 结果挂到请求上，全程不丢。
5 visual 预算budget.MultiModalBudget.fits()（准入循环 run() 第 2 阶段）+ runner.estimate()estimate() 不编码就算出 visual token 与媒体数；fits() 同时约束 visual_token_budget 与 encoder_budget，超预算的请求这一迭代不准入。
6 encoder 不重复MultiModalEngine.run() decode 阶段的 assert encoder_runs == enc_beforedecode 只调 model.forward([tok],[cur])（纯文本）；每步前后记录 runner.encoder_runs，若 decode 期间它变了立即 assert 失败。
为什么用「主动 assert」而不是「相信它不会跑」
第 6 条尤其关键：encoder 是最贵的一步，一旦被 decode 循环误触发，性能会悄悄崩塌而 结果却依然正确（只是慢），普通功能测试根本发现不了。所以引擎在每步 decode 前后 对 encoder_runs 做快照对比（enc_before → assert），把「性能不变量」变成一个 会当场爆炸的断言——这是把隐性性能 bug 变成显性错误的经典手法。

首次 vs 缓存命中：TTFT 差在哪 Learn

TTFT（Time To First Token，首 token 延迟）= 请求准入到吐出第一个 token 的时间。 多模态请求的 TTFT 大头在 prefill，而 prefill 里最贵的是 vision encoder。 用上面那条 2 图请求对比两种情形：

情形encoder 运行prefill 成本主要来自TTFT
首次请求（冷缓存）2 次（2 张图各编码一次）预处理 + 2×(encoder+projector) + 文本 prefill高
相同媒体再来（EncoderOutputCache 命中）0 次只剩文本 prefill（视觉 embedding 直接取缓存）明显更低

注意命中的前提是 key 完全一致（同一像素内容 + 同一 encoder/projector 身份 + 同 dtype/版本）。 这恰恰把第 3 条（无 stale cache）和 TTFT 联系起来：key 太粗会误命中（错但快）， key 太细会漏命中（对但慢）——正确的 key 粒度是「覆盖所有影响结果的因素、且仅覆盖这些」。

常见误解 Learn

误解为什么错
「每块砖单独测过了，合起来自然没问题」bug 往往出在交互处：并发下的共享状态、缓存 key 粒度、预算与编码的耦合——这些只有综合场景才暴露，正是本课存在的理由。
「decode 时也要重新看一眼图，才知道图里有什么」图的信息在 prefill 已经编码进 KV Cache 了。decode 是纯文本自回归，重跑 encoder 既多余又昂贵——引擎用 assert 明令禁止。
「encoder 缓存 key 用图像内容哈希就够了」换 encoder/projector 权重或改 dtype 后，同一张图的视觉 embedding 会变。key 不含身份/版本就会误命中旧结果（stale cache），静默出错。
「占位 token 数随便填，反正会被视觉 embedding 覆盖」占位数必须精确等于 visual token 数。少了则视觉行没处放/越界，多了则多余占位没被替换、混入垃圾 embedding。validate_placeholders 会当场拦下。
「视频抽了帧，模型自然知道每帧是第几秒」timestamp 只是 processor 侧的 metadata，模型看不到它，除非显式喂进 prompt 或随 media_meta 保留下来供后续使用。抽帧≠保留时间线。
「visual token 不占序列位置，不用进预算」visual token 会被 parser 插成真实占位、进入序列和 KV Cache，和文本 token 一样占位置。所以 estimate() 要把它们算进调度预算，否则一张大图能撑爆一次迭代。

代码映射：读懂 run() 的一次迭代 Learn

把上面六条不变量落到 MultiModalEngine.run() 的主循环上——一次迭代做三件事， 每件事恰好对应几条不变量：

while self.waiting or self.running:
    # ① decode 阶段：推进已 prefill 的请求（纯文本）
    enc_before = self.runner.encoder_runs
    for r in self.running:
        if r.prefill_done and not r.finished():
            logits = self.model.forward([r.output_token_ids[-1]], [r.cur], kv_cache=r.kv)  # 各自的 kv → 无串用
            ...
    assert self.runner.encoder_runs == enc_before   # ⑥ encoder 不重复

    # ② 准入 + prefill：按预算接纳新请求
    for r in 到达且待处理的请求:
        est = self.runner.estimate(r.messages)                     # ⑤ 不编码就估 visual/媒体数
        if not self.budget.fits(0, est["visual_tokens"], est["num_media"], ...): continue
        self._prefill(r, it)   # r.kv=KVCache(cfg) 各自一份；build_inputs→validate→merge  # ①对齐 ②无串用 ④timestamp

    # ③ 回收完成的请求

三层缓存的分工也在这条链上：ProcessorCache（预处理像素）与 EncoderOutputCache （视觉 embedding）由 runner._encode_image() 内部命中，省的是 ③ 无 stale cache 前提下的 encoder 算力（决定 TTFT）；而 LLM KV Cache 是每请求各持一份， 保证 ②无串用 且 decode 不重算历史。至此，本课要「同时守住」的六条正确性， 在 run() 的一次迭代里全部能对号入座。
