为什么需要这个知识 Learn

到 Lesson 23 为止，我们已经能把一张图片塞进文本序列： 图片被编码成一小段视觉 embedding，在 token 序列里占一段占位 token，再被逐位替换。 但真实请求从来不是「永远一张图」——有的纯文本（0 张图）、有的一张图、有的两三张图交错在文字里、 还有的直接甩一段视频。每一种，塞进序列的视觉 token 数都不一样。

这一课要彻底搞清三件事：①每种媒体贡献多少个 visual token（怎么算）；②同一批请求为什么序列长度各不相同（动态）； ③多段媒体与它们的占位区间怎么一一对齐、不串位（顺序）。 这三件事是后面多模态调度（按 visual token 数做预算 gating）和KV 预留的直接前提—— 序列有多长、KV 要留多少，全由这里算出来的 visual token 数决定。

先看问题：把 visual token 数写死会怎样 Learn

先用本课的真实数字感受一下「动态」到底有多动态。本课视觉配置： image_size=16、patch_size=8 → grid 2×2 → 每张图 num_patches = 4 个 visual token；视频默认采 num_frames=4 帧，逐帧编码后拼接， 所以一段视频 = 帧数 × 每帧 patch 数 = 4 × 4 = 16 个 visual token。文本用 byte-level tokenizer： 每个 UTF-8 字节 = 1 个 token，一个中文字 = 3 字节 = 3 个 token。

把这些代进「同一批」三个请求，序列长度立刻岔开（len = 1(BOS) + 文本字节数 + Σ visual token）：

请求文本媒体visual token 数序列总长
A「看图」= 61 张图41 + 6 + 4 = 11
B「看图」「还有」= 122 张图（交错）4 + 4 = 81 + 12 + 8 = 21
C「看视频」= 91 段视频（4 帧）4 × 4 = 161 + 9 + 16 = 26

如果偷懒把「每个请求 = 4 个 visual token」写死：请求 B 实际有 8 个视觉 embedding， 却只开了 4 个占位——多出来的 4 个 embedding 无处安放，合并时直接越界或错位； 请求 C 需要 16 个占位却只给 4 个，视频后 12 帧全丢。所以 visual token 数必须按媒体逐个动态算出， 不能是常数。这正是 build_inputs() 存在的理由。

一句话解释 + 直觉类比 Learn

一句话解释
「动态 visual token」= 先按每段媒体各自的类型/大小算出它要占多少个 token，再照这个数在序列里挖出对应长度的占位坑， 最后把视觉 embedding 一格一格填进去。图片挖 4 格、4 帧视频挖 16 格，坑的大小随媒体而变。

类比：给文章排版留图框。一篇图文混排的文章，每张配图尺寸不同——你得先量好每张图多大， 在正文里预留对应大小的空白框，再把图贴进去。图大就留大框，图小就留小框； 绝不能所有图都留同样大的框，否则要么图溢出、要么留一堆空白。

类比的边界（别被带偏）
「图框」是二维的空白区域，而这里的占位是一维序列上连续的一段 token（每格是一个 text_hidden=32 维向量位）。 更关键：文章排版可以先留框后填图、顺序随意调；而这里占位区间的顺序必须严格等于媒体的出现顺序—— 第 0 段媒体对第 0 个占位区间，一旦串位，视觉 embedding 就填到别人的坑里，模型看到的图文关系全错。

精确定义：visual token 数怎么算、shape 是多少 Learn

逐符号拆开每种媒体贡献的 visual token 数（num_patches = grid² = 4）：

image: nvisual= numpatches=4 video: nvisual= numframes×numpatches =4×4=16 序列长: seqlen= 1(BOS) +∑textbytes +∑nvisual
  • num_patches：一张图切成的 patch 数 = (image_size / patch_size)² = (16/8)² = 4。 每个 patch 经 vision encoder + projector 变成一个 visual token。
  • num_frames：视频采样帧数（默认 4）。视频 = 若干帧图片按时间顺序拼起来，所以乘上帧数。
  • text_bytes：文本段的 UTF-8 字节数（中文字 = 3）。BOS 单独占 1 个 token。

输入 / 输出 shape（用真实数字）：一张图的像素经 encoder.encode() 得到 (num_patches, vision_hidden) = (4, 48)，再经 projector 投影到文本维得到 (num_patches, text_hidden) = (4, 32)。所以 build_inputs() 收集到的 visual_embeds_list 里，每个元素形状是 (n_visual, 32)：图片是 (4, 32)、 4 帧视频是 (16, 32)。对应的占位区间长度 length 就等于这个 n_visual。

estimate() vs build_inputs()：算数 vs 真编码
estimate()（runner.py）不做任何编码，只按上面的公式把 visual token 数加出来—— 供调度器提前做预算 gating（这条请求会不会撑爆 KV？）。build_inputs() 才真正跑 vision encoder 拿到 embedding。两者对同一请求算出的 visual token 数必须一致。

小规模手算：请求 B「看图[图]还有[图]」怎样变成序列 Learn

合并阶段（Lesson 23 的 merge_multimodal_embeddings）把每个 visual_embeds[k] 逐格覆盖到 range[k] 的位置上，PAD 只是临时占位、其向量值无所谓。

顺序与占位对齐：media_index 是关键 Learn

「动态数量」之外，本课第二个坑是顺序。parse() 用一个计数器 media_i：每遇到一段媒体，就取 lens[media_i] 个占位、记 media_index=media_i、然后 media_i += 1。 这保证了第 k 段媒体 ↔ 第 k 个占位区间 ↔ visual_embeds_list[k] 三者下标严格一致。

为什么必须一致？合并时是按下标 k 把 visual_embeds[k] 填进 range[k] 的。 若哪一环下标错位（比如先编码图片却按错误顺序记 range），图 0 的 embedding 就会被填到图 1 的坑里—— 模型读到的图文对应关系整个错乱，但形状仍可能对得上、不报错，成为最难查的 bug。 所以 validate_placeholders（placeholders.py）会硬性检查 media_index == k、 区间不重叠不越界；parse() 结尾也检查 media_i == len(lens)，媒体数与占位数不符立刻抛错。

常见误解 Learn

误解为什么错
「每个请求的 visual token 数是固定常数，可以写死」 随媒体数量和类型而变：0 图=0、1 图=4、2 图=8、4 帧视频=16。写死会让占位数与真实视觉 embedding 数不匹配 → 合并时越界或丢内容。
「visual token 就是像素，一个像素一个 token」 visual token 是 encoder+projector 产出的 text_hidden=32 维 embedding，数量 = patch 数（4）而非像素数（16×16）。一个 patch = 一个 visual token。
「占位用哪个 id、按什么顺序记都无所谓，反正会被替换」 id（PAD）确实会被覆盖，但顺序至关重要：media_index 必须等于媒体出现次序，否则视觉 embedding 填错坑，图文关系全乱，且常常不报错。
「视觉 token 只影响图那几个位置，别处不受影响」 它改变整条序列长度 → 位置编号、KV Cache 行数、调度预算全都随之变。请求 B 比 A 多 4 个 visual token，就多占 4 行 KV。
「视频和图片一样，也只占 num_patches 个 token」 视频要逐帧编码再按时间拼接，是 帧数 × num_patches。4 帧视频 = 16，是单图的 4 倍。

代码映射 Learn

本课讲解全部对应下面两个文件里的真实逻辑：

  • MultiModalRunner.build_inputs()（runner.py）：遍历 segments，对每段媒体真编码—— 图片走 _encode_image → (4,32)；视频走 _encode_video（逐帧拼接）→ (帧数×4, 32)； 收集 lens = [len(e) for e in visual_embeds_list]，这就是「动态 visual token 数」的落地。
  • MultiModalRunner.estimate()（runner.py）：不编码，只按公式把 visual token 数加出来，供调度预算 gating。
  • MultiModalInputParser.parse()（inputs.py）：BOS + 文本 tokenize + 媒体处 [PAD_ID]*n 占位， 用 media_i 计数器记录 PlaceholderRange(offset, length=n, media_index, modality)， 并在结尾校验媒体数与 lens 一致——本课「顺序与对齐」的核心就在这里。
  • 下游 validate_placeholders / merge_multimodal_embeddings（Lesson 23）消费这些 range，把视觉 embedding 逐格填入。
