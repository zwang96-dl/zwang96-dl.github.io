Lesson 7 · KV Cache

为什么需要这个知识 Learn

KV Cache（下一课）是 vLLM 一切内存与速度魔法的起点，但如果没先亲眼看到 「不缓存有多亏」，它就只是一句口号。这一课的任务是当侦探：把朴素生成循环逐步拆开， 用具体数字量出每一步到底重算了多少、累计浪费了多少，给下一课的优化立案。 搞懂三件事就够了：朴素循环每步做什么、浪费长成什么形状（为什么是 O(n²) 而不是线性）、 以及为什么优化后输出必须一字不差。

先看问题：朴素解码到底在浪费什么 Learn

先把「一段文字」变成模型看得懂的 token。本课用的是 byte-level tokenizer（按字节切）： 一个 UTF-8 字节 = 一个 token id（0..255），再加 BOS/EOS/PAD 三个特殊符号，词表共 vocab=259。 所以英文 "Hello" = 5 个字节 = 5 个 token，前面补一个 BOS，整段 prompt 共 6 个 token （中文更贵：一个汉字 = 3 字节 = 3 个 token）。

现在要在这 6 个 token 后面生成 8 个新 token。朴素做法（generate_naive）每生成一个 token， 都把「当前整段序列」从头 forward 一遍：

生成第几个喂进 forward 的 token 数其中「重复计算」的部分
第 1 个6（整段 prompt）—（这一步不可避免）
第 2 个7前 6 个又算了一遍
第 3 个8前 7 个又算了一遍
………
第 8 个13前 12 个又算了一遍

累计喂进模型 6+7+8+…+13 = 76 个 token，可其中真正「新」的只有那 8 个新 token 各自的第一次计算， 其余全是把历史又算一遍。更扎心的是：每一步 forward 都算出了 (当前长度 × 259) 个 logits， 采样却只用最后一行那 259 个——前面所有行的 logits 也是白算。这就是「案发现场」； 下一课的 KV Cache 会把累计从 76 压到 6 + 7×1 = 13。

直觉类比 Learn

一句话解释
朴素生成 = 每写出一个新字，都先把「到目前为止的整段话」从头到尾重读一遍再动笔。 写得越长，这次「重读」就越长，累加起来是 O(n²)。

类比：不做笔记的抄书人。你要接着一段话往下写。每写一个字之前，你都强迫自己 把前面所有字从头默读一遍——写第 1 个新字前读 6 个字，写第 2 个前读 7 个字…… 其实你完全可以边读边把要点抄在便签上（这正是下一课的 KV Cache），可朴素抄书人偏不记，每次都从头来。

类比的边界（别被带偏）
别把「重读」理解成只是慢——它是逐层重算了每个历史 token 的 Query / Key / Value 向量和整套 attention， 是实打实的浮点运算。但要记住：这些重算出来的数值和第一次完全一样—— 浪费在「算了多少次」，不在「算出什么」。所以优化的目标是少算，而不是算出不同的结果。

精确定义与公式：O(n²) 是怎么来的 Learn

先把「一步」说清楚。朴素循环每一步调用一次 model.forward(ids, positions)：

  • 输入：ids 形状 (L,) 的 token 序列、positions 形状 (L,)，其中 L = 当前整段长度。
  • 输出：logits 形状 (L, 259)——每个位置一行、每行 259 个词表分数；采样只取 logits[-1]（最后一行）。
  • 我们把这一步「处理的 token 数」记作 processed = L。

设 prompt 长 p、要生成 n 个新 token。第 t 步（t = 0,1,…,n−1）时整段长度是 p + t，于是累计处理量：

Cnaive (n) = ∑ t=0 n−1 (p+t) = n·p + n(n−1) 2

逐符号看这个式子：

  • n：生成的 token 数；p：prompt 长度；t：第几步（从 0 数）。
  • n·p：线性项——每一步至少要「照顾」到 prompt 那 p 个 token。
  • n(n−1)/2：平方项，也就是纯重复。第 t 步重算了前面已经算过的 t 个 token，把这些 0+1+2+…+(n−1) 加起来就是它。它随 n 的平方增长，是 O(n²) 的元凶。

对照聪明做法：prefill 处理整段 p，之后每步只处理 1 个新 token，累计 C_cached(n) = p + (n−1)，是线性的。 代入本课例子 p=6, n=8：C_naive = 8·6 + 8·7/2 = 48 + 28 = 76、C_cached = 6 + 7 = 13。 生成越长，两者差距越拉越大——这就是 O(n²) 对 O(n) 的碾压。

为什么是「平方」而不是「指数」
每一步只多算「一整段」，而段长是线性增长的（+1 一次）；把这些线性增长的量累加起来才得到平方 （1+2+…+n ≈ n²/2）。它不是指数（指数是每步翻倍）。分清这点很重要：O(n²) 已经很糟， 但还没到无法收拾，而它恰好能被 KV Cache 干净地降回 O(n)。

小规模手算：prompt=3、生成 3 个 Learn

用一个能一眼看完的小例子：prompt 有 3 个 token（位置 0/1/2），生成 3 个新 token（位置 3/4/5）。 逐步列出朴素做法每步喂了什么：

步 t喂进 forward 的 tokenprocessed其中重复（浪费）真正新增
0[t0 t1 t2]30（首次必算）t0 t1 t2
1[t0 t1 t2 n0]43（t0 t1 t2）n0
2[t0 t1 t2 n0 n1]54（t0 t1 t2 n0）n1

累计 processed = 3+4+5 = 12；其中浪费 0+3+4 = 7，真正必要的只有 3+1+1 = 5（= cached 的量）。 套公式验证：C_naive = 3·3 + 3·2/2 = 9 + 3 = 12 ✓，C_cached = 3 + 2 = 5 ✓。下面把这个「浪费三角形」画出来：

重复计算三角形（每一行 = 朴素做法的一步）
步 0 t0 t1 t2
步 1 t0 t1 t2 n0
步 2 t0 t1 t2 n0 n1
虚线灰框 = 这一步被重复重算的历史 token（纯浪费）；实线框 = 这一步真正新增的 token。 灰框越堆越多，堆成一个三角形——三角形的面积就是 n(n−1)/2 那部分 O(n²) 浪费。

到底哪些计算被重复了：用本模型的具体数字 Learn

「处理 1 个 token」在一次 forward 里并不便宜。本课 tiny model 配置： hidden=32、layers=2、heads=4、kv_heads=2（GQA）、head_dim=8、intermediate=64。 每个 token 在每一层都要重新算出：

  • Query：heads × head_dim = 4 × 8 = 32 个数
  • Key：kv_heads × head_dim = 2 × 8 = 16 个数
  • Value：同样 16 个数
  • 再加 attention 加权求和、MLP（32 → 64 → 32）等。

朴素做法在第 t 步，把前面 p+t−1 个历史 token 的这些量在 2 层里全部重算一遍—— 而它们的数值和上一步算出的一模一样。下一课会发现：其中真正值得存下来复用的，只有 Key 和 Value （因为它们会被未来每一步的新 Query 反复查询与加权），Query 算完即弃。这正是「KV Cache」名字的由来：缓存 K 和 V。

形状（shape）随步数怎么变
朴素做法第 t 步：输入 (p+t,) → 每层的 attention 分数矩阵 (p+t, p+t) → 输出 logits (p+t, 259)，只用最后一行。 注意那个 attention 矩阵是边长随步数增长的方阵，所以每一步本身也越来越贵；本课用「processed token 数」这个更直观的指标来量化浪费，得到 O(n²)。

常见误解 Learn

误解为什么错
「越生成越慢是天经地义，模型就这样」慢的根源是重算历史，不是必然。KV Cache 能把每步压回「只算 1 个新 token」，每步耗时几乎恒定。
「累计处理量随长度线性增长」是 O(n²)：公式里的 n(n−1)/2 项随平方增长。生成 8 个就已是 76 vs 13。
「优化后输出可能会变一点点」朴素与 cached 在贪心解码下逐 token 完全一致。浪费只在「算了多少」，不在「算出什么」；输出不变是优化的正确性底线。
「被重复的是最后的 logits / 采样那一步」被重复的是每一层、每个历史 token 的 Q/K/V 和整套 attention——重头戏。logits 反而是「每步只用最后一行、其余全白算」的另一种浪费。
「prompt 短就没这个问题」prompt 短只把线性项 n·p 变小；平方项 n(n−1)/2 由生成长度决定，生成得长照样爆。

代码映射 Learn

本课两个关注函数都在 mini_vllm/engine/generate.py。先看罪魁 generate_naive()：

# generate_naive()：每步都喂「当前整段」
for step in range(max_new_tokens):
    logits = model.forward(ids, list(range(len(ids))))  # ← len(ids) 越来越大
    tok = smp(logits[-1])          # 只用最后一行 logits
    processed = len(ids)           # 本步「处理」的 token 数 = 整段长度
    res.total_processed_tokens += processed
    ids.append(tok)                # 下一步又把它连同前缀全部重算

关键就在 range(len(ids)) 与 processed = len(ids)：len(ids) 每步 +1， 于是 processed 走出 6, 7, 8, … 的递增序列，累加即 O(n²)。对照 generate_cached()：prefill 之后 decode 每步只 forward([tok], [cur])——喂 1 个 token、total_processed_tokens += 1。

processed_token_curves() 则把这两条曲线解析地算出来（动画与 run 6 都用它，不必真跑模型）：

for t in range(n_new):
    tot_n += prompt_len + t                  # naive：第 t 步处理整段 p+t
    tot_c += (prompt_len if t == 0 else 1)   # cached：只 prefill 处理整段，其余每步 1

这两行正是前面公式 C_naive = n·p + n(n−1)/2 与 C_cached = p + (n−1) 的逐步累加版； 上方动画就是把这两条曲线单步画出来。

本节总结 & 下一步

  • 朴素生成每步重算整段前缀 → 累计 O(n²)，越生成越慢。
  • 浪费在于历史 token 的 K/V 被反复重算。
  • 优化必须保持输出不变——这是正确性底线。
