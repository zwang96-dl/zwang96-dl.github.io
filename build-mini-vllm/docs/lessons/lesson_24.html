为什么需要这个知识 Learn

前面几课我们把图片变成了视觉 embedding（预处理、 vision encoder、合并进文本 embedding）。 这一课要把它们串成一条能真正跑出文字的管线，并回答一个决定性能生死的问题： vision encoder 到底该跑几次？

答案是——只在 prefill 前跑一次。想清楚这一点，你就抓住了多模态推理最大的性能杠杆。 反过来，如果没想清楚，最容易犯的错就是「每 decode 一个字都重跑一遍 encoder」， 下一节我们用具体数字看看那有多亏。

先看问题：如果每步 decode 都重跑 encoder 会怎样 Learn

设一条消息是「1 张图 + 文字问题」，模型要回答 8 个 token。 一张 16×16 的图经过 patch 切分（patch_size=8，网格 2×2）得到 4 个 patch，vision encoder + projector 把它们变成 4 个视觉 token （每个是 32 维向量）。关键事实：这 4 个视觉向量只由图片像素决定，和模型接下来吐什么字毫无关系。

做法prefill 阶段生成 8 个字期间vision encoder 运行次数
❌ 每步都重编码编码 1 次每 decode 一步再编码 1 次 × 81 + 8 = 9 次
✅ 本课做法编码 1 次decode 全程不碰 encoder1 次

同一张图算 9 次，其中 8 次结果逐值完全一样——纯浪费。在我们的玩具模型里这只是慢一点； 但真实 vision encoder（如 CLIP ViT-L）有 三亿以上参数，比文本 decode 一步重得多， 每步重跑会让延迟直接翻几十倍。所以「encoder 只跑一次」不是小优化，而是多模态推理能不能用的分水岭。

为什么可以只跑一次？（这是全课的题眼）
视觉 token 一旦算出，就以普通向量的身份进入 prefill 序列，和文本 token 一起被写进 LLM 的 KV Cache。decode 时新字要「看图」，看的是缓存里那几行视觉 K/V—— 不需要再回到 encoder。这和 Lesson 7 的核心事实一脉相承： 第 i 个位置的 K/V 只依赖前 i 个位置，历史 K/V 算一次就永远有效。

一句话解释 + 直觉类比 Learn

一句话解释
多模态推理 = prefill 时把图「读懂」一次并记进 KV Cache， 之后 decode 就当作纯文本续写，边写边看缓存里的图，绝不回头再读图。

类比：看图作文。你先花几分钟仔细看一遍图（vision encoder，跑一次）， 在脑子里记下要点（视觉 token 进 KV Cache）。接下来动笔写作文，一个字一个字往下写（text decode）， 每写一句都凭记忆瞄一眼图的要点，而不是每写一个字就把图重新盯着看一遍。

类比的边界（别被带偏）
「看图」和「记要点」在这里是同一步：encoder 的输出直接就是写进 KV 的内容， 中间没有「另外记笔记」的动作。另外——记忆（KV Cache）是逐层存的， 不是脑子里一张模糊印象；每一层 Transformer 都存了图对应的 K/V。

精确定义：视觉 embedding 不是 token id Learn

纯文本模型里，输入是一串整数 token id，模型第一步用 embed[id] 查表变成向量。 但视觉信息没有对应的 id——图片不能塞进 259 大小的词表。多模态的做法是： 绕过查表，直接提供向量。这就是 forward(inputs_embeds=...) 这条路径的意义。

逐符号看每一步的输入/输出 shape（用本课真实数字）：

步骤输入 shape输出 shape数字来源
预处理原图 (H,W,3)(3, 16, 16)image_size=16
patch embedding(3, 16, 16)(4, 48)grid 2×2=4 patch，vision_hidden=48
vision encoder（1 层非因果）(4, 48)(4, 48)patch 间互相可见，形状不变
projector（投影）(4, 48)(4, 32)投到 text_hidden=32
projector 为什么必不可少
视觉空间是 48 维，文本空间是 32 维。要把视觉向量和文本向量拼在同一个序列里， 维度必须对齐——projector 就是那把「48→32」的尺子。若不对齐， merge_multimodal_embeddings 会直接报「维度不一致」的错。

小规模手算：一条「图 + 你」消息如何变成 prefill 序列 Learn

取一条最小消息：1 张图 + 一个汉字「你」（为聚焦原理，略去 role 前缀）。 chat template 渲染出的 segments 是 [("media", 图), ("text", "你")]。 MultiModalInputParser.parse() 按下面的规则拼 token 序列：

  • 先放 BOS（id=256）；
  • 遇到媒体：图有 4 个视觉 token，就插入 4 个占位 token（用 PAD，id=258）， 并记下 PlaceholderRange(offset=1, length=4)；
  • 遇到文本「你」：byte-level 分词，一个汉字＝3 个 UTF-8 字节＝3 个 token （你 = 228, 189, 160）。

于是 input_ids（长度 1+4+3 = 8）与合并后的 embedding 如下。 注意占位 PAD 的 embedding 是废的——它在合并时被逐行换成视觉向量：

merge：把占位行换成视觉行（灰=文本，橙=视觉）

merged 形状 = (8, 32)；position 1–4 由 PlaceholderRange(offset=1,length=4) 定位，被 visual_embeds_list[0] 的 4 行逐行覆盖。

最后 prefill() 调用 model.forward(None, positions=[0..7], inputs_embeds=merged, kv_cache=cache)： 第一个参数是 None（不查表，因为我们直接给了 inputs_embeds），得到 logits 形状 (8, 259)，取最后一行 logits[-1] 采样出第一个生成 token， 同时这 8 个位置的 K/V（含 4 行视觉）已写进 cache。之后 generate() 从 position 8 起 逐 token decode，每步只喂 1 个 token、复用 cache——encoder_runs 停在 1，不再增长。

encoder 只跑一次 & 三层缓存别混淆 Learn

多模态里有三个层级完全不同的缓存，初学者极易混为一谈。它们缓存的东西、命中的时机、 由谁管理都不一样：

缓存缓存什么命中省掉哪一步本课代码
ProcessorCache预处理后的像素 (3,16,16)省掉 resize/normalizeself.processor_cache（_encode_image）
EncoderOutputCache投影后的视觉 emb (4,32)省掉整个 encoder+projectorself.encoder_cache（_encode_image）
LLM KV Cache每层每 token 的 K/V省掉历史 token 的 attention 重算KVCache（generate 里 new 一个）

要分清两件独立的事：

  • 「encoder 只在 prefill 跑一次」——这是本课默认就成立的，靠的是 generate() 的结构：只有 prefill() 走 build_inputs→_encode_image， decode 循环里完全没有 encoder。所以哪怕三个缓存都不开，同一次 generate 里 encoder 也只跑一次 （encoder_runs == 1）。
  • ProcessorCache / EncoderOutputCache——这是跨请求的进阶优化： 下次另一个请求又发来同一张图时，直接命中缓存、连这一次编码都省了。 它们是可选的（默认 None）。
一句话记住区别
LLM KV Cache 让 decode 便宜（不重算历史 attention）； Encoder/Processor Cache 让 重复的图便宜（不重复编码同一张图）。 前者本课必用，后者本课可选。

常见误解 Learn

误解为什么错
「decode 每步都要重新看一遍图」图的信息在 prefill 时已写进 LLM KV Cache，decode 只需读缓存里的视觉 K/V。重跑 encoder 是纯浪费（9 次 vs 1 次）。
「视觉 token 和文本 token 一样是整数 id」视觉是连续向量，没有词表 id。它走 inputs_embeds 直接进 hidden 空间；input_ids 里那几个位置只是 PAD 占位，其 embedding 会被换掉。
「prefill 序列长度 = 文本 token 数」视觉 token 也占序列位置、也进 KV。本例 8 个位置里有 4 个是视觉。序列长度 = BOS + 视觉 token 数 + 文本 token 数。
「开了 KV Cache 就不用管 encoder 重复了」两者层级不同：KV Cache 管的是同一次生成内的 decode 复用；同一张图跨请求不重复编码要靠 EncoderOutputCache。
「合并 embedding 时要把 PAD 的 embedding 也算进去」PAD 占位行的 embedding 被 merge_multimodal_embeddings 整行覆盖成视觉向量，原值根本不参与，算它无意义。

代码映射 Learn

本课讲解对应 mini_vllm/multimodal/runner.py 的两个焦点函数：

# prefill()：管线 ①②③④，encoder 在这里(且仅在这里)被触发
input_ids, ranges, visual_embeds_list, _ = self.build_inputs(messages)  # ①②：_encode_image → encoder_runs += 1
text_embeds = [list(self.model.embed[tid]) for tid in input_ids]         # 文本查表(PAD 行是废的)
merged = merge_multimodal_embeddings(text_embeds, ranges, visual_embeds_list)  # ③：占位行换成视觉行
logits = self.model.forward(None, positions, kv_cache=cache, inputs_embeds=merged)  # ④：多模态 prefill

# generate()：④之后进入 decode 循环，⑤⑥——不再调用 build_inputs/encoder
tok = smp(pre.logits[-1])                       # 首 token 来自 prefill 最后一行
for _ in range(max_new_tokens - 1):
    logits = self.model.forward([tok], [cur], kv_cache=cache)   # 纯文本 decode，只喂 1 个 token
    tok = smp(logits[-1]); cur += 1              # encoder_runs 保持不变 → 验证「只跑一次」

encoder_runs 这个计数器（每次真正跑 encoder 就 +1）就是本课测试的抓手： 一次 generate 结束后它应等于媒体数（本例 1 张图 → 1），证明 decode 没重复编码。
