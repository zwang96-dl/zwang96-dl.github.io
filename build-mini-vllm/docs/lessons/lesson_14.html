为什么需要这个知识 Learn

这是整个 vLLM 名字里那个「Paged」的正主。Lesson 7 我们把每层 K/V 存成一整段连续的行；Lesson 12 发现这样会严重浪费显存。 本课要解决的就是这个浪费：把 KV 切成固定大小的物理块（block），让一条序列 「逻辑上是一整段、物理上却散落在各处」。想透这一课，等于同时想透了三件事—— 为什么能省显存、地址怎么翻译、为什么散着放结果还一模一样。这三件事对应本课的三个目标： logical→physical 映射、逻辑连续物理分散、paged == 连续。

先看问题：连续 KV 到底卡在哪 Learn

回忆本课的 tiny model：layers=2、kv_heads=2（GQA）、head_dim=8。 一个 token 每层要存 K、V 各 kv_heads·head_dim = 2·8 = 16 个数，两层合计 (16+16)×2 = 64 个数。现在假设显存里能放的 KV 总量固定，我们并发跑几条序列。 连续存储的麻烦在于：每条序列都得预留一段连续空间。

做法为一条序列怎么占内存问题（用数字说）
按 max_seq_len 预留不管实际多长，一上来就占满 max_seq_len=64 个 token 的连续空间 prompt 只有 10 个 token，却占了 64 个的坑 → 浪费 84%（Lesson 12 的「内部碎片」）
按需增长但要求连续序列要变长时，原地后面必须刚好有连续空位 空间被其它序列切得七零八落：总空闲够、却没有一段够长的连续空间（外部碎片）
分页（本课）要多少 token 就借几个固定大小的块，块与块之间可以不挨着 几乎零浪费：只在最后一个块里有不满的尾巴

结论：连续存储的敌人是「必须连续」这四个字。分页把这条要求拆掉—— 逻辑上还是一整段，物理上却允许东一块西一块。

直觉类比 Learn

一句话解释
Block Table = 一张「逻辑块 → 物理块」的翻译表。序列只认自己的逻辑块 0,1,2…（永远连续）， 真实数据放在哪个物理块，由这张表说了算。这正是操作系统页表（page table）的思路。

类比：图书馆的书架与索引卡。你要读的「第 1 章、第 2 章、第 3 章」在你脑子里是连续的（逻辑块）； 但图书馆可能把第 1 章放在 3 号架、第 2 章放在 5 号架、第 3 章放在 1 号架（物理块）。 你不用关心它们在哪——索引卡（block table）告诉你「第 i 章 → 几号架」。 想读完整本书，管理员按索引卡的顺序把书搬到你桌上排好（这一步叫 gather）。

类比的边界（别被带偏）
索引卡里「第 i 章 → 几号架」是一对一的普通映射；但物理块还能被多条序列共享 （多张索引卡指向同一号架），这就是 Lesson 16 的前缀缓存。 另外真实 vLLM不搬书——它的 PagedAttention kernel 直接跑到各号架上就地计算，省掉 gather； 本课为了能逐值对比，才先老实搬书。

精确定义：一次寻址的三步 Learn

给定一个 token 的逻辑位置 pos（0,1,2,… 就是它在序列里的第几个）， 和固定的块大小 block_size（本课示例取 4），寻址分三步：

logical = pos // block_size     # 它在第几个逻辑块（整除）
offset  = pos %  block_size     # 它在这个块里的第几个槽位（取余，范围 0..block_size-1）
phys    = block_table[logical]  # 查表：逻辑块 → 物理块 id

逐符号解释：// 是整除，把连续位置切成一段段（每 block_size 个一段）； % 是取余，告诉你落在那一段里的第几格；block_table 是一个列表， 下标是逻辑块号、值是物理块号。三步下来，一个逻辑位置 pos 就被翻译成 「物理块 phys 的第 offset 个槽位」这样一个真实存放地址。

输入 / 输出 shape（具体数字）
输入：一个整数 pos。输出：(phys, offset) 一对整数。
每个物理块每层存的是 block_size 行，每行是一条 token 的 K（或 V）向量，长度 kv_heads·head_dim = 16。所以「物理块 phys、layer li」对应的 K 数组 shape = (block_size, 16) = (4, 16)，V 同形。这正是代码里 kstore[(li, phys)] 是一个长度 4、每格 16 维的列表的原因。

小规模手算：一条 10-token 序列怎么落盘 Learn

设 block_size = 4，一条序列共 10 个 token（pos = 0..9）。 它需要 ceil(10 / 4) = 3 个逻辑块：L0 装 pos 0-3，L1 装 4-7，L2 装 8-9（最后一块只用了 2 格）。 假设分配器手上 0、2 号块已被别的序列占着（空洞），于是它按「编号最小的空闲块优先」的规则， 先后发给我们物理块 3、5、1。得到的 block table 是：

逻辑块装哪些 pos→ 物理块
L00,1,2,3block_table[0] = 3
L14,5,6,7block_table[1] = 5
L28,9（还剩 2 空槽）block_table[2] = 1

所以 block_table = [3, 5, 1]——逻辑上 0→1→2 连续，物理上 3→5→1 却是跳着的 （这跟本页动画演示的映射完全一致）。现在寻址三个具体 token 试试：

为什么物理分散，结果还逐值一致 Learn

关键洞察：attention 只看「每个位置的 K/V 是什么」，不看「它们存在内存的哪里」。 位置 pos 的 K 向量该是多少，只由这个 token 和它的位置决定（Lesson 7 讲过）， 跟它被放进 3 号块还是 5 号块毫无关系。所以做 attention 之前，只要把散落的块 按逻辑顺序 pos = 0,1,2,… 收集（gather）成连续的 K、V， 后面就是一模一样的普通 attention：

for pos in range(length):        # 严格按逻辑顺序 0,1,2,...
    phys = block_table[pos // bs] # 该 pos 在哪个物理块
    off  = pos %  bs             # 在块里第几格
    K.append(kstore[(layer, phys)][off])   # 取出来，排成连续
    V.append(vstore[(layer, phys)][off])
# 此时 K, V 与「连续版 KVCache 的 K, V」逐行相同 → attention 结果误差 = 0

所以 run 14 会看到 paged 与连续 KV 的 logits 最大绝对误差 = 0。 分页只改变「数据存在哪」，不改变「数据是什么」，因此对结果零影响—— 这就是本课第三个目标 paged == 连续 的含义。

常见误解 Learn

误解为什么错
「物理块乱序会打乱 token 顺序，结果会变」 顺序由 gather 时严格按 pos 遍历保证，物理块编号只是「存哪儿」。逻辑序不变，结果就不变（误差 0）。
「block_table[logical] 里的物理块号必须连续/递增」 恰恰相反，[3,5,1] 完全合法。允许不连续正是分页的意义——否则又退回连续存储的碎片问题。
「offset 用 pos // block_size 算」 写反了。//（整除）得到逻辑块号，%（取余）才得到块内槽位 offset。搞反会 gather 出错序。
「每个 token 单独占一个物理块」 一个物理块装 block_size 个 token（本例 4 个）。块是批发单位，不是零售——这样表才短、碎片才小。
「最后一个块没装满 = 浪费很大」 只有最后一块可能有不满的尾巴（本例 L2 空了 2 格），最多浪费 block_size-1 个 token 的量，与序列长短无关，是有界的小浪费。
