为什么需要这个知识 Learn

前 17 课里，每个子系统都是单独验收的：Lesson 7 的 KV Cache 只测「cached==naive」； Lesson 11 的调度器只测「谁先跑」；Lesson 14 的分页只测「块够不够」；Lesson 16 的前缀共享只测「命中率」。 可真实推理引擎里，这些机制是同时运转、互相牵制的——它们各自正确，合在一起未必正确。 本课就是把它们塞进同一个高压场景，检验「整机」是否仍然站得住。

先用一个数字说清「值不值得搭这套东西」。本课 4 条请求，如果用最朴素的「每步重算整段」做法（Lesson 6）， 处理量会大到离谱；而组合了 KV Cache + 分页 + 前缀共享 + 连续批处理后，处理量骤降：

做法累计处理 token 数相对
naive（每步重算整段，O(n²)）7341×（基线）
本课引擎（prefill 一次 + 每步 1 token，前缀还能省）≈139约 1/5.3

（734 与 139 都是下面「基线对比」小节里逐条算出来的真实数字。）省下的 5 倍不是免费的—— 它建立在「缓存无泄漏、调度不饿死、结果不出错」这三根柱子上。任何一根塌了，省下的算力就变成了废品。 本课要学的，正是如何同时守住这三根柱子。

先看问题：子系统单独都对，合起来会怎样翻车 Learn

为什么「综合场景」值得单独开一课？因为组合会催生单测照不出的新 bug。下面每一条都是真实引擎里踩过的坑， 用本课的具体配置（num_blocks=48、block_size=4、max_num_batched_tokens=16、 max_num_seqs=3、4 条动态到达的请求）说清「不做好会怎样」：

组合处不做好会发生什么（具体数字）
前缀共享 × 块回收 sys_a 与 sys_b 共享前 29 个 token（BOS + "System: be concise. What is "）的物理块。 若 sys_a 完成时把共享块也当成自己的还掉，sys_b 读到的就是脏数据 → 结果错； 若谁都不敢还 → KV 泄漏，48 个块很快耗尽。
长 prompt × 有限 token 预算 long 的 prompt 有 53 个 token，而一次迭代最多处理 16 个。不切块，它永远调度不出去 （53>16 → 停滞报错）；切了块，又会把预算全占满，短请求 chat 迟迟拿不到首 token。
动态到达 × 并发上限 4 条请求但 max_num_seqs=3：任一时刻最多 3 条在跑，第 4 条得排队。若调度只顾推进老请求， 晚到的 chat/long（arrival=2）可能被无限期饿死（starvation）。
连续批处理 × 正确性 4 条请求的 K/V 在同一迭代里混批计算、共享同一物理块池。一旦 block table 串了行， sys_a 的输出里就会混进 long 的信息（串请求）——单条跑永远测不出来。
于是本课的验收标准，正好一一对应这四个坑
正确性（cached==reference，不串请求）、无 KV 泄漏、无 starvation（全部完成）、性能不退化（对比 naive）。 这四条就是 run_experiment 里的四个自检，下面会逐条拆开。

本课的综合场景：4 条请求的全貌 Learn

tokenizer 是 byte-level：每个 UTF-8 字节 = 1 个 token（0..255），再加句首 BOS。 所以 prompt 的 token 数 = 字节数 + 1。下表是 configs/lesson_18_quick.json 里的真实 4 条请求：

idpromptprompt tokensmax_newarrival
(迭代号)		特点
sys_aSystem: be concise. What is AI?3260长 prompt，含共享前缀
sys_bSystem: be concise. What is ML?3281与 sys_a 共享前 29 token
chatHi352超短聊天，要求低 TTFT
longSummarize the quick brown fox story in detail please5342超长 prompt，必须切块

三个关键约束一起收紧了系统：

  • token 预算 = 16：long 的 53 个 prompt token 装不下一次迭代，会被 chunked prefill 切成 [16, 16, 16, 5] 共 4 块，穿插在别的请求之间处理。
  • 并发上限 = 3：4 条请求 > 3，任一时刻至少有 1 条在 waiting。
  • 动态到达：chat/long 在第 2 迭代才「出现」，调度器要能公平地把它们接进来。
共享前缀有多长
sys_a 与 sys_b 只有结尾 AI? / ML? 不同，前面 BOS + "System: be concise. What is " 完全一致 = 29 个 token。 按 block_size=4，前 7 个整块（28 token）可以被 sys_b 直接复用， 省去这 28 个 token 的重复 prefill——这正是 Lesson 16 前缀共享在综合场景里的收益。

直觉类比 Learn

一句话解释
这一课不是学新机制，而是当「指挥」：让 KV 缓存、分页、前缀共享、连续批处理这几件已经会的乐器 同时开响，且合奏出来的曲子仍然正确、不卡壳、每个声部都有机会发声。

类比：医院急诊分诊台。病人（请求）陆续到达（动态 arrival），有的病情简单（chat）， 有的需要长时间检查（long）；诊室数量有限（max_num_seqs=3），医生每小时的接诊量也有限 （max_num_batched_tokens=16）。好的分诊台要做到三件事：不误诊（正确性）、 不占着床位不放（无泄漏）、不让任何病人在走廊里等死（无 starvation）。 长检查还可以分几次做（chunked prefill），中间穿插看几个简单病人，谁都不必干等。

类比的边界（别被带偏）
急诊台可以「加人加床」，而推理引擎的物理块是硬上限（48 个）：块用完就真的没了。 真实 vLLM 此时会抢占（把某条请求的 KV 换出到内存腾地方）；本教学引擎不抢占， 所以一旦「有 RUNNING 却推不动」就直接报错停机——这是刻意的，为了让你看清「预算/块不足」的边界， 而不是被自动救场掩盖。

四条验收标准的精确定义 Learn

「整机正确」不是一句口号，而是四个可执行的断言（对应 run_experiment 里的四段自检）。逐条看它「测什么、怎么测」：

① 正确性（cached == reference，不串请求）
对每条请求 r，把它单独用 generate_cached 跑一遍得到「参考答案」ref， 再取引擎在混批场景下给它的实际输出 got，要求逐 token 相等：

for r in requests:
    ref = generate_cached(model, encode(r.prompt), r.max_new).generated   # 单独、无干扰
    got = 引擎在混批场景下给 r 的 output_token_ids
    assert got == ref            # 逐 token 一致 → 没被别的请求"污染"

它同时封杀两类 bug：重复执行（多算/漏算会让某个 token 变化）和 串请求（block table 串行会让 got 混入别人的信息）。 由于用的是贪心采样，参考答案是确定的，任何偏差都逃不掉。

② 无 starvation（所有请求都完成）
断言每条请求都有一个「完成迭代号」：all(r.finish_iter is not None)。 只要有一条永远排不上队，它的 finish_iter 就是 None，断言失败。balanced 策略靠 给等待中的 prefill 预留 1/4 预算（prefill_reserve = 16 // 4 = 4）来防止 decode 长期霸占算力。

③ 无 KV 泄漏
一条请求完成后必须归还它借的物理块。收尾时断言「已用块数 == 前缀缓存应当持有的块数」：

eng.shutdown()                       # flush 前缀缓存 + check_no_leak()
no_leak = (eng.allocator.num_used == 0)   # flush 之后，一个块都不该剩

开着前缀缓存时，请求跑完只还「自己独占的块」，共享块留给缓存；直到 shutdown() 把缓存 flush 掉， 占用才应归零。这一条专治上面「前缀共享 × 块回收」那个坑。

④ 性能不退化（对比 naive 基线）
把 4 条请求各自用 generate_naive 跑一遍，累加 total_processed_tokens 作为基线； 引擎的处理量应远小于它。这一条保证「为了正确/公平，没有把性能牺牲回朴素做法」。

小规模手算：迭代级调度走一遍 Learn

真实 4 条请求的完整时间线太长，手算容易出错。这里用一个缩小版把「连续批处理 + chunked prefill + 动态到达 + 并发上限」 四件事一次讲透（机制与真实场景完全一致，只是数字更小）：

  • 配置：max_num_seqs=2、max_num_batched_tokens=6、policy=balanced（chunked 开）。
  • 请求：A(prompt 3, max_new 2, arrival 0)、B(prompt 8, max_new 1, arrival 0)、C(prompt 2, max_new 1, arrival 2)。
  • balanced 的预留：prefill_reserve = 6 // 4 = 1（仅当还有等待请求时，给 decode 的预算扣掉 1，留给新 prefill）。
迭代预算 6→用量调度决策（本步跑谁 / 多少 token）本步之后的状态
16 → 6 准入 A：prefill 3；准入 B：prefill 3（8 里的第 1 块，chunked）。C 尚未到达。 A 已 prefill 完 → 出首 token；B 还差 5 个 prompt token
26 → 6 decode A：1 token（usable=6−reserve1=5）；prefill B：剩下 5 token（一次吃完）。 A 生成到第 2 个 → 完成；B prefill 完 → 出唯一 token → 完成；两者 KV 归还
36 → 2 并发已空出；C 已到达（arrival 2≤3），准入并 prefill 2 → 出唯一 token。 C 完成；running/waiting 皆空 → 引擎退出

三点值得盯住：(a) 第 1 迭代 B 的 8-token prompt 被切成 3+5 两块，没有一次占满预算堵死 A； (b) 第 2 迭代 A 的 decode 与 B 的 prefill 同框混批（这就是 continuous batching）； (c) 晚到的 C 在诊位空出后被顺利接入，没有饿死。全部 3 条请求都拿到了 finish_iter， 每步用完的块下步就还——正好命中四条验收标准。

基线对比：734 vs 139 是怎么算出来的 Learn

naive 每生成第 t 个 token 都重算整段，处理量是 prompt + t；一条请求生成 M 个 token 的累计处理量 = Σ(P+t) = M·P + M(M−1)/2（O(n²)）。引擎的 cached 路径只在 prefill 处理整段 P，之后每 token 只处理 1， 累计 = P + (M−1)。逐条代入本课真实数字：

请求PMnaive = M·P + M(M−1)/2cached = P + (M−1)
sys_a3266·32 + 15 = 20732 + 5 = 37
sys_b3288·32 + 28 = 28432 + 7 = 39
chat355·3 + 10 = 253 + 4 = 7
long5344·53 + 6 = 21853 + 3 = 56
合计——734139

734 就是 run_experiment 里 baseline 那一行（各请求 generate_naive(...).total_processed_tokens 之和）。 139 是不含前缀共享的 cached 处理量；再叠加前缀共享后，sys_b 能跳过与 sys_a 共享的 28 个 prompt token， 实际处理量还会更低。这就是「正确 + 公平」并没有以牺牲性能为代价的量化证据。

常见误解 Learn

误解为什么错
「每个子系统的单测都过了，整机自然就对」 组合会产生新故障面：前缀块的共享所有权、混批时的block table 串行、长短请求争抢同一预算—— 这些都只在多请求同时跑时才暴露。所以本课要用「整机四检」而非再堆单测。
「请求跑完，把它 KV 里的块全还掉就不泄漏」 开了前缀缓存后，请求的块里混着共享块。把共享块也还掉 → 后来的请求读到脏数据（正确性挂）； 正确做法是完成时只还独占块，共享块交给缓存，最后 shutdown() 统一 flush。
「长 prompt 就该一次算完，切块反而更慢」 long 有 53 token 而预算只有 16，一次根本算不完（会触发调度停滞报错）。 chunked prefill 把它切成 [16,16,16,5] 穿插进去，既能推进，又不会让 chat 的首 token 被堵死。总算力不变，延迟更公平。
「no_leak 就是 num_used == 0」 只有在 shutdown()（flush 前缀缓存）之后才该归零。运行途中，共享块被缓存合法持有， num_used 恰好等于「缓存持有的块数」——引擎 run() 收尾断言的就是这个等式，而非 0。
「调度器只要吞吐高就行」 只顾吞吐（如纯 decode-first）会饿死晚到的新请求。balanced 故意留 1/4 预算给等待中的 prefill， 用一点吞吐换无 starvation——公平也是验收标准之一。
