为什么需要这个知识 Learn

到这一课为止，前面几课教会了模型「怎么算一个 token」：KV Cache 让 decode 只喂 1 个 token， Prefill/Decode 分清了两种阶段。但真实的推理服务从来不是「一次只跑一个请求」—— 它同时面对几十上百条并发请求，每条到达时间不同、prompt 长短不一、还要生成不同长度的输出。 谁先跑、谁后跑、一步里塞多少 token 一起算——这就是调度器（Scheduler）要回答的问题。

调度器不改变「算什么」（输出逐 token 完全一致），只决定「何时算」。可就是这个「何时」， 直接决定了两个用户能直接感受到的指标：TTFT（Time To First Token，从提交到看见第一个字） 和吞吐（每秒能服务多少 token）。这一课就是要把这个「最核心的旋钮」拆开看清楚。

先看问题：不调度会怎样 Learn

假设同一时刻来了 3 个请求（用本课的 byte-level tokenizer：每个 ASCII 字符 = 1 token，中文一字 = 3 字节 = 3 token）：

请求promptprompt token 数要生成到达时刻
A（长输出）"Halt"45 个 tokeniter 0
B（短）"Hi"21 个 tokeniter 0
C（短）"Yo"21 个 tokeniter 0

最朴素的做法：一条一条串行跑。先把 A 从头跑到尾（1 步 prefill + 5 步 decode = 6 步）， 再跑 B（2 步），再跑 C（2 步）。这有两个致命问题：

  • GPU 在浪费。decode 每步只喂 1 个 token（见 Lesson 7），矩阵乘法瘦得可怜，GPU 的算力大半闲着—— 可与此同时，B 和 C 明明也想算，却只能干等。
  • 短请求被长请求「堵死」。B 只想要 1 个 token，却要排在 A 那 6 步后面。B 的 TTFT 从「本该 1 步」 被拖到 6 + 1 = 7 步。用户体验：明明问了句「你好」，却要等前面那个写长文的人写完。

调度器的使命：把多条请求打包进同一步一起算（提升 GPU 利用率 = 吞吐），同时用排序决定谁优先 （控制 TTFT 与公平性）。但「一起算」不能无限塞，于是有了两个预算。

两个预算：并发预算与 token 预算 Learn

调度器每一步（一次 schedule() 调用）的自由度，被 SchedulerConfig 里两个数字卡死：

预算含义单位卡住的是
max_num_seqs同时处于 RUNNING 的请求数上限「条」并发度：最多几条请求同时在跑
max_num_batched_tokens一次迭代里所有请求处理的 token 总数上限「个 token」单步算力：这一步总共能算多少 token

两者是两个不同维度的闸门，缺一不可，看它们怎么记账：

  • 一个 prefill 请求，本步要算 N 个 prompt token，就从 token 预算里扣 N。
  • 一个 decode 请求（prompt 已算完），本步只算 1 个新 token，从 token 预算里扣 1。
  • 每准入一条新请求进入 RUNNING，就占掉一个 max_num_seqs 名额。
举个具体的数字
设 max_num_batched_tokens = 12、max_num_seqs = 2。这一步若有 2 条正在 decode 的请求， 各扣 1 token（共 2），预算还剩 12 − 2 = 10；这 10 可以拿去给一条新请求做 prefill（比如 prompt 8 个 token，扣 8，剩 2）。 但因为 max_num_seqs = 2 已经占满，这条新请求连准入的机会都没有——名额比 token 更早卡死它。

代码里，token 预算就是 schedule() 开头的局部变量 budget = cfg.max_num_batched_tokens， 每安排一个 item 就 budget -= take；并发预算体现在准入循环里的 if len(self.running) >= cfg.max_num_seqs: break。

直觉类比 Learn

一句话解释
调度器 = 食堂的打饭窗口调度：max_num_seqs 是「同时能开几个窗口」， max_num_batched_tokens 是「这一轮所有窗口加起来最多打多少份饭」，而策略决定「排队的人按什么顺序进窗口」。

FIFO 就是老实排队先到先打；SJF 是「只要一份的人先打，端着满满一盘的靠后」；decode-first 是「已经在打的人先打完手里这勺」； balanced 则是「给新来的留一个窗口，别让老顾客一直占着不放」。

类比的边界（别被带偏）
食堂里「一个人只进一个窗口」，但推理里一步可以同时推进多条请求（批处理）——预算约束的是「这一轮总量」， 不是「一次只服务一个」。而且这里的「作业长度」在准入时只能估计（remaining_work 是粗略估算， 因为模型可能提前吐 EOS 停下），不像食堂能一眼看清盘子里有多少饭。

四种策略：精确定义 Learn

四种策略的差别，只体现在两个排序函数里：_order_running（决定 running 里谁先被推进） 和 _order_waiting（决定 waiting 里谁先被准入）。排序键（Python 元组，从左到右依次比较、升序）如下：

策略_order_running 排序键_order_waiting 排序键效果
fifo(arrival, id)(arrival, id)先到先服务，最公平也最简单
decode-first(prefill_done is False, arrival, id)(arrival, id)已在 decode 的排前面 → 尽快榨干在途请求，利于吞吐/TPOT
sjf(remaining_work, arrival, id)(remaining_work, arrival, id)剩余工作量小的先跑 → 短请求首 token 更早
balanced(prefill_done is False, arrival, id)(arrival, id)排序同 decode-first，但额外预留 1/4 token 预算给等待中的 prefill，防饥饿
读懂 prefill_done is False 这个键
Python 里 False < True（当作 0 < 1）。当一条请求已经做完 prefill（prefill_done == True）， prefill_done is False 求值为 False = 0，排在前面；还在 prefill 的请求求值为 True = 1，排在后面。 所以 decode-first / balanced 把「已在解码的请求」顶到最前。

remaining_work 的定义（见 request.py）：remaining_prompt + (max_new_tokens − num_generated)， 即「还没算的 prompt token 数」加「还没生成的 token 数」，是对一条请求「还剩多少活」的粗略估算。SJF 就靠它给短请求让路。

小规模手算：同一批请求，FIFO vs SJF Learn

用上面「先看问题」里的 A / B / C，配置 max_num_batched_tokens = 12， 并把 max_num_seqs = 1（相当于单窗口排队，为的是把排序的效果放到最大、看得最清楚）。 回忆每条请求的 remaining_work（此刻都还没开算）：

请求remaining_promptmax_new − generatedremaining_work
A45 − 09
B21 − 03
C21 − 03

每条请求占用「单窗口」的步数 = 1 步 prefill（这一步就产出首 token）+ 之后每步 1 个 decode。 所以 A（max_new=5）= 1 prefill + 4 decode = 5 步；B、C（max_new=1）各只需 1 步（prefill 即完成）。 首 token（TTFT）就出现在该请求的 prefill 那一步。

这组数字 = python3 course.py run 11 的真实输出
下面的手算配置就是 configs/lesson_11_quick.json（A="abc"/max_new 5、B="x"、C="y"/max_new 1，单窗口）。 你亲手跑 run 11,打印的 avg_TTFT 会与这里逐一对上——不是编的示例。

FIFO：准入顺序 A → B → C（按 arrival，同刻按 id）。单窗口，只能一条条来：

iter1234567
在跑A prefill✓A decA decA decA dec✓B prefill✓C prefill✓

TTFT：A = 1、B = 6、C = 7。平均 TTFT = (1+6+7)/3 ≈ 4.67。 B、C 明明各自只要 1 个 token，却被 A 那一长串输出堵在后面。

SJF：准入按 remaining_work 升序 → B(3) → C(3) → A(9)：

iter1234567
在跑B prefill✓C prefill✓A prefill✓A decA decA decA dec✓

TTFT：B = 1、C = 2、A = 3。平均 TTFT = (1+2+3)/3 = 2.0。 两条短请求几乎立刻拿到首 token，代价是长请求 A 稍晚开始。总迭代数不变（都是 7 步、输出逐 token 一致）， 变的只是先后——平均 TTFT 从 4.67 降到 2.0。这就是 SJF「短作业优先」的收益。

SJF 的公平性风险
如果短请求源源不断地来，长请求 A 的 remaining_work 永远排在最后，可能被无限期推迟（饥饿）。 这正是 balanced 想缓解的问题——见下一节。

balanced 如何防饥饿：预留 1/4 预算 Learn

换个场景看 balanced：很多请求正在 decode，把 token 预算吃得死死的，此时一条新请求想进来做 prefill。 设 max_num_batched_tokens = 8，当前有 8 条请求都在 decode（各要 1 token），waiting 里有一条新请求 N（prompt 4 token）。

关键代码在 schedule() 里：prefill_reserve = budget // 4 if policy=="balanced" else 0； 推进 decode 时 usable = budget − prefill_reserve（仅当还有 waiting 请求时生效），usable < 1 就跳过这条 decode。

预留给 prefill可用于 decode实际跑几条 decode留给新请求 N结果
decode-first0全部 88（把预算吃光，budget→0）0 tokenN 的 take = min(4, 0) = 0 → 永远进不来（饥饿）
balanced8 // 4 = 28 − 2 = 66（budget 降到 2 就停止推进 decode）2 tokenN 做 chunked prefill，take = min(4, 2) = 2 → 每轮啃 2 个 prompt token，稳步进入

代价：balanced 每步少跑 2 条 decode（吞吐略降），换来的是新请求不会被无限期堵在门外。这就是「decode 优先但给 prefill 留一条活路」。

常见误解 Learn

误解为什么错
「换个调度策略，生成结果会变」策略只改「何时算、以什么顺序算」，不改「算什么」。四种策略跑出的 token 序列逐 token 完全一致， 变的只是 iterations 数和平均 TTFT。run 11 会验证这一点。
「SJF 一定让平均等待更短，没有代价」SJF 降低平均 TTFT/等待，但可能饿死长请求：短请求不断插队，长请求的 remaining_work 永远排最后。公平性是代价。
「decode-first 对每个用户都更好」它优先榨干在途请求 → 提升整体吞吐，但新到达请求的 TTFT 会变差（要等在途的先用预算）。对「刚提交的新用户」反而更慢。
「只要 token 预算够，请求就能被准入」还要过 max_num_seqs（并发名额）和 free_blocks（KV 空间）两关。任一关不过都进不来——名额常常比 token 更早卡死请求。
「一步只能做 prefill 或只能做 decode」调度器可以把 decode 和（chunked）prefill 混在同一步：先推进 running 的 decode，再用剩余预算给等待请求做部分 prefill。这正是 balanced/chunked prefill 的基础（Lesson 15）。

诚实提醒：本玩具引擎里，只有 SJF 会明显拉开差距
你会发现 run 11 里 FIFO / decode-first / balanced 三者的数字往往相同，只有 SJF 不同。原因有二： ① 本引擎每次迭代本来就先推进「正在运行」的请求（含 decode），再准入新请求——也就是说「decode 优先」已是默认行为，所以 decode-first 与 FIFO 常常一致； ② decode 每步只花 1 个 token、几乎总能全部装下，balanced 预留的 1/4 预算在这个小 workload 里触发不了。 真正让 decode-first / balanced 拉开差距的场景（大量并发 decode 争抢紧张预算、新请求排队等 prefill），要在 GPU 上受批大小/显存约束时才明显。 这里我们能亲手看到的收益是 SJF（短作业优先）——这也是最直观的一个。
