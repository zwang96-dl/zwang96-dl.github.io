为什么需要这个知识 Learn

从 Lesson 19 起进入多模态主线：请求里除了文字，还可能夹着图片、视频。 这条主线后面所有课（图片变 tensor、切 patch、视觉编码器、把视觉向量拼回文本序列） 都建立在一个前提上——你清楚地知道每个阶段在做什么、不做什么。

本课几乎不写新算法，只做一件事：把一条多模态请求拆成 5 个界限分明的阶段， 并牢牢记住每个阶段的输入 / 输出是什么类型。这听起来简单，却是多模态最容易张冠李戴 的地方——一旦把「图片路径」当成 embedding、把「像素」塞进 tokenizer，后面调试会毫无头绪。 所以这一课的目标不是「跑通」，而是把阶段边界刻进脑子。

先看问题：不分阶段会怎样 Learn

假设你收到这样一条请求（用本课的消息格式表示）：

user(
    text("图"),                 # 一段文字
    image(<16×16 的图片像素>),  # 一张图片
)

模型的 Transformer（Lesson 6）只吃一样东西：一串隐藏向量， 形状 (seq_len, hidden=32)。文字和图片最终都要变成这种向量才能进模型。 问题在于——它们变过去的路径完全不同。如果不分阶段，会踩到三个具体的坑：

错误做法具体会发生什么（用真实数字）
把图片当文本，直接塞进 tokenizer 本课 tokenizer 是 byte-level，词表就是 0..255 这 256 个字节值，外加 3 个特殊 token（BOS/EOS/PAD），共 259 个。 一张 16×16×3 的图片是 768 个浮点像素，根本不是字节序列—— 喂进去要么崩，要么被当成乱码。
把图片路径字符串（如 "cat.png"）当输入 tokenizer 会老老实实把这 7 个字母编成 7 个字节 token。模型看到的是「c-a-t-.-p-n-g」这几个字母， 永远看不到像素。路径只是磁盘上的一个指针，不是内容。
直接把图片向量拼在文本 token 后面，不预留位置 这张图会被切成 2×2 = 4 个 patch → 4 个 visual token。 但如果 token 序列里没有为它们预留 4 个槽位，视觉向量就无处安放、也无法与文本对齐 （谁在图前、谁在图后就乱了）。

这三个坑的共同根源都是阶段混淆。正确做法是把请求拆成阶段，让文本走文本的路、 媒体走媒体的路，最后在约定好的位置合流。

一句话解释 + 直觉类比 Learn

一句话解释
多模态请求处理 = 先给图片在 token 序列里「占好座位」（占位 token）， 再由另一条流水线单独把图片算成视觉向量，稍后把向量「对号入座」填进那些座位。 本课只负责前半段：排座位 + 占座位。

类比：宴会预留座位。chat template 是引座员：他按客人到场顺序 （文字、图片、文字……）在桌上摆好座位牌，图片那一桌先放一张写着「<image>」的占位牌， 但他不认识图片本人长什么样。tokenizer 负责把文字客人一个个安排入座（每个字节一个座位）。 真正的图片嘉宾（视觉 embedding）由另一支车队（vision encoder）接送， 稍后才到场——到了就对号入座，坐进之前预留好的那几个位子。

类比的边界（别被带偏）
座位牌上写什么不重要（本课用 PAD 这个「无语义」token 当占位），重要的是位置和数量： 预留的座位数必须精确等于图片嘉宾人数（visual token 数 = 4），多一个空着、少一个坐不下。 而且——引座员（template）绝不负责把嘉宾变出来；嘉宾是另一条流水线（encoder+projector） 算出来的。把这两件事混在一起，就是最常见的 bug。

本课的两个核心函数，正对应阶段 ② 和 ③：

阶段 ② · MultiModalChatTemplate.render(messages)
输入：结构化消息列表 list[dict]，每条形如 {"role":..., "content":[{type,...}]}。
输出：(segments, rendered)。segments 是线性化的片段列表，每个是 ("text", str) 或 ("media", {"modality","ref"})；rendered 是调试用字符串 （媒体处显示 <image>）。
关键：它只把内容项排成一条直线、加角色前缀、在媒体处放一个标记。 它一个像素都不碰，也不决定 visual token 数。
阶段 ③ · MultiModalInputParser.parse(segments, media_visual_lens)
输入：② 产出的 segments，外加 media_visual_lens——一个 list[int]， 第 i 个数是第 i 段媒体已经算好的 visual token 数（本课一张 16×16 图 = 4）。
输出：(input_ids, ranges)。input_ids 是 list[int]（一维 token 序列）； ranges 是每段媒体的 PlaceholderRange(offset, length, media_index, modality)。
关键：文本段做 byte-level tokenize；媒体段插入 length=n 个 PAD 占位 token， 并记录这段占位的起点、长度、归属——供阶段 ⑤ 之后「对号入座」。

小规模手算：一句话 + 一张图 Learn

拿最开头那条请求走一遍。tokenizer 是 byte-level：每个 UTF-8 字节 = 1 个 token id（0..255）， 中文一字 = 3 字节 = 3 个 token；BOS=256、PAD=258（占位用）。图片切成 2×2=4 个 patch， 所以 media_visual_lens = [4]。

阶段 ② render() 把消息线性化（角色前缀 "User: " + 文字 + 媒体标记 + 换行）：

segments = [
    ("text",  "User: "),                          # 角色前缀
    ("text",  "图"),                               # 文字内容
    ("media", {"modality":"image", "ref":<像素>}),  # 媒体标记（不碰像素！）
    ("text",  "\n"),
]
rendered = "User: 图<image>\n"                      # 仅供调试

阶段 ③ parse(segments, [4]) 逐段处理，边拼 input_ids 边记占位区间。 下表是每个 token 的来历（"User: " = U s e r : 空格 = 6 个 ASCII 字节；「图」= 229 155 190）：

常见误解 Learn

误解为什么错
「图片路径 / 像素就是模型的 embedding」 路径是磁盘字符串、像素是原始数值，都不是隐藏向量。embedding 要到阶段 ⑤ 经 encoder+projector（48→32）才生成。本课的 token 序列里图片只是 4 个 PAD 占位。
「chat template 会把图片编码进去」 render() 只放一个 ("media",…) 标记 / <image>， 一个像素都不碰，也不决定 visual token 数。编码是 processor / encoder 的活。
「tokenizer 能顺手处理像素」 byte-level tokenizer 只认 0..255 的字节。16×16×3=768 个浮点像素喂进去只会崩或乱码。 媒体必须走独立的 processor 分支（阶段 ④）。
「占位 token 数随便填」 必须精确等于该媒体的 visual token 数（本例 4）。多了会留空槽、少了坐不下， parse() / validate_placeholders() 会直接报错。
「占位用什么 token 值很关键」 恰恰相反：占位值（这里用无语义的 PAD）稍后会被视觉向量整段覆盖，值本身不重要。 真正关键的是位置和数量（由 PlaceholderRange 记录）。
