为什么需要这个知识 Learn

前面几课我们学会了：缓存 K/V（Lesson 7）、把缓存切成 物理块（Lesson 13）、用引用计数让一块被多条序列共用。 这一课把这三样拼起来做一件很实际的事：两条请求如果开头一模一样，就让它们共用同一批物理 KV 块， 省掉重复的 prefill。这在真实服务里几乎无处不在——多轮对话每一轮都带着整段历史重新发来、 成千上万个请求共用同一段 system prompt、few-shot 提示里同样的例子被反复送进模型。 搞懂它，你就理解了 vLLM「为什么第二次请求快得多」的核心机制。

先看问题：重复 prefill 到底浪费了什么 Learn

本课的 tokenizer 是 byte-level：每个 UTF-8 字节 = 1 个 token，中文一字 = 3 字节 = 3 个 token， 句首还会加一个 BOS。设一段共享的 system prompt 是 "System: you are a helpful assistant. "，它 37 个字节，加 BOS 共 38 个 token。 现在有 100 条请求，每条都带着这段前缀、后面接一句不同的用户问题。

做法每条请求 prefill 的前缀 token 数100 条累计 prefill 前缀 token
不开前缀缓存38（每条都从头算一遍）100 × 38 = 3800
开前缀缓存（block_size=4）第 1 条 38；之后每条只算尾部 ≈ 238 + 99 × 2 ≈ 236

前缀里那 38 // 4 = 9 个整块（36 个 token）的 K/V，其实每条请求算出来的结果一模一样—— 却被重算了 100 遍。前缀缓存把它们算一次、存下来、后面 99 条直接复用，累计 prefill 从 3800 降到约 236， 省掉约 16×。而且 attention 的代价随长度是超线性的，前缀越长，省得越狠。

和 KV Cache（Lesson 7）的区别
KV Cache 是一条请求内部省重复：decode 时不重算自己的历史。前缀缓存是跨请求省重复： 请求 A 和请求 B 之间共用同一段前缀的 K/V。前者在一条序列的时间轴上省，后者在多条序列之间省。

一句话解释 + 直觉类比 Learn

一句话解释
前缀缓存 = 把「多条请求共同的开头」那段 KV 物理块存进一个全局缓存， 后来的请求发现开头相同，就直接借用同一批物理块（引用计数 +1），跳过这段 prefill。

类比：公司公用的「入职手册」。每个新人报到都要先读同一本手册（system prompt）， 再谈自己的岗位（各自的问题）。与其给每个人重印一本，不如在前台放一本共享的： 谁来都先翻这本（借用 = incref），读完自己那部分再散去。手册只印一次，省下大量印刷（prefill）。

类比的边界（别被带偏）
共享的前提比「印手册」严格得多：手册可以从任意一页翻起，但 KV 块只能从第 0 个 token、按块对齐地共享。 因为第 i 个位置的 K/V 依赖前面所有 token 以及它的绝对位置——只要开头差一个 token，或者位置错开一格， 后面整段 K/V 就全变了，不能共用。下一节用公式说清这一点。

为什么命中不改变输出（精确定义） Learn

回忆 attention（Lesson 4）：位置 i 的输出是对它能看到的所有位置 j ≤ i 的 Value 加权求和，权重来自 Query 与 Key 的相似度：

outi= ∑j≤i softmaxj ( qi·kj d ) ·vj
  • k_j, v_j：位置 j 的 Key / Value。由于是 causal（因果）注意力， 它们只由 token 0..j 算出，与 j 之后的 token 无关。
  • 本课模型的位置编码是 RoPE，用绝对位置：位置 j 的旋转角只取决于 j 这个下标， 与「这条请求还有多长、后面接什么」都无关。

把这两点合起来：只要两条请求的 token 0..L-1 完全相同，它们在这 L 个位置上的 k_j, v_j 就逐值（bit 级）相同。所以直接复用同一批物理块存的 K/V，不是近似、不是「差不多」， 而是精确等价。后面的新 token 去 attend 这段 K/V 时，读到的数与自己重算一遍完全一样， 因此命中不改变任何一个输出 token——这就是 check 16 要验证的「与不开缓存逐 token 一致」。

两个「必须」，缺一不可
① 从位置 0 开始：位置 5 的 K/V 依赖 0..5，不能脱离前 5 个 token 单独共享。
② 按块对齐：缓存以「整块」为单位登记与复用（见下一节的哈希），半块不参与共享。
只要这两条满足，命中就是安全的；违反任何一条都可能读到「位置或前文不一致」的 K/V，输出就错了。

块级链式哈希：怎么给前缀发「身份证」 Learn

缓存要能快速判断「这段前缀我见过没有」，就得给每一整块前缀算一个指纹（哈希）当键。 难点在于：位置 i 的 K/V 依赖它前面所有 token，所以指纹也必须把前文一起编进去—— 否则两段「本块相同、但前文不同」的块会撞成同一个键，导致错误共享。做法是链式哈希（chained hash）： 每一块的哈希都接着上一块的哈希继续算，于是第 k 块的指纹自然包含了第 0..k 块的所有 token。

本课用 FNV-1a 逐 token 更新（对应 prefix_cache.py 的 _block_hashes）：

h = OFFSET                              # 起始常量（整段只初始化一次，不在块之间重置）
对每个整块 block_k：
    对块内每个 token t：
        h = (h XOR (t + 1)) × PRIME     # 逐 token 折进哈希；(t+1) 避开 token id 0
    hashes[k] = h                       # 此刻的 h 即 block_k 的链式指纹（含 0..k 全部 token）
  • h：滚动哈希值，不在块之间清零——这正是「链式」的关键，让 hashes[k] 携带前文。
  • t + 1：token id 加 1，避免 id=0 折进去时不改变哈希。
  • 只对 len(token_ids) // block_size 个整块算哈希，末尾不满一块的零头不算 （不满块的前缀不参与共享，正对应「至少留 1 个 token 跑 prefill」的规则）。
  • 输出：一个哈希列表，第 k 个就是「前 (k+1) 块前缀」的身份证；缓存用它做键 （map: 哈希 → 物理块 id）。

小规模手算：链式哈希如何拦住「错误共享」 Learn

FNV 的真常量是 64 位大数（OFFSET=1469598103934665603、PRIME=1099511628211），手算不便。 我们换一个玩具哈希把链式结构算清楚——公式换了，但「不在块之间重置、逐 token 折入」的骨架和真代码完全一样。 取 block_size = 2，起始 h = 17，更新式 h = h × 31 + (t + 1)。请求 A 的前缀 token 是 [5, 7, 9, 3]（2 个整块）：

引用计数：谁在「按住」共享块不让它被回收 Learn

共享块最怕两件事：用着用着被别人回收了，或者没人用了却一直不回收（泄漏）。 解决办法就是 Lesson 13 的引用计数——一块被几个「主人」引用，refcount 就是几，归零才真正回收。前缀缓存里 refcount 的变化：

时刻动作b0 的 refcount
A prefillA 分配 b01（A 持有）
A 结束 on_finish缓存登记 b0：incref；随后 A kv.free()1（A 释放，但缓存按住了）
D attach 命中D 借用 b0：incref2（缓存 + D）
D 结束D kv.free()1（只剩缓存按住）
缓存被 LRU 逐出_evict_if_needed 对 b0 free0 → 真正回收

关键洞察：如果 on_finish 时缓存不 incref，A 一 free，b0 的 refcount 就归零被回收， 后面 D 再想复用就扑空了——所以「缓存自己也算一个引用」是前缀共享能成立的必要条件。反过来，容量有限， 缓存用 LRU 逐出最久没被 touch 的键（order 列表越靠后越新），逐出时 decref， 让不再热门的前缀把物理块还回去。

省了多少：块数与内存 Learn

本课 tiny model：hidden=32、layers=2、heads=4、kv_heads=2（GQA）、head_dim=8。 一个 token 每层的 K/V = 2 · kv_heads · head_dim = 2·2·8 = 32 个数，两层共 64 个数。 block_size=4 的一个物理块装 4 个 token 的 KV = 4 × 64 = 256 个数。

场景（前缀 8 token = 2 块，100 条请求）物理块占用省下的前缀块
不开前缀缓存每条各自 2 块 → 100×2 = 200 块0
开前缀缓存2 块共享给全部 100 条 → 2 块198 块（≈ 198×256 = 50688 个数）

省的是两样东西：一是那 99×2 块前缀的物理显存（大家共用一份）， 二是那 99×8 个前缀 token 的prefill 计算。前缀越长、请求越多，两样都省得越夸张—— 这正是长 system prompt 高并发场景下前缀缓存不可或缺的原因。

常见误解 Learn

误解为什么错
「只要中间某段 token 相同就能共享」不行。位置 i 的 K/V 依赖 0..i 的全部 token 和绝对位置。共享必须从位置 0 起、连续命中；attach 一遇未命中就 break。
「共享块只是省内存」更重要的是省计算：命中意味着跳过这段 prefill（num_computed_tokens 前移）。省显存是附带的，省算力才是主目标。
「命中可能让输出略有偏差」命中是逐值精确等价，不是近似：因果 + 绝对位置保证相同前缀的 K/V bit 级相同。check 16 验证「与不开缓存逐 token 一致」。
「请求结束后缓存块就该释放」恰恰要留着才有意义——缓存自己 incref 持有一份引用，请求 free 后块仍存活，供后续请求复用；容量满了才 LRU 逐出。
「整段 prompt 都命中最省」会漏 bug：若把最后一块也命中，就没 token 去跑 prefill、拿不到首个输出 token 的 logits。所以 attach 里至少留 1 个 token（shared*bs ≥ prompt_len 时回退一块）。
「块哈希只 hash 本块 token 就够」那样「本块同、前文不同」的两块会撞键 → 错误共享。必须用链式哈希把前文一起编入（h 不在块间重置）。
