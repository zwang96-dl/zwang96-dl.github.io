为什么需要这个知识 Learn

前面几课我们已经能让一条请求高效地 prefill + decode（KV Cache、 Prefill/Decode）。可真实服务里请求是一条接一条、错峰到达的： 有的刚发来、有的还在解码、有的马上就要写完最后一个字。这一课要回答两个问题——

  • 每条请求现在处于什么阶段？ 这就是「请求状态机」：用 4 个状态 WAITING / RUNNING / FINISHED / ABORTED 精确描述一条请求的一生。
  • 引擎每一步该跑谁？ 这就是「迭代级调度 / continuous batching」： 不等整批跑完，每一次前向都重新决定——放新请求进来、推进老请求、送走写完的请求。

把这两件事搞清楚，你就拿到了后面所有课的运行框架： 调度策略决定「先跑谁」，分页/块管理决定「KV 放哪」， 它们全都挂在这一课的状态机与迭代循环上。

先看问题：static batching 到底浪费了什么 Learn

先看「不这么做会怎样」。设有三条错峰到达的请求（本课动画就用这三条；prompt 长度按 byte-level tokenizer算：每个 UTF-8 字节 = 1 个 token，句首再加 1 个 BOS）：

请求promptprompt token 数要生成到达迭代
A"Hello"5 字节 + BOS = 64 个第 0 步（最早）
B"Hi"2 字节 + BOS = 33 个第 1 步
C"GPU"3 字节 + BOS = 43 个第 3 步（最晚）

老办法 static batching（请求级批处理）：把当前能凑的请求打成一个「固定批」， 整批一起从头跑到尾，全批都写完了才开始下一批。代入上面的数字：

  • 第 1 步凑出批 {A, B}（C 还没到）。这一批要跑到 A、B 都写完才结束。
  • B 只要生成 3 个 token，在第 4 步就写完了——但它的「座位」被锁在批里空转， 一直陪 A 跑到第 5 步（A 要 4 个 token）。这 1 步是纯浪费的算力。
  • C 在第 3 步就到了，却因为「批还没跑完、不能中途加人」，只能干等到第 5 步， 第 6 步才轮到它 prefill，第 9 步才写完。
代价（用具体数字）
static batching 跑完三条请求需要 9 个迭代；而下面的 continuous batching 只要 5 个迭代（后面手算给出逐步过程）。差距来自两处浪费：已写完的座位空转 （B 白占 1 步）＋后到的请求排长队（C 干等 3 步）。请求越错峰、长短越不齐，这种浪费越大。

直觉类比 Learn

一句话解释
Request 状态机 = 给每条请求贴一张「现在在哪个阶段」的标签； continuous batching = 引擎每跑一步就重新点一次名，谁能上、谁下车，随时调整。

类比：拼车（而不是包车）。 static batching 像「包车」：一车人必须同时上、同时到， 哪怕你早就到站，也得陪全车绕到终点；后来的人只能等下一班空车。continuous batching 像「网约拼车」： 司机每到一个路口都看一眼——有空座就顺路捎上新乘客（准入），到站的随时下车（完成）， 座位立刻让给别人。车（GPU）几乎不空跑，这就是吞吐更高的原因。

类比的边界（别被带偏）
拼车里「上车/下车」发生在路口之间的任意时刻；而引擎里状态切换只发生在 迭代边界——一次前向（forward）跑完、下一次开始之前。所以叫「迭代级」调度， 不是「token 级」随时打断。一条请求在一次前向的中途，状态是不会变的。

精确定义：Request 是一个状态机 Learn

一条请求的一生，就是在这 4 个状态之间转移。对应源码 mini_vllm/scheduler/request.py 里的 RequestStatus 枚举：

状态存在哪几个字段里 Learn

RUNNING 其实分成两个子阶段：还在算 prompt（prefill） 和 已在逐字生成（decode）。区分它们不靠状态枚举，而靠一个计数器 num_computed_tokens——已经算进 KV 的 prompt token 数。逐符号看：

字段 / 属性含义以请求 A（prompt=6，要生成 4）为例
num_prompt_tokensprompt 有多长恒为 6
num_computed_tokens已算进 KV 的 prompt 数从 0 涨到 6；到 6 就 prefill 完
prefill_donenum_computed_tokens ≥ num_prompt_tokens算满 6 后变 True
output_token_ids已生成的 token（decode 产物）长度从 0 涨到 4
num_generatedlen(output_token_ids)= 已生成个数
is_finished()生成够数 或 末尾是 EOSnum_generated ≥ 4 时为 True
是否结束怎么判定（is_finished() 逐行）
if num_generated >= max_new_tokens:   return True   # 写够了
if stop_on_eos and output[-1] == EOS_ID: return True # 主动写出了结束符
return False
两条任一成立即结束。本课的手算里没触发 EOS，所以每条请求都是「写够 max_new_tokens」才结束。

小规模手算：三条错峰请求的一次完整调度 Learn

现在用 continuous batching 把 A / B / C 跑一遍，看状态机如何逐迭代转移。 调度参数取本课动画的设置：max_num_seqs = 3（最多 3 条同时 RUNNING）、 token 预算 = 16（一步最多算 16 个 token）。每条 prompt（≤6）都能在一步内 prefill 完 （不用切块，chunked prefill 是 Lesson 15 的话题）。decode 每步每条只算 1 个 token。

迭代调度器这一步安排本步 token 数 ABC关键事件
1准入 A、B → 各 prefill（A 6 个 + B 3 个）6+3 = 9 RUN·gen1RUN·gen1WAIT（未到） A、B：WAITING→RUNNING
2A、B 各 decode 1 个1+1 = 2 RUN·gen2RUN·gen2WAIT（未到） C 第 3 步才到，仍在等
3A、B 各 decode 1 个 ＋ 准入 C prefill 4 个1+1+4 = 6 RUN·gen3FINISHEDRUN·gen1 C 中途加入；B 写满 3 个→完成，归还 KV
4A、C 各 decode 1 个1+1 = 2 FINISHED—RUN·gen2 A 写满 4 个→完成，归还 KV
5C decode 1 个1 ——FINISHED C 写满 3 个→完成；队列空，引擎退出

盯住第 3 步：这正是 continuous batching 的灵魂——C 在 A、B 正 decode 的同一步里被准入并 prefill， 而 B 在同一步写完后立刻腾出座位与 KV。没有任何一步「等整批」。全程 5 个迭代， 对比 static batching 的 9 个——省下的正是前面那两处浪费。

同一时刻，运行 batch 的构成在逐步变化（这就是「continuous」）
迭代1 A prefill B prefill
迭代2 A decode B decode
迭代3 A decode B decode C prefill ⬅ 新加入
迭代4 A decode B 已下车 C decode
迭代5 A 已下车 C decode
同一个 batch 里 prefill（紫）和 decode（青）混排在一起跑；每一步的成员都可能不同。 static batching 里，这五行会被强行凑成「同上同下」的固定方阵，中间的空位只能空转。

常见误解 Learn

误解为什么错
「新请求要等当前这批全跑完才能进」 那是 static batching。continuous batching 每个迭代都尝试准入——手算里 C 就是在 A、B decode 的同一步加入的。
「continuous 是在一次前向的中途随时加人/踢人」 状态只在迭代边界切换。一次 forward 跑的过程中，batch 成员固定不变——所以叫「迭代级」而非「token 级」抢占。
「请求写完了留着无所谓」 不摘走它，它会一直占着 max_num_seqs 的座位和 KV 块，把并发度和显存活活堵死。必须 remove_finished() 并 kv.free()。
「用 status == RUNNING 就能判断该 prefill 还是 decode」 不能。prefill/decode 的区分靠 prefill_done（即 num_computed_tokens ≥ num_prompt_tokens），不是靠状态枚举。
「FINISHED 和 ABORTED 差不多」 都释放资源，但 FINISHED 的输出是有效结果，ABORTED 是被中止、结果作废——语义完全不同。

对照 LLMEngine.run 的骨架
while scheduler.has_unfinished:            # 还有 WAITING 或 RUNNING 就继续
    out = scheduler.schedule(it, free, bs) # 这一步跑谁、各多少 token（含准入新请求）
    for item in out.items:
        _run_item(item, it)                # prefill 一块 或 decode 一个 token → 采样
    finished = scheduler.remove_finished() # is_finished() 为真的摘出 → RUNNING→FINISHED
    for r in finished: r.kv.free()         # 归还 KV 块，座位让给别人
这段 5 行循环，就是 continuous batching 的全部。手算表格里的每一行，都是它转一圈的结果。
