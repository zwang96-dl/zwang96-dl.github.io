为什么需要这个知识 Learn

前面 16 课，我们一件件造好了零件：tokenizer（把文本切成字节 token）、Request 状态机、 Scheduler（决定这一步跑谁）、BlockAllocator + 分页 KV（管显存）、TinyTextModel（跑 forward）、 Sampler（挑下一个 token）。但零件散着放，并不会自己动——没有一个「主循环」 把它们按正确顺序串起来，你连一句话都生成不出来。这一课就是造那台发动机。

先看问题：不组装引擎会怎样？
假设同时来 3 条请求（下方动画里的 chat1 / chat2 / long）。最朴素的做法是 一条跑完再跑下一条：chat1 生成 5 个 token 期间，模型每步只处理 1 个 token， 可 GPU 明明还能同时处理好几条——算力大量闲置；与此同时 chat2、long 只能干等， 它们的「首字延迟」被硬生生拖长。真实 vLLM 的核心价值，就是把这 3 条请求在同一步里一起推进 （continuous batching），让算力吃满、让每条请求都尽早出字。这台发动机怎么转，就是本课要讲透的。

引擎要组装哪些零件 Learn

看 LLMEngine.__init__：它把前面每一课的成果持有成自己的字段。可以把整台引擎 想成一张零件清单——每个零件对应一课，缺一个引擎就转不起来：

零件（引擎字段）来自哪课在 loop 里干什么
self.tok ByteTokenizerL2把 prompt 字符串编码成字节 token（一字节=1 token，中文一字=3 token）
self.scheduler SchedulerL11 / L15每步决定「跑谁、各处理多少 token」，受两个预算约束
self.allocator BlockAllocatorL13按物理块借/还显存，全局共享一份
r.kv PagedKVCache（每请求一份）L7 / L14缓存该请求每层的 K/V，按块寻址
self.model TinyTextModelL6跑 forward，输入 token+位置+KV，输出 logits
self._samplers Sampler（每请求一份）L9从 logits 里挑出下一个 token id
self.prefix_cache（可选）PrefixCacheL16相同 prompt 前缀的请求共享物理块
一个关键的组装细节：全局共享的 kv_store
self.kv_store = {"k": {}, "v": {}} 是所有请求共用的一份 KV 行存储， 而每条请求的 PagedKVCache 只持有「块号 → 行」的映射。正因为底层存储是共享的， 两条请求命中同一个前缀块时，才能真正指向同一份物理数据（Lesson 16 前缀共享的地基）。

直觉类比：引擎就是餐厅的「出餐循环」 Learn

一句话解释
engine loop = 一个反复执行「排班 → 开火 → 装盘 → 送走已完成的桌」的循环， 直到所有客人都吃完离店。

类比：一间只有几个灶眼的厨房。灶眼数 = max_num_seqs（最多同时几桌在做菜）， 每轮能切的食材总量 = max_num_batched_tokens（token 预算）。新客人进门先在候位区（WAITING）， 厨师长（Scheduler）每一轮看现有灶眼和食材预算，决定这一轮给哪几桌做、各做多少： 有的桌在备料（prefill，一次切一大批），有的桌只差最后一道（decode，一次一个）。 哪桌上齐了（生成完），就收台、把餐具还回去（free KV block），腾出资源给下一桌。

类比的边界（别被带偏）
餐厅里不同桌的菜是各做各的；而引擎里 continuous batching 是在同一次 forward 里 把多条请求的 token 一起送进模型——它们共享一次矩阵运算，但每条请求的 KV 完全独立 （各自的 block table），所以结果逐 token 等于「单独生成」。共享的是算力批次， 不是上下文——这条边界是本课正确性的命根子（见下方常见误解）。

engine loop 精确定义：一轮迭代做四件事 Learn

看 LLMEngine.run()：外层是 while self.scheduler.has_unfinished， 只要还有 WAITING 或 RUNNING 的请求就继续。每一轮迭代（iteration）严格做四件事：

#步骤调用输入 / 输出（含具体 shape）
1排班scheduler.schedule(it, num_free, block_size) 输出 out.items：一串 ScheduledItem(request, is_prefill, num_tokens)；decode 的 num_tokens 恒为 1
2开火+装盘for item: _run_item(item, it) 对每个 item 跑 model.forward(toks, positions, kv) → logits，再 sampler(logits[-1]) → 1 个 token id
3收台scheduler.remove_finished() 返回本轮完成的请求（num_generated ≥ max_new 或末尾是 EOS）
4还餐具r.kv.free()（或 prefix_cache.on_finish(r)） 把该请求占的物理块还给 allocator——无泄漏

盯住第 2 步里 prefill 与 decode 的形状差别，这是整台引擎最核心的一处：

阶段喂进 forward 的 token 数positionslogits shape采样从哪取
prefill（备料）n 个（prompt 的一块，可 >1） [start, …, start+n-1](n, 259) 只在 prompt 全部算完（prefill_done）后，从 logits[-1]（形状 (259,)）采第 1 个输出 token
decode（收尾）恰好 1 个（上一步生成的 token） [num_prompt + num_generated − 1](1, 259) 每步都从 logits[-1] 采 1 个 token

这里的 259 正是本模型的 vocab_size（256 个字节 + BOS/EOS/PAD）。decode 之所以每步只喂 1 个 token 还能得到正确结果，靠的正是 Lesson 7 的 KV Cache：历史 K/V 都在 r.kv 里， 不必重算。

为什么第 1 个输出 token 在 prefill 阶段产生，而不是 decode？
一个新 token 需要「它前面所有 token 的信息」。prompt 的最后一个位置，在 prefill 跑完时其 logits 已经包含了整段 prompt 的上下文，从它直接采样就是「续写的第 1 个字」。所以代码在 _run_item 里：if r.prefill_done: tok = sampler(logits[-1]); r.first_token_iter = it—— 首 token 的诞生迭代号被记下来，正是后面算 TTFT 的原料。

取 max_num_seqs=3、token 预算 max_num_batched_tokens=16（与动画一致）、fifo、开启 chunked prefill。 引擎迭代从 1 开始，请求在 arrival ≤ it 时才可被准入。下面是一次可能的调度轨迹：

迭代本轮 items（谁·阶段·token 数）预算用量各请求进展
1chat1 · prefill · 3；chat2 · prefill · 69 / 16 两条都算完 prompt → 各出第 1 个输出 token（first_token_iter=1）。long 未到达。
2chat1 · decode · 1；chat2 · decode · 1；long · prefill · 1416 / 16 chat1/chat2 各 +1 token；long 到达并切块：20 个 prompt 先算 14 个（预算封顶）。
3chat1 · decode · 1；chat2 · decode · 1；long · prefill · 68 / 16 long 补算剩下的 6 个 prompt → prefill_done → 出第 1 个输出 token（first_token_iter=3）。
4三条全 decode · 各 13 / 16 chat1 已有 3 个输出、chat2 有 3、long 有 1，继续各 +1。
……… chat1 满 5 个输出 → 完成、free() 归还块；随后 chat2 满 6、long 满 4 依次收台。

注意第 2、3 轮：chat1/chat2 的 decode 和 long 的 prefill 混在同一轮里跑——这就是 continuous batching 与 chunked prefill（L15）联手的效果：长 prompt 不会一次霸占引擎，短请求的 decode 也不被堵住。由此可直接读出两个延迟指标：

  • TTFT（首字延迟，迭代级）= first_token_iter − arrival： chat1 = 1−0 = 1；chat2 = 1−1 = 0；long = 3−2 = 1。
  • 完成迭代 = finish_iter − arrival，同理由收台那一轮的迭代号算出。

性能报告：四类指标各回答什么问题 Learn

run 17 会调 benchmarks/report.py 的 build_report()，把引擎跑完的 EngineResult 换算成一份报告。指标分两大阵营——延迟（单条请求的体感快慢） 与吞吐（系统整体的产能）——它们常常此消彼长：

指标公式（代码里怎么算）关注本例数值
avg/max_ttft_itersfirst_token_iter − arrival 求平均/最大延迟 平均 (1+0+1)/3 ≈ 0.67，最大 1
avg_completion_itersfinish_iter − arrival 求平均延迟由各请求收台迭代算出
token_throughput_per_stotal_output / wall_time_s吞吐 输出总数 = 5+6+4 = 15 token ÷ 墙钟秒
kv_utilizationpeak_blocks_used / num_blocks资源 峰值占块 ÷ 24（block_size=4）
prefix_cache.hit_rate（可选）命中前缀块 / 总请求资源/延迟仅开启前缀缓存时出现
为什么 TTFT 用「迭代号之差」而不是墙钟毫秒？
report.py 特意用迭代级代理（first_token_iter − arrival）：它 确定、可复现，同一份输入每次跑都一样，适合教学与回归测试；墙钟版本受机器负载抖动影响， 只作参考。真实 vLLM 报的是毫秒级 TTFT/TPOT，但「首 token 何时产生」这个语义是完全一致的。
与 naive 基线对比
报告里的 baseline_naive_processed_tokens 记录「若每条请求都像 Lesson 6 那样每步重算整段」 会处理多少 token。引擎靠 KV Cache + 批处理把这个数字大幅压下来——这就是把前 16 课的优化量化成一个数。

常见误解 Learn

误解为什么错
「多条请求一起 batch，输出会互相污染，和单独跑不一样」 共享的只是一次算力批次；每条请求有独立的 KV（独立 block table）， attention 只看自己的上下文。所以 check 17 的验收标准正是：引擎输出与「逐请求单独生成」逐 token 一致。
「decode 每步要把整段序列重新喂进模型」 只喂1 个新 token（forward([last], [pos], r.kv)），历史全在 KV 里。喂整段就退回朴素做法、白费算力。
「请求完成后不用管 KV，反正会自动回收」 不会。必须在 remove_finished() 后显式 r.kv.free()；否则块泄漏， 后续请求借不到块、引擎抛「调度停滞」。run() 收尾用 check_no_leak() 专门自检。
「第 1 个输出 token 是 decode 阶段产生的」 它在 prefill 跑完那一刻就从 prompt 最后一个位置的 logits 采出（if r.prefill_done）。 decode 从第 2 个输出 token 起才开始。搞反会把 TTFT 算晚一轮。
「TTFT 越低、吞吐就越高，是一回事」 常常相反：为压 TTFT 要优先给新请求做 prefill，可能挤占正在 decode 的请求，拉低整体吞吐。 延迟与吞吐是需要权衡的两类目标（调度策略 L11 就在调这个平衡）。
