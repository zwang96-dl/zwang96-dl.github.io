为什么需要这个知识 Learn

到这里，KV Cache（Lesson 7）已经能存历史 K/V，但它是一整条连续数组： 每来一条序列，就得预留 max_seq_len 那么长的空间。Lesson 12 已经算过—— 这会造成大量浪费（一条只用 20 个 token 的序列，却把 64 个 token 的位置全占了）。 本课要做的，是把这块显存切成固定大小的物理块（block），用一个「分配器」按需发放、用完回收。

这个分配器是后面所有内存魔法的地基：分页 KV（Lesson 14）靠它拿块、 前缀共享（Lesson 16）靠它的引用计数让多条序列安全地共用同一块。 所以这一课要彻底搞清四件事：块怎么发（allocate）、怎么回收（free）、怎么被多方共享而不出错（引用计数）、 以及三类经典 bug（OOM / double free / leak）为什么会发生、如何被主动拦下。

先看问题：不做分配器会怎样 Learn

本模型 max_seq_len=64，块大小 block_size=16（每块装 16 个 token 的 KV）。 假设显存池一共只有 4 个物理块（=64 个 token 的容量）。两种做法对比：

做法一条序列占多少块4 块能同时跑几条
预留式（不分块，按 max_seq_len 连续预留）永远 4 块（64/16），哪怕只用 20 个 token1 条
分配器 + 分块（按需申请）只占 ceil(20/16)=2 块2 条

同样 4 块显存，前者只能塞 1 条 20-token 序列（剩下 2 块被白白锁住），后者能塞 2 条——并发翻倍。 再算一次这块显存有多大：每块 = block_size × 每 token 每层(K+V) × 层数 = 16 × (2·kv_heads·head_dim) × layers = 16 × (2·2·8) × 2 = 1024 个数。省下的每一块都是实打实的显存。这就是要写分配器的动机。

直觉类比 Learn

一句话解释
Block Allocator = 一个物理块的「借还台」：谁要用就发一块（allocate）， 用完还回来（free）；还用一个计数器（refcount）记住「这块此刻有几个人在用」， 只有降到 0 才敢真正收回，避免收错。

类比：图书馆的储物柜。柜子数量固定（num_blocks），前台有一张「空柜清单」（free list）。 你来存东西，前台挑一个编号最小的空柜发给你；你取走东西、退柜，它才回到清单里给下一个人。 如果一份资料被两个人共用（合租一个柜子），前台会记「这柜有 2 个使用者」， 必须两个人都退了才清空——这就是引用计数。

类比的边界（别被带偏）
储物柜里放的是你的物品，而 block 里放的是模型算出的 K/V 向量（Lesson 7）； 更关键的一点：柜子之间不必物理相邻——序列可以持有块 [0, 3, 7] 这种不连续的组合， 靠一张「块表」把它们串起来（那是 Lesson 14 的事）。分配器只管发号与计数，不关心谁和谁挨着， 正是这种「不要求连续」消灭了 Lesson 12 的碎片问题。

精确定义：free list、refcount 与四个状态 Learn

分配器内部只有两样核心数据（对应源码 __init__）：

  • free list（空闲清单）：self._free，一个最小堆（heap）。 堆保证每次弹出的都是编号最小的空闲块——结果确定、可读、好 Trace。 初始 [0, 1, …, num_blocks−1] 全部空闲。
  • refcount 数组：self.ref[b] 记录第 b 块此刻有几个引用。 初始全 0。ref[b]=0 表示空闲，ref[b]≥1 表示在用（值就是使用者个数）。

每一块在任意时刻处于下面 4 个状态之一，状态只由 ref[b] 决定：

状态ref[b]含义在 free list 里吗
空闲 Free0没人用，可被 allocate是
独占 Owned1刚分配，一个序列在用否
共享 Shared≥2多条序列共用（前缀共享）否
非法 Invalid<0不该出现——一旦要变负就是 double free，主动报错—

还有一个纯算术的辅助函数 blocks_needed(n)：装 n 个 token 要几块。逐符号看——

blocks_needed(n) = { 0 若 n=0 max(1, ceil( nblock_size )) 若 n>0 
  n           = num_tokens，要缓存的 token 数（输入，标量整数）
  block_size  = 每块能装的 token 数（本课 = 16）
  ceil(·)     = 向上取整（装不满一块也要占整块）
  输出         = 需要的块数（标量整数）

代入具体数字：blocks_needed(16)=1、blocks_needed(17)=2（多出 1 个 token 就得再占一整块）、 blocks_needed(20)=2、blocks_needed(64)=4。注意 free list / refcount 都是「按块」的整数运算， 不涉及任何浮点——分配器只搬「块编号」，从不碰块里那 1024 个 K/V 数。

小规模手算：4 块的一生 Learn

为什么要引用计数，而不是布尔「已用/空闲」？ Learn

如果只用一个布尔标志（used[b] = True/False），单条序列独占时够用；但一旦要共享就崩了。 设想前缀共享（Lesson 16）：序列 A 和 A' 有相同前缀，共用 b0。

布尔标志 used[b]引用计数 ref[b]
A 与 A' 共用 b0只能记 True，不知道有 2 个用户ref=2，如实记录
A 先结束、free(b0)置 False → b0 被回收 → A' 的数据被别人覆盖！ref 2→1，不回收，A' 安全
A' 再结束、free(b0)（已经错了，无从谈起）ref 1→0，此刻才安全回收

结论：「有没有人用」是布尔，「有几个人用」才是引用计数。共享的本质是「一块多主」， 只有计数能回答「现在还能不能收」。这也是 incref 存在的唯一理由——它就是「多一个人来登记使用」。

三条红线（会被主动拦下报错）：
· 从 Free 再 free（ref 要变 −1）→ double free
· 对 Free 的块 incref（incref 一个没分配的块）→ RuntimeError
· free list 空时 allocate → OOM（MemoryError）

为什么这三条要「主动报错」而不是默默容忍？因为它们都会导致数据静默损坏： double free 会把还在用的块发给别人（覆盖数据）；对未分配块 incref 会凭空产生一个「幽灵引用」， 让这块永远收不回（leak）；OOM 若不报，就会越界访问不存在的块。让 bug 大声崩溃，好过让它悄悄算错。

常见误解 Learn

误解为什么错
「free(b) 一调用，块就立刻回收」只有 refcount 减到 0 才回收。共享块（ref≥2）free 一次只是减 1，块仍在用。
「用布尔『已用/空闲』就够了」共享时会出错：一个用户 free 就把还有人用的块收走，覆盖别人数据。必须计数。
「OOM 说明程序写错了」不一定。free list 空只是显存暂时不够，真实系统会靠抢占/换出腾块（本课在调度层处理）。分配器只负责如实抛 MemoryError。
「分配器要保证序列的块是连续的」恰恰不需要。分配器只发编号、只管计数，序列拿到 [0,3,7] 也没问题——「不要求连续」正是它消灭碎片的关键。
「incref 和 allocate 都让 ref 变大，是一回事」allocate 从 free list 取一个新块并把 ref 从 0 置 1；incref 是对已在用的块把 ref +1。对空闲块 incref 会直接报错。
