为什么需要这个知识 Learn

到上一课为止，调度器（Lesson 11）已经能决定「这一步跑哪些请求」， 分页 KV（Lesson 14）也能放下它们的缓存。但还有一个现实约束没露面： 一次迭代到底能处理多少 token？ GPU 一步能并行吃进的 token 是有限的—— 喂太多会爆显存、也让这一步本身变慢。这条上限就是 max_num_batched_tokens （本课默认 64，见 SchedulerConfig）。

一旦有了这条预算线，麻烦就来了：如果一个用户发来 很长的 prompt， 而另一个用户只打了个 "Hi"，朴素做法会把整段长 prompt 在一步里全部 prefill， 短请求只能干等——它的首 token 延迟（TTFT，time to first token）被这段长 prompt 整个拖住。 更糟的是：如果这段 prompt 比整条预算还长，它会一步都排不进去，永远卡住。

Chunked prefill（切块预填）就是来治这两件事的：把长 prompt 切成若干小块、 分几步算完，中间穿插别的请求的 decode，让短请求不必苦等，也让超预算的长 prompt 至少能一点点推进。 这一课就把这条预算线和切块逻辑讲透。

先看问题：不切块，长 prompt 会怎样 Learn

用具体数字把问题摆出来。设一次迭代的预算 max_num_batched_tokens = 8， 两个请求同时到达：

  • long：prompt = "你好世界"。本课 tokenizer 是 byte-level， 一个汉字 = 3 个 UTF-8 字节 = 3 个 token，所以这段是 4 × 3 = 12 个 token。
  • short：prompt = "Hi" = 2 个 token（两个 ASCII 字节）。

关掉 chunked prefill 时，prefill 必须一次算完整段。long 需要 12 个 token， 但预算只有 8：12 > 8，这一步放不下它；下一步预算重置回 8，还是放不下…… 于是 long 每一步都被跳过，永远排不进去：

迭代long 需要 prefill预算能否放下结果
012（整段）8否（12>8）跳过，long 仍等待
112（整段）8否跳过
…128否永远跳过 → 调度停滞
核心痛点
只要单条 prompt 长度 > 整条 token 预算，且不切块，这个请求就一步都跑不了—— 不是慢，是彻底卡死。run 15 在这种配置下会明确报「调度停滞」。这正是 chunked prefill 存在的第一个理由。

直觉类比 Learn

一句话解释
token 预算 = 一次迭代最多能端上桌的 token 总量； chunked prefill = 把一大锅长 prompt 分成几托盘端，端的间隙还能顺手给别的桌上菜（decode）。

类比：食堂打饭窗口。窗口一次只能端出一托盘（预算 8 份）。有个人点了一大锅（12 份）的菜—— 如果规定「必须一托盘端完」，这锅菜永远端不出去，后面只想打二两饭（short 的 2 个 token）的人也跟着堵死。 改成分托盘端：第一托盘先端 8 份，第二托盘端剩下的 4 份，中间的空档还能给别人盛饭。 大家都能往前挪，没人被无限期堵在门口。

类比的边界（别被带偏）
分几托盘端，不会改变最终每个人碗里的菜。切块只改「每一步算多少 token」， 绝不改变模型算出的结果——因为每个 token 的位置（position）和它写进 KV 的内容都不变， 切在哪里都一样。这一点和 Lesson 7 里「decode 只喂 1 个 token 仍与重算整段逐值一致」 是同一个道理，只是这里一次喂的是「一块」而非「一个」。

精确定义：两条预算线与一步的 token 账 Learn

调度器每次迭代受两条独立的预算线约束（都在 SchedulerConfig 里）：

符号含义本课默认约束的是
max_num_seqs同时 RUNNING 的请求条数上限4并发度（有几条在跑）
max_num_batched_tokens一次迭代处理的 token 总数上限64单步计算量（这步算多少 token）

本课主角是第二条。一次迭代实际处理的 token 数，是所有被安排的请求各自 token 数之和，逐符号看：

本步 token 数 = ∑每个 prefill 请求 chunk + ∑每个 decode 请求 1 ≤ max_num_batched_tokens
  • decode 请求：每步永远只产生 1 个新 token，所以恰好花预算里的 1。
  • prefill 请求：可花 1 … remaining_prompt 之间任意块大小。开启切块时块大小取 min(剩余 prompt, 当前剩余预算)；关闭切块时块大小 = 整段剩余 prompt（放不下就整步跳过）。
形状（shape）
被安排进这一步的所有 token 会拼成一条扁平序列喂给模型：输入形状 (本步 token 数,)，只要 ≤ 64 即可。例如某步安排「long 的 8 个 prefill token + short 的 1 个 decode token」， 模型这步的输入就是长度 9 的一维张量；每个 token 各自带着自己的绝对位置（position）去查 KV。

为什么切块与混排不改变输出 Learn

会不会「切一刀」就把结果切坏了？不会。关键是 num_computed_tokens 记着「这条 prompt 已经算到第几个 token」。 第二块从断点续算：迭代 0 算了位置 0..7，迭代 1 的这一块就从位置 8 开始， 算位置 8..11 的 q/k/v，把新的 k/v 追加进 KV，再让它们去查缓存里已有的位置 0..7。

这和 Lesson 7 的 KV Cache 是同一套机制：历史 K/V 一旦算出就永久有效 （第 i 个位置的 K/V 只依赖前 i 个 token，与后面切没切、什么时候算无关）。 decode 是「每步续算 1 个」，chunked prefill 只是「每步续算一块（多个）」——本质完全一样， 所以无论切成 12=8+4 还是 12=4+4+4，甚至一次算完， 最终每个位置的 logits 逐值相同。

形状（shape）在断点续算时怎么走
迭代 0：输入 (8,)（位置 0..7），KV 长度 0 → 8。
迭代 1：输入 (4,)（位置 8..11），新 q 去看「缓存 8 行 + 本块 4 行」= 全部 12 个位置，KV 长度 8 → 12。
因果掩码保证每个位置只看到 ≤ 自己的位置，与「一次喂 12 个」得到的掩码完全一致。

常见误解 Learn

误解为什么错
「chunked prefill 是为了省显存」它省的是单步的计算峰值和排队延迟，不是显存。整段 prompt 的 KV 最终一样要全部存下；切块只是把「一步算完」摊成「几步算完」。
「切块会让长 prompt 更快出结果」恰恰相反：切块通常让长 prompt 的 TTFT 略微变慢（分几步才算完），代价是换来短请求的 TTFT 大幅变好。这是一次公平性权衡，不是纯加速。
「decode 也要按预算切块」decode 每步天生只算 1 个 token，没有可切的东西；预算里它恒定占 1。可切的只有 prefill。
「切块可能改变生成的 token」不会。位置和 KV 内容与切点无关，12=8+4 和一次算 12 逐值一致——切块只影响什么时候算，不影响算出什么。
「只要预算够大就不用切块」预算大确实能一次装下更多，但也意味着单步更慢、更容易让短请求等更久；且一旦出现比预算还长的 prompt，不切块仍会直接停滞。切块是结构性保障，不是预算不够时的补丁。
