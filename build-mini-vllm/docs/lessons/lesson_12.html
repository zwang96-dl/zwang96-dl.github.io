为什么需要这个知识 Learn

Lesson 7 告诉我们：KV Cache 是花显存换速度，而且它随序列长度线性增长。 但那一课只回答了「一条序列要占多少内存」，没回答一个更要命的问题—— 你在什么时候、按什么粒度把这块内存「分」给一条序列？

这个「怎么分」的决策，直接决定一张卡能同时塞下几条请求。分得笨，显存明明还剩一大半， 却告诉你「装不下了」；分得巧，几乎每一格显存都用在刀刃上。这一课就是要用具体数字 看清「连续预留」这种笨办法到底浪费了多少，从而理解 vLLM 为什么非要发明分页 KV不可。 它是后面 Block Allocator、分页 Attention 一整条主线的动机。

先看问题：连续预留到底浪费了什么 Learn

矛盾出在一个残酷的事实：请求还没跑完，你根本不知道它最终会生成多长。 可 KV Cache 又要求「每一层每个 token 的 K/V 挨着放」（连续内存才能高效做 attention）。 朴素分配器只能采取最保守的策略：一上来就按上限 max_seq_len=64 给每条序列预留一整段连续空间， 免得中途长过头、需要「搬家」。

先统一度量单位：本课把一个 token 在所有层的 K/V 记作一个「格子 / slot」。 用 Lesson 7 的公式，本模型每格 = 2 · num_kv_heads · head_dim · num_layers = 2·2·8·2 = 64 个数 （2 是 K+V，kv_heads=2 是 GQA，head_dim=8，layers=2）。下面所有「浪费 N 格」都可乘以 64 换算成真实数字量。

现在来 5 条并发请求（seq_lens = [5, 12, 3, 20, 7]，一共只用到 5+12+3+20+7 = 47 格）， 看看连续预留发生了什么：

请求实际用到（格）连续预留（格）白占（格）
s056459
s1126452
s236461
s3206444
s476457
合计47320273

预留了 64×5 = 320 格，真正用到 47 格——273 格（85.3%）在空转。 换成真实数字量就是 273×64 = 17472 个数被白白占着。更糟的是：这 5 条请求本可以 只用 47 格就装下，现在却因为「每条都吃满 64 格的坑」而提前把显存占光， 第 6 条请求即使很短也进不来。这就是「显存还剩一大半，却报装不下」的由来。

直觉类比 Learn

一句话解释
连续预留 = 「怕客人临时加人，干脆给每桌都预定一整张 64 座的大圆桌」； 结果 5 桌客人（每桌才来 3~20 人）就把整个宴会厅订满，门口还排着队进不来。

类比：餐厅订座。你不知道每桌最终来几个人（≈ 不知道序列最终多长）， 为了保险，索性每桌都锁定一张最大的 64 座桌（≈ 按 max_seq 预留）。 大部分桌子空着一多半（≈ internal 碎片），而且这些空座还分散在不同桌子上，拼不到一起 给一个 30 人的团用（≈ external 碎片）。分页的思路则是：把大桌拆成一个个 8 座小桌（block）， 来几个人就开几张，用完归还给公用桌池——几乎不浪费，还能灵活拼桌。

类比的边界（别被带偏）
餐桌是物理挪不动的，而分页 KV 的精髓恰恰是逻辑地址和物理地址解耦： 一条序列在「逻辑上」是连续的 0,1,2,…，但这些块在物理显存里可以东一块西一块。 真正让分页威力翻倍的是这层「查表映射」，餐桌类比表达不出这一点——那是 Lesson 13 的主角。

精确定义：两种碎片与分页公式 Learn

把上面的直觉写成能计算的定义。记第 i 条序列实际用到 s_i 格， 并发 N 条，上限 M = max_seq_len，块大小 B = block_size。

① Internal fragmentation（内部碎片）：分到了却没用
已经分配给某条序列、但它没用上的空间。连续预留下每条分 M 格： internal_waste= ∑i (M−si) = N·M− ∑isi 逐符号：N·M = 总预留（reserved），Σ s_i = 总实际用量（used），两者之差就是内部碎片。 浪费率 = internal_waste / (N·M)。
② External fragmentation（外部碎片）：有空间却凑不出连续一段
空闲总量足够，但被切成不连续的小洞，拼不出一段连续内存去满足新请求。 连续预留把显存切成一段段 M 大的坑，某条序列结束释放后留下一个洞； 下一条需要连续 M 格时，即使总空闲够，也可能因为「没有一整段连续的 M」而失败。 这一项难用单个数字概括，但它和 internal 一样，都是「连续 + 定长预留」逼出来的。
③ 分页分配：向上取整到块
分页不再按 M 预留，而是用几格算几块，每块 B 格，按需取整： pagedi= ⌈siB⌉·B pagedtotal= ∑i ⌈siB⌉·B paged_waste = paged_total − used
⌈·⌉ 是向上取整。每条序列只可能浪费最后一块里没填满的零头， 最多 B−1 格。且因为块是定长的、物理可不连续，external 碎片也一并消失（块能自由复用）。

常见误解 Learn

误解为什么错
「按 max_seq 预留只是稍微多占一点，问题不大」 本例就浪费了 85.3%（273/320 格）。浪费随 max_seq_len 和并发数线性暴涨， 上限调大、请求变多时会吃掉大半张卡。
「分页零浪费」 分页仍有 internal 碎片——最后一块没填满的零头（本例 17 格，26.6%）。 它只是把浪费压到「每条最多 B−1 格」的常数级，且不再随 max_seq 增长。
「碎片是因为显存总量不够」 恰恰相反：本例显存够装 47 格的 6~7 倍，是「连续 + 定长预留」的分配策略 把它切成了浪费，不是量不够。
「block_size 越小越省内存，就该取 1」 块越小 internal 零头越小，但每条序列的块数变多，块表更长、管理开销更大， attention 也更碎。这是 Lesson 13 要权衡的取舍，不是越小越好。
「external 碎片靠多留点连续空间就能躲开」 只要「定长连续段 + 释放」的模式在，空洞就会不断产生；治本靠定长块 + 逻辑/物理解耦 （分页），让物理不连续也能拼给一条逻辑上连续的序列。

代码映射 Learn

本节所有数字都出自 experiments/lesson_12_kv_waste.py 的 run_experiment()（即「碎片计算」）。 对照着看每一行算的就是上面的哪个量：

reserved      = max_seq * len(seqs)              # N·M：连续预留总量 → 320
used          = sum(seqs)                        # Σ s_i：实际用量 → 47
internal_frag = reserved - used                  # ① 内部碎片 → 273（85.3%）
paged  = sum(math.ceil(s / block_size) * block_size for s in seqs)  # ③ ⌈s/B⌉·B 逐条向上取整 → 64
paged_waste   = paged - used                     # 分页浪费（块内零头）→ 17（26.6%）
对应关系
contiguous_waste / contiguous_waste_pct 就是定义 ①（internal）， paged_allocated / paged_waste_pct 就是定义 ③（分页向上取整）。 external 碎片是「连续段释放后凑不出连续空间」的定性问题，代码不单独打数，但正是它和 internal 一起 构成了下一课 Block Allocator 要解决的目标。
