为什么需要这个知识 Learn

前面十几课把多模态拆成了一地零件：chat template、image processor、vision encoder + projector、 embedding 合并、三层缓存、以及上一课的 visual budget。每一件单独都能跑， 但真实服务收到的从来不是单个请求，而是同一刻涌进来的一批：有的纯文本、有的带一张图、 有的带好几张图、有的带一段视频。本课就是把这些零件装进一个能同时喂多种请求、还不出错的引擎—— MultiModalEngine，这是整条多模态主线的集大成。

难点不在「能跑」，而在三条红线一条都不能踩：① 视觉编码器绝不能在 decode 阶段被重复调用； ② 一个请求的视觉 embedding 绝不能串到另一个请求；③ 显存/算力绝不能被一个带视频的大请求吃光、饿死别人。 这一课就是把这三条红线焊死，并用一份端到端报告（encoder 运行数 / 缓存命中 / 视觉 token / TTFT）证明它们成立。

先看问题：mixed batch 朴素跑法会崩在哪 Learn

先用本课的真实视觉配置把「一批请求」量化。图片 image_size=16、patch_size=8， 切成 2×2 = 4 个 patch，所以一张图 = 4 个 visual token；视频默认抽 4 帧、每帧当一张图， 即 4 帧 × 4 = 16 个 visual token。每个 visual token 都是一条 text_hidden=32 维向量， 和文本 token 一样占一个序列位置、进一份 KV Cache。下表是四类请求的成本：

请求类型序列里的 visual token需要跑几次 vision encoder
text-only（纯文本）00
单图4（2×2 grid）1
多图（2 张）82
视频（4 帧）164（逐帧编码）

如果不用引擎、拿单请求的思路硬套，三处会立刻崩：

  • 重复编码：视觉编码是最贵的一步。若每个 decode step 都重跑 encoder，一个视频请求生成 20 个 token 就要 4 帧 × 20 步 = 80 次编码，而正确答案是只在 prefill 编码 4 次——足足 20 倍的浪费。
  • 跨请求串视觉 embedding：若几个请求共享同一份状态/KV，带图请求 B 的视觉向量会「漏」进纯文本请求 A， A 会答出它根本没见过的图。这类 bug 极隐蔽，只有把每个请求隔离才能杜绝。
  • 大请求吃光资源：4 个视频请求一次全放进来就是 4 × 16 = 64 个 visual token 同时占 KV， 外加 4 × 4 = 16 次编码挤在一个迭代里，纯文本小请求被活活饿死、延迟爆炸。

一句话解释 + 直觉类比 Learn

一句话解释
MultiModalEngine = 一个迭代级循环：每一步先推进正在跑的请求（纯文本 decode）， 再按预算接纳新请求并做多模态 prefill（编码 + 合并 + 首 token），最后回收已完成的请求； 三层缓存让「一样的媒体」不必重复处理/编码。

类比：一间忙碌餐厅的后厨。「三段式循环」就是主厨的节奏——先把已经在灶上的菜翻炒一下往前推 （decode），再看还有没有余力接新单（准入），接了单先做最费时的备料（prefill 里的视觉编码）， 最后把出好的菜端走清台（回收）。三层缓存就是备料台：切好的同款配菜（ProcessorCache）、 炒好的同款浇头（EncoderOutputCache）不必重做。预算就是灶眼和砧板的数量，防止一张大桌子占满全部工位。

类比的边界（别被带偏）
后厨可以把两桌相同的菜合并成一锅炒；但引擎绝不能把两个请求的视觉 embedding 合并/共享—— 缓存命中复用的只是「同一份媒体的编码结果」（只读、纯函数产物），每个请求仍各持独立的 KV Cache， 上下文互不相通。「复用编码结果」≠「共享上下文」，这正是红线②。

精确定义：三段式主循环 + 预算门 Learn

MultiModalEngine.run() 是一个 while 循环，只要还有 waiting 或 running 就继续。 每一轮 it 严格按顺序做三件事：

阶段做什么受哪个预算约束形状 / 数字
① decode对每个 prefill_done 且未完成的 running 请求，喂上一个输出 token前进一格 forward([last],[cur],kv)，各产 1 个新 tokentext_token_budget=64：每请求每轮 1 个文本 token 输入 1 个 token → logits (1, vocab=259)
② 准入 + prefill把到达的 waiting 按 (arrival, id) 排序，逐个用 budget.fits(...) 过门； 通过就 _prefill：跑 encoder（命中缓存则跳过）→ 合并 embedding → 整段 forward → 产首 token max_num_seqs=4、visual_token_budget=64、encoder_budget=4 merged embeds (seq_len, 32)；每图视觉块 (4, 32)
③ 回收把 finished()（已生成够 max_new_tokens）的请求移出 running、记 finish_iter —释放该请求独立的 KV

准入的门就是 MultiModalBudget.fits，逐符号看这三个不等式（used_* 是本轮已累计用量）：

used_text    + text_tokens          <= text_token_budget      # 文本 token 上限
used_visual  + visual_tokens        <= visual_token_budget    # 视觉 token 上限（占 KV 位置）
used_encoder + num_media_to_encode  <= encoder_budget         # 本轮编码的媒体段数上限
两个容易忽略的工程细节
(a) 准入调用是 fits(0, visual, num_media, ...)——text_tokens 传 0：prefill 自身的文本 token 不计入本预算， text_token_budget 只用来限制①里 decode 的步数。
(b) num_media 数的是媒体段数（一段视频算 1），而 runner.encoder_runs 数的是 真正逐帧/逐图的编码次数（一段 4 帧视频 = 4 次）。所以 encoder_budget 是个偏粗的算力代理。

小规模手算：2 个请求、2 次迭代 Learn

用最小的例子把循环走一遍。两个请求都在 it=1 到达：

  • A（纯文本）：prompt「你好」= 2 汉字 × 3 字节 = 6 个文本 token（byte-level tokenizer）， visual=0，encoder=0，max_new_tokens=2。
  • B（单图 + 文本）：一张图（4 visual token）+「看」（1 汉字 = 3 token）， 序列 = 4 + 3 = 7 个 token，encoder 要跑 1 次，max_new_tokens=2。
迭代① decode② 准入 + prefill③ 回收累计 encoder_runs
it=1running 为空，无事 排序 [A,B]。A：fits(0,0,0,…)✓ → prefill（不跑 encoder）→ 产首 token，A.cur=6。 B：fits(0,4,1,…)✓（4≤64，1≤4）→ prefill 跑 encoder 1 次 → 产首 token，B.cur=7 A、B 都才 1 个输出（<2），留在 running1
it=2A 前进 1 格（A.output=2 个），B 前进 1 格（B.output=2 个）； assert encoder_runs 不变 守住红线① waiting 已空A、B 均达 max_new_tokens=2 → 回收1

结果：iterations=2，encoder 全程只跑 1 次（都在 prefill，it=2 的 decode 一次都没碰它）， 两个请求 first_token_iter=1（TTFT = 1 个迭代）。这就是引擎「对了」的样子。

形状与数字：visual token 与三层缓存 Learn

把一个单图请求 B 的数据流按形状走一遍（文本模型 hidden=32、layers=2、 heads=4、kv_heads=2、head_dim=8）：

步骤输入形状输出形状说明
vision encoder像素 → (4, 48)(4, vision_hidden=48)2×2=4 个 patch，各 48 维
projector(4, 48)(4, text_hidden=32)把视觉维投影到文本维，才能和文本 embedding 拼接
合并 embedding文本 (7, 32) + 视觉 (4, 32)(7, 32)4 个占位位置被替换成视觉向量
prefill forwardmerged (7, 32)logits (7, 259)取最后一行采样出首 token
该请求 KV—每 token 2·2·8·2 = 64 数7 token → 7×64 = 448 数，B 私有

三层缓存各治一种重复，key 的粒度决定对错：

缓存存什么命中省掉什么key 覆盖
ProcessorCache预处理后的像素/抽帧resize/normalize媒体内容 + processor 配置
EncoderOutputCache投影后的视觉 embedding (4,32)整段视觉编码（最贵）再加 encoder/projector 身份 + dtype + schema 版本
LLM KV Cache各层 K/V历史 token 重算每请求独立，不入上表两缓存
为什么 EncoderOutputCache 的 key 要那么细
若换了 encoder/projector 或改了 dtype 却命中旧缓存，就会悄悄用错的视觉 embedding（stale cache bug）。 所以 key 必须把「一切会影响结果的因素」都算进去——这也是 self.encoder_id / self.projector_id 存在的原因。

常见误解 Learn

误解为什么错
「decode 时也要把图再编码一遍」视觉信息在 prefill 已被 encoder 编码、合并进 KV。decode 是纯文本自回归， 代码里 assert self.runner.encoder_runs == enc_before 就是守这条红线①。重编码 = 20 倍浪费。
「缓存命中就相当于请求共享上下文」命中的是同一份媒体的只读编码结果；每个请求仍有独立 KV， 上下文从不相通。复用编码 ≠ 共享上下文（红线②）。
「一个视频请求 = 1 个视觉 token」视频按帧编码：4 帧 × 每帧 4 patch = 16 个 visual token，每个都占序列位置、进 KV， 是纯文本请求成本的许多倍。
「visual token 只是算力开销，不占内存」每个 visual token 和文本 token 一样占一个序列位置、进一份 KV Cache， 所以它同时进 visual_token_budget 与显存。
「encoder_budget=4 就是最多编码 4 帧」它数的是媒体段数（一段视频算 1），不是帧数。真正逐帧的编码次数看 encoder_runs。
