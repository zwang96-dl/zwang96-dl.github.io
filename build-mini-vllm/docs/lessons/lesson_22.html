为什么需要这个知识 Learn

到这一课为止，我们的文本模型只会一件事：拿到一串 token id（整数 0..258）， 查 embedding 表，然后跑 transformer。可一张图片是像素——它根本不是 token id。 要让模型「看懂图」，必须先把像素翻译成模型能吃的东西：一批和文本 embedding 住在同一个向量空间（本模型是 32 维）的连续向量。这一课就是造这台翻译机，它由两块拼成： vision encoder（读懂一张图内部各部分的关系）和 projector（把维度对齐到文本空间）。 后面 Lesson 23 会把这批向量塞进文本序列里和文字并排， 所以这一课是整条多模态主线的入口——务必把「像素怎么一步步变成 4 个 32 维向量」彻底看清。

先看问题：文本模型压根不认识像素 Learn

用本课的真实数字把「不翻译会怎样」摆出来。一张图是 (3, 16, 16)——3 个颜色通道、 16×16 像素，一共 3×16×16 = 768 个浮点数。而文本模型的入口只接受整数 token id （范围 0..258，因为 vocab_size = 259）。两者的类型和形状都对不上：

比较项文本输入图片（原始像素）
数据类型整数 token id浮点像素值
取值范围0..258（vocab_size=259）连续实数（如 0..1）
一个单位是什么1 个字节 = 1 个 token（中文一字 = 3 字节 = 3 token）1 张图 = 768 个数，没有「id」可言
模型入口查 embedding 表 (259, 32)无表可查——像素不是索引

硬把 768 个像素值当成「token id 去查表」是荒谬的：它们不是整数索引，即便强行取整， 768 也远超 259 的表长。唯一的出路是绕开 embedding 表，直接算出一批 32 维向量， 让它们「看起来就像文本 embedding」——这正是 vision encoder + projector 的职责。

一句话解释 + 直觉类比 Learn

一句话解释
vision encoder 把一张图切成小块（patch），让这些小块互相看一眼、 彼此融合信息，得到每块的「含义向量」；projector 再把这些向量的维度从视觉的 48 投影到文本的 32，使它们能和文字 embedding 并排站在同一个空间里。

类比：把外文文件翻译成本国语言再归档。 图片好比一份外文文件。先把它拆成段落（patch embedding：切 4 块并向量化）； 再让一个精通该语言的读者通读全文、前后对照理解每段的意思 （vision encoder：patch 之间双向互看）；最后翻译成归档系统统一使用的语言 （projector：投影到 text_hidden=32）。翻译完的段落，就能和本国语写的文件放进同一个文件柜。

类比的边界（别被带偏）
翻译强调「换语言但保意思」；这里 projector 换的是向量维度（48→32），不是把图「变成文字」。 视觉 embedding 始终是连续向量，永远不会变回某个 token id，也不对应词表里任何一个词。 它只是被摆到了「和文本 embedding 同一个坐标系」里，好让下游的文本 transformer 能一视同仁地处理。

精确定义：三步各在算什么 Learn

① Patch embedding（回顾）。把 (3,16,16) 按 patch_size=8 切成 grid=2×2=4 个不重叠小块；每块展平成 3·8·8 = 192 维向量，得到 (4, 192)； 再乘一个投影矩阵 proj: (192, 48)，得到 (4, 48)。每一行就是一个 visual token。

② Vision encoder（本课重点，1 层）。对 h: (4, 48) 做一层「双向」transformer block：

hn  = rms_norm(h)                      # (4,48)  归一化
q,k,v = hn·Wq, hn·Wk, hn·Wv            # 各 (4,48)，Wq/Wk/Wv 都是 (48,48)
attn = 非因果多头注意力(q,k,v)          # (4,48)，4 个头，每头 head_dim=48/4=12
h    = h + attn·Wo                      # (4,48)  残差①（Wo: 48×48）
hn2  = rms_norm(h)                      # (4,48)
h    = h + swiglu(hn2)                  # (4,48)  残差②（前馈，中间维 inter=96）

逐符号看：q,k,v 是每个 patch 的查询/键/值；非因果指注意力 不加因果掩码——每个 patch 都能看到全部 4 个 patch（含它自己）。 head_dim = vision_hidden / vision_heads = 48/4 = 12；4 个头各自算 (4,12) 再拼回 (4,48)。 两处残差（h = h + …）和文本 transformer 完全同构，唯一区别就是那一个掩码开关。 输出形状仍是 (4, 48)——encoder 不改形状，只让每个 patch 的向量「吸收」了其它 patch 的信息。

③ Projector（本课重点）。两层线性 + 中间 silu 激活，把 48 维压到 32 维：

h = visual · W1        # (4,48)·(48,32) = (4,32)
h = silu(h)            # 逐元素非线性
out = h · W2           # (4,32)·(32,32) = (4,32)

输入 (4, 48)，输出 (4, 32)。W1: (48,32) 负责真正的降维 （48→32），silu 提供非线性（否则两个线性层会塌成一个），W2: (32,32) 在 32 维空间里再做一次线性变换，让结果更贴合文本 embedding 的分布。输出的 32 正好等于 text_hidden_size = 32——这是能和文本合并的硬性前提。

小规模手算 Learn

手算 A：非因果注意力，patch 0 到底看到谁。为了能用手算，我们把「一个头、head_dim 缩到 1」， 只看 4 个 patch 的一维值。设 4 个 patch 的 value 是 v = [10, 20, 30, 40]， 且 patch 0 对四者的注意力权重（softmax 后）恰为 [0.4, 0.3, 0.2, 0.1]。 非因果下，patch 0 的输出是对全部 4 个 value 加权求和：

out0= 0.4·10+ 0.3·20+ 0.2·30+ 0.1·40 =4+6+6+4 =20

对比：如果误加了文本那种 causal mask，patch 0 只能看自己（位置 ≤ 0），权重被迫变成 [1, 0, 0, 0]，则 out0 = 10——它就再也「看不到」patch 1/2/3 的内容了。 一张图的左上角块本该能参考右下角块（比如判断整体是猫还是狗），加了因果掩码就等于蒙住了它的眼睛。 下表把两种掩码的 4×4 可见性并排画出来（✓=可见，·=被屏蔽）：

非因果（视觉，本课）因果（文本 decode，反例）
✓ ✓ ✓ ✓　patch0 看 0,1,2,3　✓ · · ·　patch0 只看 0
✓ ✓ ✓ ✓　patch1 看 0,1,2,3　✓ ✓ · ·　patch1 看 0,1
✓ ✓ ✓ ✓　patch2 看 0,1,2,3　✓ ✓ ✓ ·　patch2 看 0,1,2
✓ ✓ ✓ ✓　patch3 看 0,1,2,3　✓ ✓ ✓ ✓　patch3 看全部
一句话
因果掩码是下三角（只能看过去），因为文本是从左到右一个个生成的、不能偷看未来； 而一张图整幅同时到达，patch 之间没有先后顺序，掩码是全 1 方阵——人人可见。

手算 B：projector 的 silu 为什么不能省。silu(x)=x·σ(x)（σ 是 sigmoid）。取几个点： silu(0)=0、silu(2)≈2×0.88=1.76、silu(-2)≈-2×0.12=-0.24。 它对正数近似放行、对负数强烈压小——是一条弯曲的曲线。若把它去掉，projector 就变成 out = (visual·W1)·W2 = visual·(W1·W2)，两个线性层塌成单个 (48,32) 矩阵， 表达能力退回一层线性映射。正是中间这一下非线性，让「48→32 的翻译」能学到更复杂的对应关系。

最终 (4, 32) 的 32 必须等于文本模型的 hidden_size = 32。 对齐后，这 4 个 visual token 就能被 Lesson 23 当作「4 个已经算好 embedding 的 token」 插进文本序列，和文字 token 一起喂进同一个 transformer——这也是为什么本课末尾一定要检查 projector 输出维 == text_hidden。

常见误解 Learn

误解为什么错
「视觉 embedding 也是某种 token id，先转成整数再喂模型」它是连续向量，不查词表、不对应任何词。它绕过 embedding 表，直接就是 hidden 空间里的一行 32 维向量。
「视觉注意力也要加 causal mask，保持和文本一致」恰恰相反。文本 decode 加因果掩码是因为不能偷看未来；一张图整幅同时到达，patch 无先后，必须双向互看（全 1 方阵）。加了掩码会让每个 patch 瞎掉一半视野（见手算 A）。
「projector 可有可无，vision_hidden 直接喂文本模型不就行了」维度对不上：encoder 输出 48 维，文本 hidden 是 32 维，无法相加/拼接。projector 的本职就是 48→32 的对齐，缺了它下游直接报维度错。
「projector 用一个线性层就够，中间的 silu 是装饰」去掉 silu，两个线性层会数学上塌成一个矩阵（见手算 B），表达力退化。中间的非线性是它比「单层投影」更强的原因。
「vision encoder 会改变 visual token 的数量或维度」不会。encoder 进出都是 (4, 48)，形状不变；它只让每个 patch 的向量融合了其它 patch 的信息。改维度的是 projector，改数量的（切 patch）是上一课。

代码映射 Learn

本课讲解逐一对应 mini_vllm/multimodal/vision_encoder.py 里的两处：

  • 非因果 encoder → TinyVisionEncoder.encode()。函数体正是「精确定义 ②」那段伪码： rms_norm → Wq/Wk/Wv → 非因果注意力 → +Wo（残差）→ rms_norm → +swiglu（残差）。 「非因果」落在辅助函数 _full_attention() 里，它调用 scaled_dot_product_attention(..., causal=False)——这个 causal=False 就是本课全部「双向」的来源。encode() 进出都是 (4, 48)。
  • 维度对齐 → MultimodalProjector.__call__()。正是「精确定义 ③」： M.matmul(visual, self.w1) → silu → M.matmul(., self.w2)，把 (4,48) 投到 (4,32)。 w1 的形状 (vision_hidden, text_hidden) = (48, 32) 就写死了「48→32」这条对齐规则。
