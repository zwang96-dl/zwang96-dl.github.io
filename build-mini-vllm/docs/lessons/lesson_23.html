为什么需要这个知识 Learn

文本模型的 forward 只认识一件东西：一张 (seq, hidden) 的浮点矩阵。可图片不是文字—— 它经过视觉编码器与 projector 后，是一小块 (4, 32) 的视觉向量（本课每张图 num_patches=4 个 visual token，projector 把 vision_hidden=48 投影到 text_hidden=32）。 那问题来了：怎么把这块视觉向量塞进一条纯文本的序列里，还能让文本模型「以为」它一直就在那儿？

vLLM 的答案分三步，本课就是它的骨架： (1) 占位——在 token 序列里预留一段连续的占位 token（本课用 PAD）； (2) 校验——确认这段占位和视觉向量严格对得上（validate_placeholders()）； (3) 合并——把占位那几行 embedding 逐位换成视觉向量（merge_multimodal_embeddings()）。 合并后的 (seq, 32) 直接喂给 forward(inputs_embeds=...)，文本模型完全无感，照常做因果注意力。 这一步是多模态最容易出对齐 bug、也最需要守护的地方——错一行，整段视觉信息就贴错了位置。

先看问题：不校验，一个 token 的错位有多致命 Learn

本课 tokenizer 是 byte-level：每个 UTF-8 字节 = 1 个 token id（0..255），中文一字 = 3 字节 = 3 个 token； vocab 一共 259（256 字节 + BOS/EOS/PAD）。设我们要处理这条输入： BOS + "Hi" + <图> + "?"。展开成 token：

位置 pos01234567
tokenBOSHiPADPADPADPAD?
来源文本文本文本图片占位（4 个，= num_patches）文本

正确的占位是 [3, 4) 共 4 个 PAD，恰好等于这张图的 4 个 visual token。现在假设上游少写了一个 PAD， 只留了 3 个（占位 [3, 3)… 其实变成 [3,6) 长度 3）：

  • 视觉向量有 4 行，占位却只有 3 行——第 4 行视觉向量无处安放，被悄悄丢弃， 模型只看到 ¾ 张图。
  • 更糟的是：后面的 "?" 本该在 pos 7，现在整条序列的位置全部左移， RoPE 位置编码、因果掩码全部错位——一个 off-by-one 让整段生成崩坏，却不报错。

这类 bug 最阴险的地方是静默：结果只是「变差」，不会崩，很难定位。所以本课的核心不是「合并」这一行赋值， 而是合并之前那道带定位信息的校验闸门——把错误在源头拦下，并直接告诉你错在第几个媒体。

一句话解释 + 直觉类比 Learn

一句话解释
PlaceholderRange = 一张「预留座位」的登记条：记下这段媒体从哪一行开始（offset）、 占几行（length）、是第几个媒体（media_index）、什么模态（modality）。 merge = 按登记条把这几个座位上的「文本 embedding」换成「视觉 embedding」。

类比：剧院预留座位。先买 3 张连号票（BOS/H/i 是普通观众），中间给一个 4 人乐团预留 4 个连座 （4 个 PAD），最后再来 1 位观众（?）。开场前，检票员（validate_placeholders）逐条核对： 乐团人数是否正好等于预留座位数？座位是否在场内、没和别人重叠、编号连续？核对无误，乐团才入座 （merge 逐位替换）。

类比的边界（别被带偏）
预留的座位「本来是空的」，而占位 PAD 是有 embedding 的——只是它的值无所谓，反正整行都会被覆盖。 另外，检票员不会「帮你补票」：校验只发现并报错，绝不自动改数量或挪位置。真正的对齐必须在上游 （inputs.py 的 parse）就算对，校验只是最后一道保险。

精确定义：字段、shape 与 merge 公式 Learn

PlaceholderRange 是一个只装 4 个字段的数据类（placeholders.py）：

字段含义上例（那张图）取值
offset在 input_ids 中的起始行号3
length占多少行（= 该媒体的 visual token 数）4
media_index第几个媒体（从 0 数）0
modality"image" 或 "video""image"

它描述的区间是左闭右开的 [offset, offset+length)，即上例的 [3, 7)，覆盖 pos 3、4、5、6 共 4 行。

merge 的形状（用本模型的具体数字）
merge_multimodal_embeddings(text_embeds, ranges, visual_embeds) -> out
  text_embeds   : (seq, hidden) = (8, 32)   # 按 input_ids 查 embedding 表（vocab=259）
  visual_embeds : list，第 k 项形状 (ranges[k].length, hidden) = (4, 32)
  out           : (seq, hidden) = (8, 32)   # 与 text_embeds 同形，只有占位那几行被替换
逐符号：seq=8 是序列长度；hidden=32 是 text_hidden； (4, 32) 里的 4 是这张图的 visual token 数，32 必须 = text_hidden （视觉侧原本是 vision_hidden=48，靠 projector 投到 32——对不上就是 projector 输出维度没接对）。

替换规则本身极简，就是一段逐行赋值（embedding_merge.py 第 35–36 行）：

for k, r in enumerate(ranges):        # 遍历每个媒体
    for i in range(r.length):          # 遍历这段占位的每一行
        out[r.offset + i] = vis[i]     # 占位第 i 行 ← 视觉第 i 行

也就是把 out[offset .. offset+length-1] 这几行，逐位换成 visual_embeds[k] 的第 0..length-1 行。 其余行（真正的文本 token）原样保留。

五道校验闸门逐条拆解 Learn

validate_placeholders(ranges, media_visual_lens, seq_len) 按顺序设了 5 道闸（对应源码第 26–44 行）， 任何一道不过就抛出带定位信息的 ValueError：

#闸门判据（源码）拦住什么
1数量一致len(ranges) == len(media_visual_lens)占位段数 ≠ 媒体数（多传/漏传一张图）
2顺序正确r.media_index == k（第 k 段就该指向媒体 k）占位顺序和媒体顺序对不上
3长度匹配r.length == media_visual_lens[k]占位行数 ≠ 该图 visual token 数（如 3≠4）
4不越界0 ≤ offset 且 offset+length ≤ seq_len区间超出序列（尾部溢出/负起点）
5不重叠offset > prev_end（上段闭区间终点）两段占位抢同一行
为什么顺序这么重要
第 2 关要求「第 k 个 placeholder 的 media_index 恰好是 k」。这保证 ranges[k] 与 visual_embeds[k] 是下标一一对应的——merge 里直接用 vis = visual_embeds[k] 去填 ranges[k]。一旦顺序乱了，就会「图 0 的向量填进图 1 的座位」，静默贴错。

常见误解 Learn

误解为什么错
「占位用哪个 token、它的 embedding 是什么，得算准」无所谓。占位那几行整行都会被视觉向量覆盖，原值再也不参与计算。本课随手用 PAD 只是为了占位方便。
「图片会被 tokenizer 变成一些特殊 token id」不会。图片走的是另一条路（视觉编码器 → projector → 视觉向量），token 序列里只是预留 4 个空位，真正的信息在 merge 时才注入 embedding 层。
「一张图固定就是 1 个 token」本课一张图 = num_patches = 4 个 token（grid 2×2）。图越大 / patch 越小，token 越多；length 必须等于这个数，不是恒为 1。
「视觉向量维度和文本不一样也能塞」不行。合并进的是同一张 (seq, 32) 矩阵，视觉向量必须先被 projector 投到 text_hidden=32；否则 merge 里 len(vis[0]) != hidden 直接报错。
「校验会顺手帮我补齐/纠正对不上的地方」不会。validate_placeholders 只发现并抛错，不改数据。对齐必须在上游 parse 时就正确。
