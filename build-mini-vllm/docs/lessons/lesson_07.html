为什么需要这个知识 Learn

KV Cache 是理解 vLLM 的第一块基石。它是「用内存换计算」的经典权衡： 花一点显存把中间结果存下来，换回来的是几十倍的解码速度。更重要的是——后面几乎所有话题 （Prefill/Decode 的性能画像、显存碎片、 分页 KV、前缀共享）都是在管理这一块缓存。 所以这一课值得慢下来，彻底搞清楚三件事：缓存什么、为什么正确、要花多少内存。

先看问题：朴素解码到底在浪费什么 Learn

回忆 Lesson 6 的现场：prompt 是 "Hello"（含 BOS 共 6 个 token）， 要生成 8 个新 token。朴素做法每生成一个 token，都把「当前整段序列」重新跑一遍 forward：

生成第几个朴素做法这一步处理的 token 数其中「重复计算」的部分
第 1 个6（整段 prompt）—（这是必要的 prefill）
第 2 个7前 6 个又算了一遍
第 3 个8前 7 个又算了一遍
………
第 8 个13前 12 个又算了一遍

累计处理 6+7+8+…+13 = 76 个 token。可每个历史 token 在每一层算出来的东西， 其实每一步都完全一样——只是被反复重算。如果能把它存下来，累计只需处理 6（prefill）+ 1×7（decode）= 13 个 token。这就是 KV Cache 要做的事。

直觉类比 Learn

一句话解释
KV Cache = 把「每个历史 token 在每一层算出来的 Key 和 Value」存起来， 解码时直接取用，不再重算。

类比：开卷考试做笔记。第一遍通读题目（prefill）时，你把每一段的要点抄成便签（K/V）。 之后每写一句答案（decode 一个 token），你只需扫一眼已有的便签， 而不是把整本书从头再读一遍。写得越长，省下的重读就越多。

类比的边界（别被带偏）
便签是「你写的」，而 K/V 是「模型算出来的中间向量」；更关键的是——一段话的便签内容 只取决于这段话本身，与你后面写了什么无关。这正对应一个数学事实： 第 i 个位置的 K/V 只依赖前 i 个 token。正因如此，历史 K/V 一旦算出就永远有效、可以放心缓存 （这也是 Lesson 16 前缀共享能成立的根本原因）。

为什么缓存 K 和 V，而不缓存 Q？ Learn

要回答这个问题，先把 attention 的公式摆出来（Lesson 4 讲过）。 第 i 个位置的输出是：对它「能看到的」所有位置 j ≤ i 的 Value 做加权求和，权重来自它的 Query 与各 Key 的相似度：

outi= ∑j≤i softmaxj ( qi·kj d ) ·vj

盯住这个式子，看每个量「被谁、在什么时候用到」：

  • k_j 和 v_j（历史的 Key/Value）：只要 j 之后还有 token 要生成， 它们就会被后面每一步反复用到（每个新 query 都要和所有历史 k 比、对所有历史 v 加权）。 → 反复使用，值得缓存。
  • q_j（历史的 Query）：只出现在计算 out_j（也就是它自己那一步）里， 用完再也不出现。生成后面的 token 时，用的是新的 query，绝不会回头用旧 query。 → 一次性，缓存它毫无意义。
一句话
Key/Value 是「被查询、被加权」的对象，会被未来反复读取，所以缓存； Query 是「发起查询」的一方，每步都用新的，历史的用完即弃，所以不缓存。

于是解码一个新 token 的流程变成：只为这一个新 token 算 q、k、v；把新的 k、v 追加进缓存；再让「新 q」去和「缓存里全部的 k、v」做一次 attention。历史部分一个都不用重算。

小规模手算：生成第 3 个 token 时，到底需要什么 Learn

设序列已经有 3 个 token（位置 0、1、2），现在要生成第 3 个新 token（它在位置 3）。 下表列出「算 out_3 需要的每个量」以及它「从哪来」：

需要的量作用朴素做法KV Cache 做法
q_3新 token 的查询现算现算（只此一个）
k_3, v_3新 token 的 K/V现算现算，并追加进缓存
k_0,k_1,k_2 / v_0,v_1,v_2历史 K/V，被 q_3 查询/加权整段重算从缓存取（前几步已算好）
q_0, q_1, q_2历史 Query整段重算（但根本用不上！）不需要

看最后一行：朴素做法把 q_0,q_1,q_2 又算了一遍，却完全没用到——纯浪费。 KV Cache 只保留真正会被复用的 K/V，因此结果逐值不变，只是不再做无用功。

为什么「只喂 1 个 token」还能得到正确结果？
关键在绝对位置：新 token 在位置 3，它的 attention 要看位置 0..3。缓存里的 k/v 本来就带着它们各自的位置（0、1、2）；新 token 用位置 3 算 q/k/v。位置掩码保证 「新 q 能看到 0..3 的全部 k/v」——这和「把 0..3 整段重算一遍」在数学上完全等价。 所以 run 7 会看到 cached 与 naive 的 logits 最大误差 = 0。

缓存里到底存什么？用本模型的具体数字 Learn

本课 tiny model 的配置：hidden=32、layers=2、heads=4、 kv_heads=2（GQA）、head_dim=8。缓存是逐层的，每个 token 每层要存：

  • K：kv_heads × head_dim = 2 × 8 = 16 个数
  • V：同样 16 个数
  • 合计 K + V = 32 个数 / token / 层

所以一个 token 在 2 层里共占 32 × 2 = 64 个数。若 prompt 6 个 token、再生成 8 个， 缓存最终长度 14 个 token，共 14 × 64 = 896 个数。动画里看到的「缓存行数增长」， 每一行就对应一个 token 的这 32（每层）个数。

形状（shape）
逐层看，缓存里的 K 是 (已缓存 token 数, kv_heads·head_dim) = (seq_len, 16)，V 同形。 decode 一步 = 给每层的 K、V 各追加一行（seq_len → seq_len + 1）。

内存公式：为什么长上下文这么吃显存 Learn

KV Cache 大小（元素个数）
2 · num_kv_heads · head_dim · num_layers · seq_len
其中 2 是 K 和 V 两份。它随 序列长度线性增长，随层数、KV 头数线性增长。

下表用本模型（2 层、head_dim 8、一条 128-token 序列）对比三种注意力变体的 KV 头数如何影响缓存：

变体kv_heads每 token 每层 (2·kv·hd)128 token × 2 层 合计相对
MHA（多头，kv=heads）42·4·8 = 6416384 个数1×（最大）
GQA（分组，本课）22·2·8 = 328192 个数½
MQA（多查询）12·1·8 = 164096 个数¼

GQA/MQA 用更少的 KV 头换取更小的缓存——这就是它们在长上下文、高并发下流行的原因。

真实模型的体感（本课是玩具，但公式一样）
以 Llama-2-7B（32 层、32 个 KV 头、head_dim 128、fp16 每数 2 字节）代入：
每 token = 2·32·128·32 = 262144 个数 ≈ 512 KB；一条 2048 token 的序列 ≈ 1 GB 显存。
并发几十条长序列，KV Cache 就吃掉大半张卡—— 这正是后面要分页管理（Lesson 12–14）和前缀共享（Lesson 16）的直接动机。

常见误解 Learn

误解为什么错
「decode 时要把整段序列重新喂进模型」只喂 1 个新 token，历史 K/V 从缓存取。喂整段就退化成朴素做法了。
「Query 也缓存一下能更省」历史 Query 之后再也用不到，缓存它纯占内存、零收益。
「用了缓存，输出可能会变一点」缓存的是精确的中间量，不是近似。cached 与 naive 逐值一致（误差 0）——它只加速、不改变结果。
「KV Cache 是省内存的优化」恰恰相反：它花内存（存 K/V）来省计算。是「以空间换时间」。
