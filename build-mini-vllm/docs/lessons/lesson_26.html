为什么需要这个知识 Learn

视频和图片最大的不同，是它多了一个时间维度：一张图就是一张图，一段视频却是 成百上千张图排成一队。上一课（Lesson 25）我们把一张 16×16 的图切成 2×2 = 4 个 patch，得到 4 个 visual token。视频只是把这件事重复很多遍——但「很多遍」正是问题所在。

先算一笔账。本课模型的关键限制是 max_seq_len = 64（一条序列最多 64 个 token）， 而每一帧图片都要占 grid² = 2×2 = 4 个 visual token。于是「帧数」直接换算成「token 数」：

视频总帧数× 每帧 4 个 visual token能塞进 max_seq_len=64 吗？
一段 10 秒、30fps 的手机录像300 帧= 1200 个 token❌ 超了近 20 倍
本课示例：6.5 秒、2fps13 帧= 52 个 token⚠️ 勉强塞下，但没地方放文字了
抽 4 帧（uniform）4 帧= 16 个 token✅ 还剩 48 个 token 给文字

结论很直接：把每一帧都喂给模型，几乎必然爆掉上下文。所以视频进模型前一定要先 抽帧（frame sampling）——从整段里挑出有代表性的几帧。怎么挑（抽帧策略） 决定了模型「看到什么」，挑几帧决定了「花多少成本」。这就是本课要彻底讲清的两件事。

先看问题：抽帧策略挑错会怎样 Learn

既然必须只挑几帧，那「挑哪几帧」就成了关键。用一个具体例子体会不同挑法的差别： 一段 6.5 秒、source_fps=2 的视频，共 13 帧（frame_index 0..12， 时间戳 0.0, 0.5, 1.0 … 6.0 秒）。假设它的内容是「前 2 秒静止 → 中间发生关键动作 → 最后 2 秒定格」。 现在只允许抽 4 帧：

策略抽到的帧（frame_index）对应时间戳后果
head（取前 4 帧）0,1,2,30.0–1.5 秒❌ 只看到开头静止段，完全错过中间的关键动作
tail（取后 4 帧）9,10,11,124.5–6.0 秒❌ 只看到结尾，同样错过关键动作
uniform（均匀 4 帧）0,4,8,120.0, 2.0, 4.0, 6.0 秒✅ 全片均匀覆盖，关键动作大概率被采到

同样是抽 4 帧、同样的成本，head/tail 可能让模型对整段视频得出完全错误的理解。 这说明抽帧不是「随便挑几帧」——策略本身就是信息取舍。

直觉类比 Learn

一句话解释
抽帧 = 从一整段视频里，按某种规则挑出少量代表帧喂给模型；每个被挑中的帧 还随身带着两个「时间标签」——它是第几帧（frame_index）、它在第几秒（timestamp）。

类比：把电影做成连环画。一部两小时的电影有十几万帧，你没法把每一格都画进连环画。 你会怎么挑？均匀地每隔几分钟挑一格（这就是 uniform），或者每秒固定挑一格 （这就是 fixed_fps）。挑完还要在每格下面标上「第 12 分钟」——不然读者不知道这几格之间隔了多久。 那行小字，就是 timestamp。

类比的边界（别被带偏）
连环画下面的「第 12 分钟」是画在图上、读者一眼能看到的；但在模型里，timestamp 并不会自动出现在模型眼前。模型只收到几帧图像变成的 visual token，时间戳被单独存在 metadata 里。除非你显式把「第 12 分钟」写进文字 prompt，否则模型根本不知道两帧之间隔了多久。 这是本课最容易踩的坑，后面会专门再讲一次。

四种抽帧策略：精确定义 Learn

设输入是 n 帧（n = len(frames)），每帧 pixels 的形状是 (16, 16, 3)（image_size=16 的 RGB 图）。sample() 的输出是一个 更短的帧列表，形状 (k, 16, 16, 3)，k 就是抽到的帧数。 四种策略只是用不同规则决定「抽哪些下标」：

1. uniform（均匀 N 帧）——把 k 个采样点均匀铺在 [0, n-1] 上。 其中 k = min(num_frames, n)（想抽的帧数，但不能超过总帧数）。第 i 个采样点的下标：

idxi= round( i·(n-1) k-1 ) , i=0,1,…,k-1
  • i：第几个采样点（0 到 k-1）；
  • n-1：最后一帧的下标（下标从 0 数）；
  • k-1：把区间平均分成 k-1 段，所以每段跨度 (n-1)/(k-1) 帧；
  • round(...)：帧下标必须是整数，四舍五入到最近的一帧；
  • 特例：k==1 时只取 [0]；算出的下标去重并保序（避免同一帧被抽两次）。

2. fixed_fps（固定帧率抽帧）——不管总长多少，每隔固定时间抽一帧。 设视频原始帧率 source_fps、目标帧率 target = fps，则采样步长：

step= max(1, round( source_fps target )) , 抽取下标  0,step,2·step,…
  • source_fps / target：原视频每秒 source_fps 帧，只想留 target 帧，所以每 source/target 帧留 1 帧；
  • max(1, …)：步长至少为 1（目标帧率高于原帧率时不可能「补帧」，只能全留）；
  • 关键区别：uniform 的输出帧数固定（k 帧），fixed_fps 的输出帧数随视频时长变化——长视频抽得多，短视频抽得少。

3. head（取前 N 帧）：frames[:min(num_frames, n)]——直接切前 k 帧。
4. tail（取后 N 帧）：frames[max(0, n-num_frames):]——直接切后 k 帧。

四种策略一眼对比
策略规则输出帧数覆盖范围适合
uniform均匀铺满全片固定 k 帧整片不知道重点在哪、要总览
fixed_fps每隔固定时间抽随时长变（长→多）整片要保持「每秒信息密度」一致
head取开头固定 k 帧片头只关心开头（如封面/开场）
tail取结尾固定 k 帧片尾只关心结尾（如结果/字幕）

小规模手算：13 帧视频，四种策略各抽出什么 Learn

沿用前面那段 13 帧、source_fps=2 的视频（frame_index 0..12，时间戳 0.0…6.0 秒）。 一步步算给你看。

uniform，num_frames=4：先定 k = min(4, 13) = 4，代入 idx_i = round(i·(13-1)/(4-1)) = round(i·12/3) = round(4i)：

i算式 round(4·i)抽到的 frame_index时间戳
0round(0)00.0 s
1round(4)42.0 s
2round(8)84.0 s
3round(12)126.0 s

得到 4 帧、时间上正好每 2 秒一帧，铺满全片。

fixed_fps，target fps=1：步长 step = max(1, round(source_fps/target)) = max(1, round(2/1)) = 2。 从 0 开始每隔 2 帧抽一次：下标 0, 2, 4, 6, 8, 10, 12，共 7 帧， 时间戳 0,1,2,3,4,5,6 秒——正好「每秒 1 帧」。注意它抽出的是 7 帧而不是 4， 因为帧数由时长决定，不由 num_frames 决定。

head / tail，num_frames=4：head → frames[:4] 得 frame_index 0,1,2,3（0.0–1.5 秒）；tail → frames[9:] 得 frame_index 9,10,11,12（4.5–6.0 秒）。都只是简单切片。

三个时间元数据：fps、frame_index、timestamp Learn

每个 VideoFrame 是一个很小的数据结构，只装三样东西（见 media.py）：

字段含义本例中的值
pixels这一帧的图像，形状 (16,16,3)一张 16×16 RGB 图
frame_index它是原视频里的第几帧（从 0 数，整数）0, 1, 2, …, 12
timestamp_seconds它出现在第几秒（presentation timestamp，浮点）0.0, 0.5, …, 6.0

三者的关系可以用一条公式串起来。fps（frames per second）是「每秒多少帧」， 那么第 frame_index 帧的时间戳就是：

timestamp≈ frame_index source_fps

本例 source_fps=2，所以 frame_index=4 的帧在 4/2 = 2.0 秒——和上面手算完全对上。 反过来说：光看 frame_index 不知道两帧隔了多久，必须结合 fps 才能换算成真实时间。 这就是为什么每帧要单独存一个 timestamp——尤其抽帧后帧号会变得不连续（0,4,8,12）， 此时 timestamp 是唯一可靠的「它到底在第几秒」的依据。

timestamp 是 metadata，LLM 不会自动理解它
这是本课最关键、也最反直觉的一点。抽帧后每帧变成 4 个 visual token（图像 embedding）， 时间戳并没有被编码进这些 token——它被单独放在 timeline() 返回的 metadata 里。模型看到的只是「一串图像 token 按顺序排列」，它能感知先后顺序，却不知道 间隔多久：它分不清相邻两帧是隔了 0.5 秒还是隔了 5 分钟。
想让模型「懂时间」，唯一的办法是把时间显式写进文字 prompt，例如在每帧前插入文本 "[t=2.0s]"。不写，模型就永远不知道。

成本账：抽几帧 = 花多少 token Learn

抽帧之后，每一帧都会走 Lesson 25 的图像 patch 流程：16×16 的图切成 2×2 = 4 个 patch， 每个 patch → 1 个 vision_hidden=48 的向量 → 投影到 text_hidden=32 → 1 个 visual token。 所以总成本是一道乘法：

视频 visual token 数= 抽到的帧数 k×grid2 =k×4
策略（13 帧视频）抽到帧数 k× 4 = visual token 数占 max_seq_len=64 的比例
不抽帧（全喂）135281%——几乎塞满，没空间放文字
fixed_fps（1fps）72844%
uniform / head / tail（4 帧）41625%——留下 48 个 token 给文字问题

这张表把本课的两条主线合到了一起：抽帧策略决定「看到什么」，抽帧数量决定「花多少 token」。 两者都要在「覆盖度」和「上下文预算」之间权衡——这正是视频进多模态模型时最核心的工程取舍。

常见误解 Learn

误解为什么错
「抽帧就是随便挑几帧，挑哪几帧无所谓」大错。head/tail 只看片头/片尾，会整段错过发生在中间的关键内容；uniform 才铺满全片。策略本身就是信息取舍。
「fixed_fps 和 uniform 都抽固定数量的帧」只有 uniform 是固定 k 帧。fixed_fps 的帧数随视频时长变化——长视频抽更多帧、更吃 token，短视频抽更少。
「模型能从帧的顺序里看出时间间隔」模型只知道先后顺序，不知道间隔多久。抽帧后帧号变成 0,4,8,12，可这时间差没进 token，模型分不清是隔 2 秒还是隔 2 分钟。
「timestamp 是 metadata，所以模型自然会读到它」metadata 是给 processor（数据处理）用的，不在模型输入里。不显式写进文字 prompt，LLM 永远看不到 timestamp。
「抽帧后不用管顺序，反正都是同一段视频」必须保序！sample() 里刻意「去重并保序」，打乱帧序等于打乱时间线，模型会误读事件的发生顺序。
