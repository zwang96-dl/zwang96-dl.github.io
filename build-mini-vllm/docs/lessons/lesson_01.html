Lesson 2 · Tensor Shape 与矩阵乘法

为什么需要这个知识 Learn

这是整门课的第 0 号地基。神经网络本质是一堆矩阵乘法，它只吃数字， 从不认识 "你好" 这样的字符串。所以任何一次推理，第一步永远是把文本翻译成一串整数 ——这串整数就叫 token IDs。后面每一课都建立在它之上：

  • Lesson 2 的张量形状 [batch, seq, hidden] 里的 seq，就是「这段文本有多少个 token」。
  • Lesson 7 的 KV Cache 大小、Lesson 9 调度器的 token 预算，单位都是 token。
  • 模型最后吐出的 logits 形状是 [1, seq, 259]，最后那个 259 就是本课的词表大小。

换句话说：token 是这门课自始至终的计价单位。搞不清一段文本会变成多少个、哪几个 token， 后面所有的 shape、显存、吞吐都成了空中楼阁。这一课就把「文本 → token IDs」这一步彻底摊开。

先看问题：为什么不能直接喂字符 Learn

最朴素的想法是「一个字符给一个编号」。看起来可行，但 Unicode 有 约 15 万 个字符（还在不断新增 emoji）。如果按字符建词表：

  • 词表要开到 15 万 起步——本课模型的 embedding 表本来只有 259 × 32 = 8288 个数，按字符建表会瞬间膨胀成 150000 × 32 = 480 万 个数，光一张表就大了近 580 倍。
  • 还是会漏：明天出个新 emoji，词表里没有它，就成了「未登录词 (OOV)」，只能用一个 <unk> 兜底，信息直接丢失。

换个角度：无论多生僻的字符，计算机存储它时早已把它编码成了一串 UTF-8 字节， 而字节永远只有 256 种取值（0–255）。于是我们干脆直接把字节当作 token：

方案词表大小会不会遇到 OOV？代价
按字符建表约 15 万，且不封闭会（新字符）词表巨大、embedding 表爆炸
按字节建表（本课）256 + 3 = 259，永久封闭永不序列变长（一个汉字 = 3 个 token）

这就是 byte-level tokenizer：用「序列更长一点」换来「词表极小、永不 OOV、零依赖」。 下面这张表先把三个最容易混的概念分清楚——它们不是一回事：

character / byte / token 三者的区别（以「你」为例）
概念是什么「你」对应几个
character 字符人眼看到的一个字1 个字符
byte 字节UTF-8 编码后的原始字节3 个字节：228, 189, 160
token 词元模型真正吃进去的输入单位；本课里 1 字节 = 1 token3 个 token（id 228,189,160）
记忆锚点：1 个汉字 = 3 个字节 = 3 个 token；纯英文字母/数字是 1 字节 = 1 token； 常见 emoji 是 4 字节 = 4 个 token。

直觉类比 Learn

一句话解释
byte-level tokenizer 就是「把文本按 UTF-8 拆成字节，每个字节 (0–255) 当作一个 token id」， 再在词表里补上三个特殊 token：BOS(256) / EOS(257) / PAD(258)。

直觉类比：拼音字母 vs. 完整词语。你可以用几万个「词」来记一门语言（像真实大模型的子词词表）， 也可以只用 26 个字母把任何词拼出来。byte-level 走的是后者的极端版：不用 26 个字母，而用 256 个字节积木，任何文本——中文、emoji、乱码——都能用这 256 块积木拼出来，每块积木编号固定。

类比的边界（别被带偏）
字母能拼出词，但字母本身没什么语义；byte-level 也一样——单个字节几乎不携带语义，语义要靠模型 在多个字节 token 之间自己学出来。真实大模型（GPT/Llama）用的是 BPE / 子词 tokenizer： 一个 token 往往是「一个常见词或词根」（多个字节合成一块），词表有几万个，序列更短、语义更集中。 本课选 byte-level 只为一件事：完全透明、可手算、零依赖——数据流和张量 shape 与真实系统一模一样。

精确工程定义 Learn

tokenizer 对外只有三个核心方法。先看签名，再逐个拆解它们的输入 / 输出到底是什么形状：

方法签名作用
encodestr → list[int]取 UTF-8 字节为 id，可选前加 BOS、后加 EOS
decodelist[int] → str丢弃特殊 token，其余字节按 UTF-8 一次性还原
padlist[list[int]] → (ids, mask)补齐到同长，并生成 attention mask

逐符号解释 pad 的输出形状（它是三者里唯一涉及二维形状的）。设一批有 B 条序列， 补齐后的目标长度为 L（默认取批内最长那条的长度）：

input_ids      : shape (B, L)   # 每行是一条补齐后的序列，值 ∈ [0, 259)
attention_mask : shape (B, L)   # 同形；1 = 真实 token，0 = PAD 位置
  • B = batch 里序列条数（比如同时来了 3 个请求，B = 3）。
  • L = 目标长度。pad 里 target = length or max(len(seq))——不给 length 就取批内最长。
  • 补进去的位置：input_ids 填 PAD_ID = 258，attention_mask 填 0。
  • side="right"（默认）把 PAD 加在右边；side="left" 加在左边（生成任务常用左 pad）。
mask 为什么必须存在
PAD 是为了凑齐形状硬塞进去的假 token，本身没有任何含义。如果不告诉下游「这些位置是假的」， attention 就会把 PAD 也当成真词去计算，污染结果。attention_mask 就是那张「哪些是真、哪些是假」的清单， 下游据此把 PAD 位置的注意力压成 -∞（Lesson 4 会用到）。

所以词表大小 vocab_size = 256 + 3 = 259。这个数会一路贯穿全课：它既是 tiny model 的 vocab_size，也是模型输出 logits 的最后一维——形状 [1, seq, 259]， 意思是「对序列里每个位置，给出下一个 token 落在这 259 个候选上的分数」。

常见误解 Learn

误解为什么错
「一个字符 = 一个 token」只对单字节字符（ASCII 字母/数字）成立。中文一字 = 3 字节 = 3 个 token，emoji 常是 4 token。token 数看的是字节数，不是字符数。
「byte-level 会遇到不认识的字 (OOV)」永远不会。任何 Unicode 文本都能编码成 UTF-8 字节，而字节只有 256 种、全在词表里。这正是 byte-level 最大的好处——词表永久封闭。
「decode 时一个字节一个字节地转字符」会把多字节字符截断成 �。正确做法是先收集全部字节，再 bytes(...).decode("utf-8", errors="replace") 一次性还原。
「PAD 和普通 token 一样会参与计算」PAD 只是凑形状的假 token。必须靠 attention_mask（PAD 位置 = 0）让下游忽略它，否则结果被污染。
「特殊 token 也要按字节 decode 回文本」BOS/EOS/PAD 的 id ≥ 256，超出字节范围。decode 时要跳过它们（skip_special=True），否则往返对不上。

代码导读 Learn

上面讲的每一步，都能在 mini_vllm/tokenizer.py 里一一对上。三个方法各自的核心就一两行：

# encode()：str -> list[int]
ids = []
if add_bos: ids.append(self.bos_id)   # 前加 BOS(256)
ids.extend(text.encode("utf-8"))      # ← byte-level 的核心：每个字节即一个 id
if add_eos: ids.append(self.eos_id)   # 后加 EOS(257)

# decode()：list[int] -> str
byte_vals = [tid for tid in ids if tid < 256]     # 跳过 ≥256 的特殊 token
return bytes(byte_vals).decode("utf-8", errors="replace")  # 先收集、再一次性还原

# pad()：list[list[int]] -> (input_ids, attention_mask)
target = length or max(len(x) for x in batch)
# side="right": seq + [PAD]*n , mask = [1]*len(seq) + [0]*n
